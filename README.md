## 短链项目定位

当前只维护一个正式项目：Go 短链服务。目标是把它逐步做成能够写进简历、也经得住追问的实战项目，而不是堆多个浅层 Demo。

| 版本 | 目标 | 主要内容 |
|---|---|---|
| V1 · 正确业务 | 核心链路完整可用 | 创建短链、跳转、用户与鉴权、MySQL 持久化、参数校验、统一错误处理 |
| V2 · 简历级 | 具备完整工程交付能力 | Redis 缓存、测试、结构化日志、配置管理、Docker、CI、监控与部署说明 |
| V3 · 强亮点 | 对真实瓶颈做有证据的优化 | 防缓存穿透、限流、可靠统计、降级策略、压测、pprof 与优化前后对比 |

项目质量标准：

- 每个功能都有明确业务目的，不为“技术栈数量”硬加组件。
- README 能说明架构、启动方式、接口、数据模型和关键取舍。
- 重要链路有测试或可复现的验证步骤。
- 性能数字必须来自可复现的本机实验，记录环境、命令和结果；没有数据就不写夸张结论。
- 用户需要能独立讲清核心链路，并能在不依赖 AI 的情况下完成小改动和定位常见错误。

## AI 协作方式

Codex 或 Cursor 可以直接参与正式项目的设计和编码，但 AI 生成代码不等于已经掌握。

| 场景 | 默认方式 |
|---|---|
| 课程小练习、语法题 | 自己先写，AI 负责提示、审查和解释 |
| 正式短链项目 | AI 可以直接实现和重构，用户负责理解、验证和复述 |
| 核心业务链路 | 必须逐段讲清输入、处理、数据变化、异常和输出 |
| 新组件或复杂优化 | 先说明它解决什么问题，再决定是否引入 |
| 简历描述 | 只写真实完成、能够解释、能够复现的内容 |

跨工具接力时，Codex 与 Cursor 按 [.memory/readme.md](.memory/readme.md) 写入对话摘要；实时学习进度仍统一维护在 `learn.md`，避免多个文件互相矛盾。
