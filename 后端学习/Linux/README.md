# Linux for Go 后端：学习与部署手册

这套笔记面向正在学习 Go/Gin、准备后端实习，并最终要把“短链服务”部署到 Linux 的学习者。它不是一份命令大全，也不要求你先学完 Linux 才能写后端；它的目标是让你在真正需要某项能力时，知道系统发生了什么、该用什么命令验证、出错后按什么顺序排查。

文档统一以 **Ubuntu 24.04 LTS** 为主要环境。日常命令练习优先放在 WSL2，涉及 systemd、网络边界、SSH 与正式部署时使用虚拟机或云服务器。

## 这套笔记怎么学

不要从第一页背到最后一页。建议采用“主线项目遇到问题，再把相关章节学透”的方式：

1. 当前学习 Go/Gin 时，只把 README 与第 00 章当地图；遇到命令、权限或端口问题，再查第 01～03 章，不要暂停编码去线性通读。
2. 准备接入 MySQL、Redis 时，重点完成第 04 章，并用第 02～03 章补进程、日志和网络直觉。
3. 短链项目开始部署时，集中完成第 05～09 章，在 VM 中真正跑一遍发布、回滚和恢复。
4. 项目稳定运行后，用第 10 章制造故障、形成排障证据链并准备面试表达。

Linux 学习的闭环应当是：

```text
理解概念 → 执行命令 → 观察输出 → 主动制造一个小故障 → 根据证据修复
```

只看命令不执行，容易产生“看起来会了”的错觉；只复制命令不理解，则会在环境稍有变化时卡住。

## 贯穿全书的项目

所有部署例子围绕一个 Go 短链服务展开。最终形态大致如下：

```text
浏览器 / curl
      │ HTTPS :443
      ▼
    Nginx
      │ HTTP 127.0.0.1:8080
      ▼
 Go/Gin 短链服务
      ├── MySQL：用户、短链、访问记录
      └── Redis：缓存、限流或热点数据
```

建议服务至少提供：

- `GET /healthz`：只表示进程存活，可用于最基础探活。
- `GET /readyz`：表示必要依赖可用，适合判断能否接收业务流量。
- `POST /api/links`：创建短链。
- `GET /:code`：重定向到原始地址。

前几章不会要求你已经完成整套业务。可以先用一个只含 `/healthz` 的最小 Go 程序练习进程、systemd 和 Nginx，之后再替换为真实短链二进制。

## 环境分工

| 环境 | 推荐用途 | 不适合作为唯一练习环境的原因 |
|---|---|---|
| WSL2 + Ubuntu 24.04 | 日常命令、Git、Go 编译、Shell、文本处理 | 网络由 Windows 和 WSL 共同管理，与独立服务器仍有差异 |
| Ubuntu 24.04 虚拟机 | systemd、SSH、防火墙、Nginx、完整部署演练 | 需要分配内存、磁盘并维护快照 |
| Ubuntu 24.04 云服务器 | 公网 DNS、TLS、真实安全组和线上部署 | 可能产生费用，错误开放端口会暴露到公网 |

推荐顺序是 **WSL2 → 虚拟机 → 云服务器**。云服务器不是“更高级的终端”，而是会持续受到公网扫描的真实主机；不要把练习用弱密码、开放的 MySQL 或 Redis 直接搬上去。

## 命令约定

文档中的提示符不需要输入：

```text
$ command       普通用户执行
# command       root 权限执行；通常应通过 sudo command 获得
```

示例中的占位符需要替换：

```text
<SERVER_IP>       服务器 IP
<YOUR_USER>       你的 Linux 用户名
<DOMAIN>          你的域名
```

代码块中的输出是典型示例，PID、IP、时间、用户名、包版本等通常不会与你完全相同。判断实验是否成功，应看字段含义和状态，不要逐字比对。

## 安全边界

学习 Linux 时，最危险的不是“命令不会”，而是不清楚命令作用范围。执行以下操作前必须停下来确认：

- `rm -rf`、递归 `chown`、递归 `chmod`：先用 `pwd`、`ls` 或 `find` 确认目标。
- 修改 SSH、防火墙、网络：保留一个已登录会话，并先验证新配置，避免把自己锁在服务器外。
- 修改 systemd、Nginx、数据库配置：先做语法检查，再 reload/restart。
- 对公网开放端口：只开放业务确实需要的 `22`、`80`、`443`；数据库和 Redis 默认不直接暴露公网。
- 处理口令和密钥：不写入 Git，不直接写进 systemd unit，不出现在截图和聊天记录里。
- 使用 `sudo`：只给需要提权的那一条命令加 `sudo`，不要长期停留在 root shell。

文档会把可能破坏状态或改变系统配置的命令标成“需要 sudo”或“仅在实验机执行”。即便如此，也要先理解再执行。

## 一个命令应该怎样学

以 `systemctl restart shortlink` 为例，不要只记住拼写。至少要回答：

1. 它操作的是哪个 unit？
2. 重启时旧进程收到什么信号？
3. 新进程以哪个用户运行？
4. 配置和环境变量从哪里读取？
5. 启动失败去哪里看日志？
6. 如何确认端口真的开始监听？

同理，学习 `chmod` 时不仅要会写 `755`，还应知道目录的执行位为什么决定能否穿过目录；学习 `curl` 时不仅要看到 JSON，还要分清 DNS、TCP、TLS、HTTP 分别在哪一层失败。

## 推荐的实验目录

普通练习统一放在自己的家目录，避免误改系统文件：

```bash
mkdir -p ~/linux-lab
cd ~/linux-lab
```

部署章节会使用更贴近生产的路径：

```text
/opt/shortlink/          应用二进制和发布版本
/etc/shortlink/          配置文件
/var/lib/shortlink/      应用持久状态（若需要）
/var/log/                系统日志；应用日志优先交给 journald
```

这些目录通常需要 sudo 创建，但应用进程不应以 root 身份运行。

## 每章的完成标准

不要求背下所有选项。满足下面三点即可继续：

1. 能用自己的话解释本章核心模型。
2. 能在不照抄整段答案的情况下完成主实验。
3. 遇到失败时，能先收集证据再改配置。

适合做个人速查表的是高频命令和排障顺序，不适合速记的是几十个低频参数。低频参数应通过 `--help`、`man` 或官方文档查询。

## 版本与资料范围

- 操作系统：Ubuntu 24.04 LTS。
- Shell：Bash。
- 服务管理：systemd。
- 应用语言：Go，Web 框架以 Gin 为主，但 Linux 原理不依赖 Gin。
- 容器：Docker Engine 与 Compose v2。
- Web 入口：Nginx。
- 数据服务：MySQL 与 Redis。

命令在 Ubuntu 24.04 上验证思路最明确，但 Linux 的进程、权限、网络和日志模型在大多数现代发行版上相通。包名、默认路径和防火墙工具可能因发行版不同而变化，不要把 Ubuntu 命令机械套到 CentOS、Alpine 或 macOS。
