# Go 短链服务完整部署：从二进制到可回滚上线

这一章完成一次真正的单机生产式部署。目标不是“把二进制扔到服务器运行”，而是建立一条可以重复、检查、回滚和恢复的链路。

最终形态：

```text
                        /etc/shortlink/
                        ├── shortlink.env       非敏感配置
                        └── credentials/        root-only 凭据源文件
                                  |
                                  v
Internet -> Nginx -> 127.0.0.1:8080 -> shortlink.service
                                             |
                                             +-> MySQL（真实数据源）
                                             +-> Redis（优先只作缓存）

/opt/shortlink/releases/20260715T103000Z-a1b2c3d/
/opt/shortlink/releases/20260720T091500Z-d4e5f6a/
/opt/shortlink/current -> releases/20260720T091500Z-d4e5f6a
```

本章主线采用“Go 二进制 + systemd”。Docker 方案已经在第 07 章给出。单机项目两种都能用，但不要同时让 systemd 和 Compose 启动两份应用并争抢 `8080`。

---

## 1. 上线完成的定义

只有同时满足下面条件，才算部署完成：

- Go 服务由专用低权限用户运行，而不是 root 或日常登录用户。
- 应用只监听 `127.0.0.1:8080`，公网只能经 Nginx 访问。
- 非敏感配置和密钥分离，密码不写进 unit、命令行、Git 或发布包。
- 发布物有版本、提交号和 SHA-256，可追溯到唯一构建。
- 每次发布进入独立 release 目录，通过原子符号链接切换。
- systemd 能守护进程，但不会因错误配置无限高速重启。
- 应用有 liveness、readiness 和优雅退出。
- 发布后自动检查 readiness，失败自动切回上一 release。
- MySQL 有异机备份，并且做过恢复演练。
- Nginx、应用、数据库日志能通过 request ID 或时间线关联。

“浏览器能打开一次”只能说明某条请求当时成功，不能证明上述链路成立。

## 2. 应用需要提供的运行契约

部署脚本和 systemd 依赖清晰的应用契约。建议短链二进制支持：

```text
shortlink-server serve [flags]        启动 HTTP 服务
shortlink-server migrate [flags]      运行数据库迁移，成功返回 0
shortlink-server version              输出版本、commit、构建时间
shortlink-server check-config         只校验配置，不启动监听
```

HTTP 端点：

| 路径 | 含义 | 是否查询外部依赖 |
|---|---|---|
| `/healthz` | 进程和 HTTP loop 活着 | 否 |
| `/readyz` | 当前是否适合接收业务流量 | 是，但必须短超时 |
| `/api/v1/version` | 返回版本/commit，避免泄露环境秘密 | 否 |

readiness 可检查 MySQL `PingContext` 和 Redis `PING`，但每项必须有很短的上下文超时。它不是用来执行全表扫描、写测试数据或检查所有第三方服务的。

### 2.1 优雅退出不是 systemd 自动替你完成的

systemd 停止服务时先发送 `SIGTERM`。Go 程序应停止接收新请求，并给在途请求有限时间结束：

```go
srv := &http.Server{
    Addr:              cfg.HTTPAddr,
    Handler:           router,
    ReadHeaderTimeout: 5 * time.Second,
    ReadTimeout:       10 * time.Second,
    WriteTimeout:      10 * time.Second,
    IdleTimeout:       60 * time.Second,
}

ctx, stop := signal.NotifyContext(
    context.Background(),
    os.Interrupt,
    syscall.SIGTERM,
)
defer stop()

errCh := make(chan error, 1)
go func() {
    err := srv.ListenAndServe()
    if err != nil && !errors.Is(err, http.ErrServerClosed) {
        errCh <- err
        return
    }
    errCh <- nil
}()

select {
case err := <-errCh:
    if err != nil {
        return fmt.Errorf("http server: %w", err)
    }
case <-ctx.Done():
    shutdownCtx, cancel := context.WithTimeout(context.Background(), 20*time.Second)
    defer cancel()
    if err := srv.Shutdown(shutdownCtx); err != nil {
        _ = srv.Close()
        return fmt.Errorf("graceful shutdown: %w", err)
    }
}

return nil
```

主函数将返回错误记录后以非零码退出，正常 `SIGTERM` 路径退出码为 `0`。应用 shutdown 超时应小于 systemd 的 `TimeoutStopSec`，给日志刷新等收尾留出空间。

## 3. 创建专用用户和目录

### 3.1 专用服务用户

```bash
sudo useradd \
  --system \
  --user-group \
  --home-dir /var/lib/shortlink \
  --create-home \
  --shell /usr/sbin/nologin \
  shortlink
```

验证：

```bash
getent passwd shortlink
id shortlink
```

这个用户：

- 不能密码登录；
- 不需要 sudo；
- 不能写发布目录和配置目录；
- 只写 systemd 分配的状态目录。

### 3.2 目录布局与权限

```bash
sudo install -d -o root -g root -m 0755 /opt/shortlink
sudo install -d -o root -g root -m 0755 /opt/shortlink/releases

sudo install -d -o root -g shortlink -m 0750 /etc/shortlink
sudo install -d -o root -g root -m 0700 /etc/shortlink/credentials

sudo install -d -o shortlink -g shortlink -m 0750 /var/lib/shortlink
```

权限意图：

| 路径 | 所有者 | 模式 | 谁能改 |
|---|---|---:|---|
| `/opt/shortlink/releases` | root:root | 0755 | 只有 root/部署流程 |
| release 目录 | root:shortlink | 0750 | root 写，服务用户读/执行 |
| `/opt/shortlink/current` | root:root | symlink | 只有部署流程切换 |
| `/etc/shortlink/shortlink.env` | root:shortlink | 0640 | root 写，服务用户读 |
| `/etc/shortlink/credentials/*` | root:root | 0600 | 只有 root 读源文件 |
| `/var/lib/shortlink` | shortlink:shortlink | 0750 | 服务可写 |

应用不应把上传文件、临时文件或 SQLite 数据写到 `/opt/shortlink/current`。release 是不可变发布物；可变状态统一进入 `/var/lib/shortlink`、数据库或对象存储。

## 4. 数据库与 Redis 的最小权限边界

MySQL 中至少分离运行账号和迁移账号。

```sql
CREATE DATABASE shortlink
  CHARACTER SET utf8mb4
  COLLATE utf8mb4_0900_ai_ci;

CREATE USER 'shortlink_app'@'127.0.0.1'
  IDENTIFIED BY '通过安全方式设置的随机密码';

GRANT SELECT, INSERT, UPDATE, DELETE
  ON shortlink.* TO 'shortlink_app'@'127.0.0.1';

CREATE USER 'shortlink_migrator'@'127.0.0.1'
  IDENTIFIED BY '另一个随机密码';

GRANT SELECT, INSERT, UPDATE, DELETE,
      CREATE, ALTER, INDEX, DROP, REFERENCES
  ON shortlink.* TO 'shortlink_migrator'@'127.0.0.1';

FLUSH PRIVILEGES;
```

权限集合应根据实际迁移工具收紧。不要给应用账号 `GRANT ALL ON *.*`，也不要让线上服务使用 MySQL root。

若数据库不在同一台机器，`'user'@'host'` 的 host 范围、TLS 和防火墙都要单独设计；不能为了远程连接改成 `'%'` 后再依赖密码碰碰运气。

Redis 在同机部署时至少应：

- 绑定 `127.0.0.1` 或只位于私有容器网络；
- 保持 protected mode；
- 不开放公网 6379；
- 使用 ACL/密码时把凭据放入 systemd credential；
- 明确它是缓存还是数据源。

短链映射的唯一真实数据应优先存 MySQL。Redis 缓存丢失时可以根据 MySQL 重建，运维复杂度会小很多。

## 5. 配置与密钥分离

### 5.1 EnvironmentFile 只放非敏感配置

创建 `/etc/shortlink/shortlink.env`：

```dotenv
APP_ENV=production
GIN_MODE=release
HTTP_ADDR=127.0.0.1:8080
PUBLIC_BASE_URL=https://s.example.com

DB_HOST=127.0.0.1
DB_PORT=3306
DB_NAME=shortlink
DB_USER=shortlink_app
DB_MAX_OPEN_CONNS=30
DB_MAX_IDLE_CONNS=10
DB_CONN_MAX_LIFETIME=30m

REDIS_ADDR=127.0.0.1:6379
REDIS_DB=0

LOG_FORMAT=json
LOG_LEVEL=info
```

```bash
sudo chown root:shortlink /etc/shortlink/shortlink.env
sudo chmod 0640 /etc/shortlink/shortlink.env
```

systemd 的 EnvironmentFile 语法不是完整 shell 脚本：不要在里面写 `export`、命令替换或依赖变量递归展开。配置值若包含空格和特殊字符，应先查当前 systemd 版本的解析规则并测试。

数据库密码、Redis 密码、JWT 私钥、第三方 API token 不放这里。环境变量可能出现在诊断信息、core dump、同权限进程观察或误打日志中，systemd 官方也不建议用环境变量传递高价值 secret。

### 5.2 systemd credentials

创建 root-only 源文件：

```bash
sudo install -m 0600 -o root -g root /dev/null \
  /etc/shortlink/credentials/db_password
sudo install -m 0600 -o root -g root /dev/null \
  /etc/shortlink/credentials/db_migrator_password
sudo install -m 0600 -o root -g root /dev/null \
  /etc/shortlink/credentials/redis_password
```

用交互式方式写入，避免密码进入 shell 历史：

```bash
read -rsp 'DB app password: ' SECRET; echo
printf '%s' "$SECRET" | sudo tee /etc/shortlink/credentials/db_password >/dev/null
unset SECRET
sudo chmod 0600 /etc/shortlink/credentials/db_password
```

其他文件同理。

unit 中通过 `LoadCredential=` 加载后，systemd 会在服务启动时把凭据放进该服务私有的 credentials 目录。`%d` specifier 可在 `ExecStart` 中指向这个目录：

```ini
LoadCredential=db_password:/etc/shortlink/credentials/db_password
ExecStart=/opt/shortlink/current/shortlink-server serve --db-password-file=%d/db_password
```

源文件方案改善了运行期传递和权限隔离，但源文件本身仍是磁盘上的明文。支持的 systemd 版本还可以用 `systemd-creds encrypt` 与 `LoadCredentialEncrypted=`。加密凭据可能绑定本机密钥或 TPM，迁移服务器和灾难恢复前必须测试解密流程，不能只加密后不留恢复方案。

应用读取 secret 后：

- 去掉末尾换行；
- 不输出内容；
- 不把 DSN 整串写日志；
- 文件缺失或为空时直接启动失败；
- 不提供弱默认密码。

## 6. 构建不可变发布物

发布物应在 CI 或受控构建机上生成一次，然后原样分发到服务器。不要在每台生产机 `git pull && go build`，否则工具链、依赖缓存和源码状态都可能不同。

### 6.1 注入版本信息

```bash
VERSION=v0.3.2
COMMIT=$(git rev-parse --verify HEAD)
BUILD_TIME=$(date -u +%FT%TZ)

CGO_ENABLED=0 GOOS=linux GOARCH=amd64 \
go build \
  -trimpath \
  -ldflags="-s -w \
    -X main.version=$VERSION \
    -X main.commit=$COMMIT \
    -X main.buildTime=$BUILD_TIME" \
  -o dist/shortlink-server ./cmd/server
```

如果使用需要 CGO 的依赖，应在与目标系统 ABI 兼容的环境构建并包含所需动态库，不能只删除 `CGO_ENABLED=0` 就假定可运行。

### 6.2 发布包结构

```text
artifact-root/
├── shortlink-server
├── migrations/
│   ├── 0001_init.up.sql
│   └── 0002_add_expire_at.up.sql
└── RELEASE
```

`RELEASE` 可以记录：

```text
version=v0.3.2
commit=d4e5f6a完整提交号
build_time=2026-07-15T10:30:00Z
go_version=go1.x.y
target=linux/amd64
```

打包和生成校验值：

```bash
tar -C artifact-root -czf shortlink-v0.3.2-linux-amd64.tar.gz .
sha256sum shortlink-v0.3.2-linux-amd64.tar.gz \
  > shortlink-v0.3.2-linux-amd64.tar.gz.sha256
```

SHA-256 能检测传输损坏或内容变化，但如果攻击者可以同时替换发布包和 checksum，它不能证明来源。成熟流程还应使用受保护的 CI artifact、签名或具备 provenance 的镜像/制品系统。

在目标机确认架构：

```bash
uname -m
file shortlink-server
sha256sum -c shortlink-v0.3.2-linux-amd64.tar.gz.sha256
```

## 7. systemd 服务单元

创建 `/etc/systemd/system/shortlink.service`：

```ini
[Unit]
Description=Go Shortlink API
After=network-online.target mysql.service redis-server.service
Wants=network-online.target
StartLimitIntervalSec=60
StartLimitBurst=5

[Service]
Type=exec
User=shortlink
Group=shortlink
WorkingDirectory=/opt/shortlink/current

EnvironmentFile=/etc/shortlink/shortlink.env
LoadCredential=db_password:/etc/shortlink/credentials/db_password
LoadCredential=redis_password:/etc/shortlink/credentials/redis_password

ExecStart=/opt/shortlink/current/shortlink-server serve \
    --db-password-file=%d/db_password \
    --redis-password-file=%d/redis_password

Restart=on-failure
RestartSec=3s
TimeoutStartSec=30s
TimeoutStopSec=30s
KillSignal=SIGTERM

UMask=0027
StateDirectory=shortlink
StateDirectoryMode=0750
RuntimeDirectory=shortlink
RuntimeDirectoryMode=0750

NoNewPrivileges=true
PrivateTmp=true
PrivateDevices=true
ProtectSystem=strict
ProtectHome=true
ProtectKernelTunables=true
ProtectKernelModules=true
ProtectKernelLogs=true
ProtectControlGroups=true
ProtectClock=true
RestrictSUIDSGID=true
RestrictRealtime=true
LockPersonality=true
MemoryDenyWriteExecute=true
CapabilityBoundingSet=
AmbientCapabilities=
RestrictAddressFamilies=AF_UNIX AF_INET AF_INET6

[Install]
WantedBy=multi-user.target
```

如果 MySQL/Redis 不由上述本机 unit 管理，就删掉对应 `After=` 名称。`After=` 只声明启动顺序，不保证依赖已经能接受连接；应用仍需有限重试和超时。

安全选项应逐项测试。某些 native 库、调试器或特殊运行方式可能与 `MemoryDenyWriteExecute`、`PrivateDevices` 冲突。出现问题应查明具体需求并最小化放宽，而不是直接删光 hardening。

验证 unit：

```bash
sudo systemd-analyze verify /etc/systemd/system/shortlink.service
sudo systemctl daemon-reload
sudo systemctl enable shortlink.service
sudo systemd-analyze security shortlink.service
```

首次 release 尚未安装时不要立即启动。

### 7.1 单独的迁移 unit

数据库 DDL 不应由每个应用实例在启动时争抢执行。创建 `/etc/systemd/system/shortlink-migrate.service`：

```ini
[Unit]
Description=Run Shortlink Database Migrations
After=network-online.target mysql.service
Wants=network-online.target

[Service]
Type=oneshot
User=shortlink
Group=shortlink
WorkingDirectory=/opt/shortlink/current

EnvironmentFile=/etc/shortlink/shortlink.env
Environment=DB_USER=shortlink_migrator
LoadCredential=db_migrator_password:/etc/shortlink/credentials/db_migrator_password

ExecStart=/opt/shortlink/current/shortlink-server migrate \
    --migrations-dir=/opt/shortlink/current/migrations \
    --db-password-file=%d/db_migrator_password

UMask=0027
NoNewPrivileges=true
PrivateTmp=true
PrivateDevices=true
ProtectSystem=strict
ProtectHome=true
ProtectKernelTunables=true
ProtectKernelModules=true
ProtectControlGroups=true
RestrictSUIDSGID=true
CapabilityBoundingSet=
AmbientCapabilities=
RestrictAddressFamilies=AF_UNIX AF_INET AF_INET6
```

oneshot 迁移成功后退出码为 `0`，失败为非零。不要给它 `Restart=always`；迁移失败需要停下检查，而不是无休止重复 DDL。

### 7.2 `Restart=on-failure` 和退出码 143

需要准确理解：

- `systemctl stop shortlink` 是管理员主动停止，`Restart=on-failure` 不会把它重新拉起。
- 进程异常非零退出、被非“干净”信号终止或超时时，通常会触发重启。
- 启动频繁失败超过 `StartLimitBurst` 后，systemd 会停止继续尝试；修复后可用 `systemctl reset-failed shortlink` 清除失败计数。
- systemd 发送 `SIGTERM` 并不等于进程一定返回退出码 `143`。

`143 = 128 + 15` 是 shell/某些运行时表示“因 SIGTERM 结束”的常见约定。原生 Go 服务捕获 SIGTERM、完成 `Shutdown` 后正常返回，通常退出码是 `0`。因此不应机械添加：

```ini
SuccessExitStatus=143
```

`SuccessExitStatus=` 只是在实际程序确实以 143 退出时，把这个额外状态视作成功；它不会改变 systemd 发什么信号，也不会让 Go 自动优雅退出。对本章的原生 Go 服务通常不需要。

## 8. 数据库迁移必须允许应用回滚

最稳妥的是 expand/contract：

### 阶段 A：Expand

- 增加允许为空的新列；
- 增加新表或新索引；
- 保持旧字段和旧接口仍可用；
- 避免长时间锁表，先在相近数据量环境验证。

### 阶段 B：兼容发布与回填

- 新程序同时兼容新旧结构；
- 必要时双写；
- 后台分批回填，限制批量和锁时间；
- 监控错误率、延迟和复制延迟。

### 阶段 C：Contract

- 确认旧程序不会再回滚；
- 停止读取旧字段；
- 再单独发布删除旧列、约束或索引的迁移。

MySQL 中并非所有 DDL 都能在事务里整体回滚，自动回滚脚本也无法神奇恢复已经删除的数据。发布前的备份和向后兼容迁移是两道不同防线。

## 9. 版本化 release 与原子符号链接

### 9.1 为什么不是覆盖原二进制

直接执行：

```bash
cp new-server /opt/shortlink/shortlink-server
```

会带来几个问题：

- 发布中途只有半个文件或权限不完整；
- 很难知道当前文件对应哪个 commit；
- 旧版本被覆盖，回滚需要重新上传；
- 迁移、静态资源和二进制可能来自不同版本。

每次发布先完整落入独立目录，验收后再切换 `current`。

### 9.2 原子切换原理

不要先删除 `current` 再创建新链接，这会留下一个短暂空窗。正确做法是在同一文件系统、同一目录创建临时链接，再通过 `rename` 替换：

```bash
ln -s /opt/shortlink/releases/新版本 /opt/shortlink/current.next
mv -Tf /opt/shortlink/current.next /opt/shortlink/current
```

Linux 同一文件系统上的 rename 是原子的：观察者看到旧链接或新链接，不会看到半个链接。原子只保证切换动作完整，不保证新版本业务正确，所以后面仍需 readiness 检查和回滚。

## 10. 可自动回滚的发布脚本

下面脚本安装为 `/usr/local/sbin/deploy-shortlink`，由 root 或受控 sudo 运行。它假设发布包根目录直接包含 `shortlink-server`、`migrations/` 和 `RELEASE`。

```bash
#!/usr/bin/env bash
set -Eeuo pipefail

APP=shortlink
BASE=/opt/shortlink
RELEASES=$BASE/releases
CURRENT=$BASE/current
UNIT=shortlink.service
MIGRATE_UNIT=shortlink-migrate.service
HEALTH_URL=http://127.0.0.1:8080/readyz

usage() {
  echo "usage: $0 ARTIFACT.tar.gz EXPECTED_SHA256 RELEASE_ID" >&2
  exit 2
}

[[ $# -eq 3 ]] || usage

ARTIFACT=$(readlink -f "$1")
EXPECTED_SHA=$2
RELEASE_ID=$3

[[ -f "$ARTIFACT" ]] || { echo "artifact not found" >&2; exit 1; }
[[ "$EXPECTED_SHA" =~ ^[0-9a-fA-F]{64}$ ]] || {
  echo "invalid sha256" >&2
  exit 1
}
[[ "$RELEASE_ID" =~ ^[0-9A-Za-z._-]+$ ]] || {
  echo "invalid release id" >&2
  exit 1
}

FINAL=$RELEASES/$RELEASE_ID
NEXT=$BASE/current.next

exec 9>/run/lock/shortlink-deploy.lock
flock -n 9 || { echo "another deployment is running" >&2; exit 1; }

install -d -o root -g root -m 0755 "$RELEASES"
[[ ! -e "$FINAL" ]] || { echo "release already exists: $FINAL" >&2; exit 1; }

echo "$EXPECTED_SHA  $ARTIFACT" | sha256sum -c -

if tar -tzf "$ARTIFACT" | grep -Eq '(^/|(^|/)\.\.(/|$))'; then
  echo "unsafe path found in artifact" >&2
  exit 1
fi

STAGING=$(mktemp -d "$RELEASES/.staging-${RELEASE_ID}.XXXXXX")
cleanup() {
  if [[ -n "${STAGING:-}" && -d "$STAGING" ]]; then
    rm -rf -- "$STAGING"
  fi
  rm -f -- "$NEXT"
}
trap cleanup EXIT

tar --no-same-owner --no-same-permissions -xzf "$ARTIFACT" -C "$STAGING"

if find "$STAGING" -type l -print -quit | grep -q .; then
  echo "symbolic links are not allowed inside the release artifact" >&2
  exit 1
fi

[[ -x "$STAGING/shortlink-server" ]] || {
  echo "shortlink-server missing or not executable" >&2
  exit 1
}
[[ -f "$STAGING/RELEASE" ]] || {
  echo "RELEASE metadata missing" >&2
  exit 1
}

"$STAGING/shortlink-server" version

chown -R root:shortlink "$STAGING"
find "$STAGING" -type d -exec chmod 0750 {} +
find "$STAGING" -type f -exec chmod 0640 {} +
chmod 0750 "$STAGING/shortlink-server"

mv -- "$STAGING" "$FINAL"
STAGING=

PREVIOUS=$(readlink -f "$CURRENT" 2>/dev/null || true)

switch_to() {
  local target=$1
  rm -f -- "$NEXT"
  ln -s "$target" "$NEXT"
  mv -Tf -- "$NEXT" "$CURRENT"
}

wait_until_ready() {
  local attempt
  for attempt in $(seq 1 30); do
    if curl -fsS --max-time 2 "$HEALTH_URL" >/dev/null; then
      return 0
    fi
    sleep 2
  done
  return 1
}

rollback() {
  echo "release failed, rolling back" >&2
  if [[ -n "$PREVIOUS" && -d "$PREVIOUS" ]]; then
    switch_to "$PREVIOUS"
    systemctl restart "$UNIT"
    if wait_until_ready; then
      echo "rollback healthy: $PREVIOUS" >&2
    else
      echo "rollback started but previous release is not ready" >&2
    fi
  else
    systemctl stop "$UNIT" || true
    echo "no previous release exists; service left stopped" >&2
  fi
}

switch_to "$FINAL"

systemctl reset-failed "$MIGRATE_UNIT" 2>/dev/null || true
if ! systemctl start "$MIGRATE_UNIT"; then
  journalctl -u "$MIGRATE_UNIT" -n 100 --no-pager >&2 || true
  rollback
  exit 1
fi

if ! systemctl restart "$UNIT"; then
  journalctl -u "$UNIT" -n 100 --no-pager >&2 || true
  rollback
  exit 1
fi

if ! wait_until_ready; then
  journalctl -u "$UNIT" -n 100 --no-pager >&2 || true
  rollback
  exit 1
fi

echo "release healthy: $FINAL"
systemctl --no-pager --full status "$UNIT"
```

安装和校验：

```bash
sudo install -o root -g root -m 0750 deploy-shortlink \
  /usr/local/sbin/deploy-shortlink
sudo bash -n /usr/local/sbin/deploy-shortlink
```

执行：

```bash
sudo /usr/local/sbin/deploy-shortlink \
  /srv/artifacts/shortlink-v0.3.2-linux-amd64.tar.gz \
  真实的64位SHA256 \
  20260715T103000Z-d4e5f6a
```

### 10.1 这段脚本做了什么

1. 使用 `flock` 阻止两次部署并发执行。
2. 验证 release ID 和 SHA-256。
3. 拒绝 tar 中的绝对路径和 `..` 路径。
4. 解压到 release 目录中的 staging，统一重设所有者和权限。
5. 通过临时 symlink + `mv -T` 原子切换 `current`。
6. 先执行一次迁移，再重启应用。
7. systemd 启动成功后仍轮询 readiness，因为“进程已 exec”不等于“业务已就绪”。
8. 新版本失败时恢复上一 symlink 并重启旧版本。

### 10.2 自动回滚的边界

脚本能回滚：

- 二进制；
- migration 文件；
- 与 release 一起发布的只读资源；
- `current` 指针。

脚本不能自动安全回滚：

- 已经删除或不可逆修改的数据；
- 外部服务配置；
- 已发送的消息、邮件或第三方请求；
- 新版本写入、旧版本无法理解的新数据；
- Nginx、systemd unit 和 `/etc` 中另行修改的配置。

因此“有自动回滚脚本”不等于可以放心执行破坏性 migration。

### 10.3 首次部署与普通发布的差异

首次部署没有 `PREVIOUS`。如果健康检查失败，脚本会停止服务并保留失败 release 供排查，不会假装成功。首次部署前应先：

- 创建用户、目录、配置和 credentials；
- 安装两个 unit 并 daemon-reload；
- 确认 MySQL/Redis 正常；
- 确认 Nginx upstream 指向 `127.0.0.1:8080`；
- 准备可重复执行的初始 migration。

## 12. 日志、指标和告警

Go 服务日志写 stdout/stderr，由 journald 接收：

```bash
journalctl -u shortlink.service -f
journalctl -u shortlink.service --since '10 minutes ago'
journalctl -u shortlink.service -p warning..alert --since today
```

结构化日志至少包含：

- timestamp、level、service、version、commit；
- request_id、method、route、status、latency；
- 错误分类和必要上下文；
- 脱敏的用户/租户 ID。

不要记录：

- 密码、JWT、API key、Cookie；
- 完整 DSN；
- 未脱敏原始长 URL；
- credential 文件内容；
- 创建请求的完整 body。

最小监控指标：

- 请求量、4xx/5xx 比例、p50/p95/p99 延迟；
- Go heap、goroutine、GC、进程 RSS；
- 数据库连接池 in-use/wait count、慢查询；
- Redis 命中率、错误率、延迟；
- 磁盘可用空间、inode、系统负载；
- systemd 重启次数；
- TLS 证书剩余天数；
- 备份是否按时完成、最近一次恢复演练时间。

readiness 失败可以告警，但不要只依赖它：应用可能返回 200，却已经出现高错误率或极慢响应。

## 13. MySQL 备份：生成文件不等于完成备份

先定义两个目标：

- RPO：最多能接受丢失多长时间的数据。例如每日备份意味着最坏可能丢近 24 小时，除非还有 binlog。
- RTO：故障后多快恢复服务。备份文件很大、恢复从未演练，RTO 就只是猜测。

小型短链项目可先用每日逻辑备份，并把副本传到另一台机器或对象存储。随着数据量增长，再引入物理备份和 binlog 时间点恢复。

### 13.1 备份专用账号

权限应根据 MySQL 版本和备份参数确定，通常至少需要对目标库的读取及相关元数据权限。不要直接在脚本使用 root。示意：

```sql
CREATE USER 'shortlink_backup'@'127.0.0.1'
  IDENTIFIED BY '独立随机密码';
GRANT SELECT, SHOW VIEW, TRIGGER, EVENT
  ON shortlink.* TO 'shortlink_backup'@'127.0.0.1';
```

若 `mysqldump` 报 PROCESS、LOCK TABLES 等权限错误，不要立即授予全局 ALL；先确认所用参数、表引擎和 MySQL 版本需要什么最小权限。

建立 root-only 客户端配置 `/etc/shortlink/backup-my.cnf`：

```ini
[client]
host=127.0.0.1
user=shortlink_backup
password=真实备份密码
```

```bash
sudo chown root:root /etc/shortlink/backup-my.cnf
sudo chmod 0600 /etc/shortlink/backup-my.cnf
```

密码不应出现在 `mysqldump -p密码` 命令行，因为其他本机用户可能通过进程列表看到。

### 13.2 备份脚本

`/usr/local/sbin/backup-shortlink-db`：

```bash
#!/usr/bin/env bash
set -Eeuo pipefail

BACKUP_DIR=/var/backups/shortlink/mysql
CNF=/etc/shortlink/backup-my.cnf
STAMP=$(date -u +%Y%m%dT%H%M%SZ)
FINAL=$BACKUP_DIR/shortlink-$STAMP.sql.gz
TMP=$FINAL.tmp

install -d -o root -g root -m 0700 "$BACKUP_DIR"
trap 'rm -f -- "$TMP"' EXIT

mysqldump --defaults-extra-file="$CNF" \
  --single-transaction \
  --quick \
  --routines \
  --triggers \
  --events \
  --hex-blob \
  --no-tablespaces \
  --set-gtid-purged=OFF \
  --default-character-set=utf8mb4 \
  shortlink \
  | gzip -9 > "$TMP"

gzip -t "$TMP"
chmod 0600 "$TMP"
mv -- "$TMP" "$FINAL"
sha256sum "$FINAL" > "$FINAL.sha256"

find "$BACKUP_DIR" -maxdepth 1 -type f \
  -name 'shortlink-*.sql.gz' -mtime +14 -delete
find "$BACKUP_DIR" -maxdepth 1 -type f \
  -name 'shortlink-*.sql.gz.sha256' -mtime +14 -delete

echo "backup created: $FINAL"
```

`--single-transaction` 对 InnoDB 的一致性备份很有用，但不能保证非事务表的一致性，备份期间的 DDL 也可能带来问题。先明确项目全部业务表使用 InnoDB，并把 schema 变更与备份窗口协调好。

脚本只在本机生成备份。随后还必须：

1. 加密或传输到受控的异机/对象存储；
2. 设置独立保留策略，防止攻击者同时删除生产和备份；
3. 监控文件大小、checksum 和上传结果；
4. 定期恢复验证。

### 13.3 systemd timer

`/etc/systemd/system/shortlink-backup.service`：

```ini
[Unit]
Description=Backup Shortlink MySQL Database

[Service]
Type=oneshot
ExecStart=/usr/local/sbin/backup-shortlink-db
User=root
Group=root
Nice=10
IOSchedulingClass=best-effort
IOSchedulingPriority=7
```

`/etc/systemd/system/shortlink-backup.timer`：

```ini
[Unit]
Description=Daily Shortlink MySQL Backup

[Timer]
OnCalendar=*-*-* 03:15:00
Persistent=true
RandomizedDelaySec=10m
Unit=shortlink-backup.service

[Install]
WantedBy=timers.target
```

启用和测试：

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now shortlink-backup.timer
sudo systemctl start shortlink-backup.service
systemctl status shortlink-backup.service --no-pager
systemctl list-timers shortlink-backup.timer
journalctl -u shortlink-backup.service -n 100 --no-pager
```

`Persistent=true` 表示机器错过计划时间后，在下次启动 timer 时补跑一次；它不保证异机上传，也不验证备份能恢复。

## 14. 恢复演练：永远先恢复到新库

不要第一次恢复就覆盖生产数据库。使用隔离 MySQL 实例或创建临时恢复库：

```bash
sha256sum -c shortlink-20260715T031500Z.sql.gz.sha256
gzip -t shortlink-20260715T031500Z.sql.gz

mysql --defaults-extra-file=/etc/shortlink/restore-my.cnf \
  -e "CREATE DATABASE shortlink_restore CHARACTER SET utf8mb4 COLLATE utf8mb4_0900_ai_ci"

gunzip -c shortlink-20260715T031500Z.sql.gz \
  | mysql --defaults-extra-file=/etc/shortlink/restore-my.cnf shortlink_restore
```

恢复后的验证不能只有“命令退出 0”：

- 表数量、migration version 是否正确；
- 关键表行数是否在合理范围；
- 随机抽样短码是否能解析到目标；
- 禁用、过期、唯一约束是否仍成立；
- 字符集和时区是否正确；
- 用测试配置启动应用连接恢复库，完成创建和跳转 smoke test；
- 记录恢复耗时，得到真实 RTO。

最终切换到恢复库前，应冻结或协调生产写入，明确故障时间点、binlog 应用范围和数据差异。大型数据恢复应有单独 runbook，不能临场拼命令。

### 14.1 Redis 恢复策略

若 Redis 纯粹是缓存：

1. 恢复 MySQL；
2. 启动空 Redis；
3. 让应用 cache miss 后回源并逐步重建；
4. 对热点数据可受控预热，避免瞬间压垮 MySQL。

若 Redis 保存唯一计数、队列或 session，则它不再只是缓存，必须明确 AOF/RDB、复制、备份和恢复一致性。不要一边依赖其持久数据，一边按“缓存随便丢”运维。

## 15. release 清理与保留策略

健康发布后至少保留：

- 当前 release；
- 上一 release；
- 最近若干次已验证版本；
- 与数据库 schema 仍兼容的回滚版本。

检查：

```bash
readlink -f /opt/shortlink/current
find /opt/shortlink/releases -mindepth 1 -maxdepth 1 \
  -type d -printf '%TY-%Tm-%Td %TH:%TM %p\n' | sort
```

删除前逐个确认目录不等于 current、没有被回滚流程引用、发布物已经有可信制品副本。不要用模糊通配符直接删除整个 releases，也不要把“磁盘快满了”留到发布时才临时处理。

## 16. 故障时的手工回滚

假设确认上一版本为 `/opt/shortlink/releases/20260710T090000Z-a1b2c3d`：

```bash
sudo ln -s /opt/shortlink/releases/20260710T090000Z-a1b2c3d \
  /opt/shortlink/current.rollback
sudo mv -Tf /opt/shortlink/current.rollback /opt/shortlink/current
sudo systemctl restart shortlink.service

curl -fsS http://127.0.0.1:8080/readyz
systemctl status shortlink.service --no-pager
journalctl -u shortlink.service -n 100 --no-pager
```

手工回滚前确认：

- 旧版本与当前数据库 schema 兼容；
- 当前 release 路径和目标旧路径都正确；
- 问题确实来自应用版本，而不是 MySQL、Redis、磁盘或 Nginx；
- 回滚后继续监控错误率和核心业务，而不是看到 active 就结束。

## 17. 灾难恢复 runbook 骨架

服务器或磁盘完全丢失时，恢复顺序建议固定下来：

1. 新建受支持版本的 Ubuntu 服务器，完成安全更新和时间同步。
2. 恢复 SSH、防火墙、安全组，只开放必要端口。
3. 安装与备份兼容的 MySQL、Redis、Nginx、systemd 环境。
4. 创建 `shortlink` 用户和目录权限。
5. 从异机存储取得配置、加密 credentials 和已验证 release。
6. 恢复 MySQL 到隔离库并验证，再决定切换。
7. Redis 若为缓存则空实例启动；若持久化则按专用流程恢复。
8. 安装 unit，部署 release，先通过本机 readiness。
9. 配置 Nginx 与 TLS，检查证书或重新签发。
10. 通过临时 hosts/测试域名做端到端 smoke test。
11. 修改 DNS 或负载均衡指向，监控错误率和延迟。
12. 记录实际 RPO/RTO、缺失步骤和后续改进。

备份中还应保存或可重建：

- Nginx 站点配置；
- systemd unit 和发布/备份脚本；
- 非敏感 EnvironmentFile；
- 加密后的 secret 或 secret manager 恢复说明；
- 数据库 migration 源码；
- 域名、证书和 DNS 管理权限；
- 发布 artifact 与 checksum/签名。

不要把服务器本机唯一的一份 credentials 和备份称作灾难恢复方案，因为服务器丢失时它们会一起丢失。
