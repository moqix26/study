# 故障排查与面试速查：围绕 Go 短链链路定位问题

排障不是“记住更多命令”，而是用最少的变更回答三个问题：

1. 故障影响了谁、从什么时候开始？
2. 请求在链路的哪一层开始异常？
3. 最近发生了什么变化，证据是否支持它是根因？

短链服务的一次请求路径是：

```text
DNS
  -> TCP / TLS
    -> Nginx
      -> 127.0.0.1:8080
        -> Go / Gin
          -> Redis cache
          -> MySQL
        <- HTTP status / Location
      <- upstream status / latency
    <- HTTPS response
```

最有效的办法是逐层缩短路径：公网失败时先测 Nginx，再测本机 Go，再测依赖。不要看到 502 就重装 Nginx，也不要看到延迟高就先重启整台服务器。

---

## 1. 排障的基本纪律

### 1.1 先观察，再做可逆变更

故障开始时先记录：

- 当前时间和时区；
- 影响范围：所有用户、某地区、某接口、某个短码还是单账号；
- 错误率、状态码和延迟从何时变化；
- 最近发布、配置、证书、DNS、数据库 migration；
- 当前 release、进程 PID、重启次数和资源状态。

```bash
date -Is
readlink -f /opt/shortlink/current
systemctl show shortlink.service \
  -p MainPID -p ActiveState -p SubState -p Result \
  -p ExecMainStatus -p NRestarts -p ActiveEnterTimestamp
```

如果影响严重且刚刚发布，快速回滚可能比完整分析更优先；但仍应在回滚前后保留最小证据：版本号、日志时间窗、状态码、资源快照。否则服务恢复后很难复盘。

### 1.2 不要把这些当第一步

- `kill -9`：不给程序清理连接和刷新日志的机会。
- 重启服务器：同时破坏多个现场状态，且未必修复根因。
- 删除日志：失去证据，还可能删错正在使用的文件。
- 无限制增大 timeout/连接池：可能放大资源耗尽。
- 关闭防火墙、TLS 或认证：会把可用性故障变成安全事件。
- `docker system prune -a --volumes`：可能直接删数据。
- MySQL `KILL` 所有长查询：可能回滚大事务并制造更大压力。

### 1.3 每个动作都要有验证条件

“重启试试”没有判定标准。更好的记录是：

```text
假设：v0.3.2 引入数据库连接泄漏。
证据：发布后 DB wait_count 持续上升；旧版本正常。
动作：原子切回 v0.3.1 并重启服务。
验证：5 分钟内 5xx 恢复、连接池等待下降、核心跳转成功。
```

## 2. 十分钟快速分层检查

### 2.1 系统是否还能正常工作

```bash
uptime
free -h
df -hT
df -ih
vmstat 1 5
```

关注：

- load average 是否突然升高；
- available memory 是否很低、swap 是否频繁换入换出；
- 根分区、日志分区、数据库分区是否满；
- inode 是否耗尽；
- `vmstat` 的 runnable、blocked、I/O wait 是否异常。

load average 不是 CPU 使用率。Linux 中它大致包含可运行任务和处于不可中断睡眠的任务；高 load 可能来自 CPU 饱和，也可能来自大量 I/O 等待。

### 2.2 服务是否存在、是否反复重启

```bash
systemctl status shortlink.service --no-pager --full
systemctl show shortlink.service \
  -p MainPID -p ActiveState -p SubState -p Result \
  -p ExecMainCode -p ExecMainStatus -p NRestarts
journalctl -u shortlink.service --since '15 minutes ago' --no-pager
```

`active (running)` 只说明主进程仍在，不代表 API、MySQL 和 Redis 都健康。

### 2.3 端口和本机 Go 是否可达

```bash
ss -lntp | grep -E ':(80|443|8080)\b'
curl -v --max-time 3 http://127.0.0.1:8080/healthz
curl -v --max-time 3 http://127.0.0.1:8080/readyz
```

结果快速解释：

| 结果 | 初步方向 |
|---|---|
| 8080 没监听 | 应用未启动、崩溃、监听配置错误 |
| live 失败 | HTTP 进程层问题 |
| live 成功、ready 503 | MySQL/Redis 等依赖或 readiness 逻辑 |
| 两者成功，公网失败 | Nginx、TLS、DNS、防火墙 |

### 2.4 Nginx 链路

```bash
sudo nginx -t
curl -v --max-time 5 -H 'Host: s.example.com' http://127.0.0.1/已知短码
curl -v --max-time 5 https://s.example.com/已知短码
sudo tail -n 100 /var/log/nginx/shortlink_error.log
sudo tail -n 100 /var/log/nginx/shortlink_access.log
```

### 2.5 最近是否刚变更

```bash
readlink -f /opt/shortlink/current
cat /opt/shortlink/current/RELEASE
journalctl -u shortlink.service --since today | tail -n 200
```

不要在公共工单或聊天中粘贴可能包含 secret 的 EnvironmentFile、完整进程环境或数据库 DSN。

## 3. 按症状判断故障层

| 用户看到的现象 | 优先检查 | 典型根因 |
|---|---|---|
| 域名无法解析 | DNS | 记录错误、TTL、错误 AAAA |
| TCP 连接超时 | 安全组/防火墙/路由 | 443 未放行、服务未监听、链路丢包 |
| TLS 证书错误 | 证书/SNI/时间 | 证书过期、域名不匹配、链不完整 |
| Nginx 502 | Nginx -> Go | 8080 未监听、upstream 错、应用提前断开 |
| Nginx 504 | Go/依赖耗时 | 慢 SQL、连接池等待、外部调用超时 |
| HTTP 500 | Go 业务层 | panic 被恢复、数据库错误、代码缺陷 |
| HTTP 404 | 路由/短码 | Nginx server block、Gin 路由、短码不存在 |
| HTTP 410 | 业务状态 | 短链过期或被禁用（如果这样设计） |
| HTTP 429 | 限流 | Nginx IP 限流或应用配额 |
| 跳转到旧地址 | 客户端/缓存 | 301 缓存、Redis 未失效、CDN 缓存 |
| 偶发极慢 | 资源/连接池/热点 | GC、慢 SQL、热点 key、锁竞争 |

状态码只是线索，不是根因。一定结合 Nginx 的 `$upstream_status`、`$upstream_response_time` 和同 request ID 的应用日志。

## 4. systemd：状态、退出与重启语义

### 4.1 `active`、`failed` 和业务健康不是一回事

```bash
systemctl is-active shortlink.service
systemctl is-failed shortlink.service
systemctl status shortlink.service --no-pager --full
```

- `active (running)`：主进程存在。
- `activating`：正在启动，可能卡在前置命令或 `Type=notify` readiness。
- `failed`：启动、运行或停止阶段以失败结果结束。
- `inactive (dead)`：当前未运行，可能是正常 stop，也可能是 oneshot 执行结束。

业务健康要用 health endpoint、核心请求和指标验证。

### 4.2 一组比 status 更明确的字段

```bash
systemctl show shortlink.service \
  -p LoadState \
  -p ActiveState \
  -p SubState \
  -p Result \
  -p MainPID \
  -p ExecMainCode \
  -p ExecMainStatus \
  -p NRestarts
```

- `ExecMainCode=exited` 时，`ExecMainStatus` 通常是退出码。
- `ExecMainCode=killed` 时，status 可能对应信号编号。
- `Result=exit-code`、`signal`、`timeout`、`oom-kill` 能帮助分类。

日志看当前启动和指定时间窗：

```bash
journalctl -u shortlink.service -b --no-pager
journalctl -u shortlink.service --since '2026-07-15 10:20:00' \
  --until '2026-07-15 10:30:00' --no-pager
```

### 4.3 `Restart=on-failure` 到底何时重启

它通常在下列情况重启：

- 非零退出码；
- 未被视为正常的信号终止；
- 启动/停止超时；
- watchdog 失败；
- OOM 等失败结果，具体还受 systemd 版本和 OOM 策略影响。

它通常不会因为：

- 程序正常退出码 0；
- 管理员执行 `systemctl stop`；
- 被 systemd 视为干净的退出信号；
- readiness 返回 503 但主进程仍活着。

因此 systemd 不会因为 HTTP 健康检查失败自动知道要重启，除非你另有 watchdog/监控机制。

### 4.4 SIGTERM、143 与 `SuccessExitStatus`

systemd 停止服务时通常先发送 SIGTERM。对非 oneshot 服务，SIGTERM 等若干信号本身可被 systemd 视作干净终止；更推荐 Go 捕获 SIGTERM，完成 `http.Server.Shutdown` 后正常返回 0。

`143` 是 `128 + 15`，常被 shell 或某些包装程序用来表示 SIGTERM。它不是 Linux 内核强制规定所有程序必须返回的退出码。

```ini
SuccessExitStatus=143
```

这行只会把“程序确实退出 143”额外视为成功：

- 不会改变 systemd 发送的信号；
- 不会让程序自动优雅退出；
- 不会修复错误的 wrapper；
- 原生 Go 服务正常处理 SIGTERM 时通常不需要。

### 4.5 启动频率限制

错误配置可能让服务快速失败：

```ini
Restart=on-failure
RestartSec=3s
StartLimitIntervalSec=60
StartLimitBurst=5
```

达到限制后，先查日志、修复配置，再执行：

```bash
sudo systemctl reset-failed shortlink.service
sudo systemctl start shortlink.service
```

不要只反复 `reset-failed`，它不会修复缺失 credential、端口冲突或坏二进制。

### 4.6 三个容易混淆的命令

| 命令 | 作用 |
|---|---|
| `systemctl daemon-reload` | 重新读取 unit 文件，不自动重启服务 |
| `systemctl reload nginx` | 请求服务自身热加载配置，前提是服务支持 |
| `systemctl restart shortlink` | 停止旧进程并启动新进程 |

`After=mysql.service` 只表示顺序，不表示 MySQL readiness，也不会让应用连接自动重试。`Wants`/`Requires` 表达依赖关系，但仍不等于业务可用。

## 5. 端口、连接和 TCP 状态

### 5.1 监听端口

```bash
ss -lntp
ss -lntp '( sport = :8080 )'
```

监听地址的含义：

- `127.0.0.1:8080`：仅 IPv4 本机。
- `[::1]:8080`：仅 IPv6 本机。
- `0.0.0.0:8080`：所有 IPv4 接口。
- `[::]:8080`：所有 IPv6 接口，是否也接受 IPv4 取决于系统配置。

生产短链的 Go 服务在宿主机部署时应只监听 loopback，由 Nginx 暴露公网。

### 5.2 TIME_WAIT 的准确解释

TCP 主动关闭连接的一方通常进入 TIME_WAIT，并在内核中保留一段时间，主要用于：

- 确保最后 ACK 丢失时还能重传；
- 避免旧连接的延迟报文污染相同四元组的新连接。

查看：

```bash
ss -s
ss -tan state time-wait | head
ss -tan state time-wait | wc -l
```

最重要的纠正：TIME_WAIT 通常已经没有对应用户进程持有的文件描述符。应用 close 后，内核保留 TCP 状态和少量控制信息；因此不能简单说“TIME_WAIT 占满了进程 FD”，也很难用 `lsof` 把每个 TIME_WAIT 归到仍存活的进程。

大量 TIME_WAIT 不一定是故障。它可能只是大量短连接正常完成。真正需要继续调查的信号包括：

- 客户端源端口耗尽，出现 `cannot assign requested address`；
- 内核连接跟踪或内存压力；
- Nginx 到 Go/MySQL/Redis 未复用连接；
- QPS 和新建连接速率异常高；
- 对端或本端过早关闭连接。

合理优化顺序：

1. 使用 HTTP keepalive 和数据库连接池；
2. 检查 upstream keepalive 是否真正生效；
3. 避免每个请求创建新的 Redis/HTTP client；
4. 检查超时和谁在主动关闭；
5. 压测确认是否真的发生端口/资源耗尽；
6. 最后才评估内核参数，并在对应内核文档和测试环境验证。

不要一看到 TIME_WAIT 就缩短系统超时、启用来源不明的 `tcp_tw_*` 参数或试图“杀掉 TIME_WAIT”。它是 TCP 正确性机制，不是僵尸进程。

### 5.3 TIME_WAIT 与 CLOSE_WAIT 的区别

| 状态 | 通常说明 | 排查重点 |
|---|---|---|
| TIME_WAIT | 本端主动关闭后的内核等待 | 连接新建速率、复用、源端口压力 |
| CLOSE_WAIT | 对端已关闭，本端应用还没 close | 应用是否泄漏连接/响应体/FD |

大量长期 CLOSE_WAIT 往往比 TIME_WAIT 更值得怀疑应用代码。检查：

```bash
ss -tanp state close-wait
lsof -nP -p 应用PID | wc -l
ls /proc/应用PID/fd | wc -l
cat /proc/应用PID/limits | grep -i 'open files'
```

Go HTTP 客户端必须关闭响应体，能读取完时尽量读取到 EOF 以便连接复用：

```go
resp, err := client.Do(req)
if err != nil {
    return err
}
defer resp.Body.Close()
```

同时复用一个配置好 timeout 的 `http.Client`，不要每次请求都创建新的 Transport。

## 6. DNS、路由、防火墙和 TLS

### 6.1 DNS

```bash
dig A s.example.com
dig AAAA s.example.com
getent ahosts s.example.com
```

常见问题：

- A 记录仍指向旧服务器；
- 配置了不可用 AAAA，部分用户优先 IPv6 失败；
- TTL 尚未过期；
- 不同递归 DNS 缓存不同；
- 本机 `/etc/hosts` 覆盖了真实 DNS。

### 6.2 网络路径和防火墙

```bash
ip addr
ip route
sudo ufw status verbose
ss -lntp | grep ':443'
```

云环境同时检查安全组、网络 ACL 和主机防火墙。连接超时通常表示丢包或无响应；立即 `Connection refused` 通常表示目标可达但端口没有监听或主动拒绝。

### 6.3 TLS

```bash
openssl s_client \
  -connect s.example.com:443 \
  -servername s.example.com \
  -showcerts </dev/null

curl -vI https://s.example.com
sudo certbot certificates
date -u
```

检查：

- 证书 SAN 是否包含域名；
- 有效期和服务器时间；
- full chain 是否完整；
- SNI 是否命中正确 server block；
- Certbot timer 和 dry-run；
- CDN 到源站是否也严格校验证书。

用 IP 直接访问 HTTPS 出现名称不匹配通常是预期现象，不能据此判断域名证书坏了。

## 7. Nginx 502、504、499 与 404

### 7.1 502 Bad Gateway

```bash
curl -v --max-time 3 http://127.0.0.1:8080/healthz
ss -lntp | grep ':8080'
sudo tail -n 100 /var/log/nginx/shortlink_error.log
journalctl -u shortlink.service -n 100 --no-pager
```

典型原因：

- Go 没启动或刚崩溃；
- upstream 端口/协议写错；
- Go 监听地址错误；
- 应用在发出完整响应头前关闭连接；
- Nginx 没权限访问 Unix socket（若使用 socket）；
- 发布切换后 unit 仍指向错误路径。

### 7.2 504 Gateway Timeout

先看访问日志中的：

```text
request_time
upstream_connect_time
upstream_response_time
upstream_status
```

- connect time 高：建立 upstream 连接困难、accept backlog 或网络问题。
- connect 快、response time 高：Go 内部、MySQL、Redis或外部调用慢。
- 应用日志没有对应结束记录：可能阻塞、context 未传播或进程异常。

不要先把 Nginx timeout 放大十倍。先确认慢在哪里，并让内层超时早于外层超时。

### 7.3 499 Client Closed Request

499 是 Nginx 记录的非标准状态，通常表示客户端在 Nginx 返回前断开。例如：

- 用户取消请求；
- 上游负载均衡或客户端 timeout 更短；
- 后端太慢，客户端等不及；
- 网络中断。

少量 499 很常见；突然升高时要对照 upstream latency 和客户端 timeout。不能简单归咎为“用户网络差”。

### 7.4 404 来自哪一层

检查 access log 的 `$upstream_status`：

- 没有 upstream status，可能由 Nginx location/静态文件层直接返回；
- upstream status 是 404，通常是 Gin 路由或业务返回；
- 只有部分域名 404，可能命中默认 server block；
- 某个短码 404，可能真实不存在、大小写归一化错误或读库/缓存不一致。

## 8. Go 进程：崩溃、CPU、内存和 goroutine

### 8.1 进程启动即退出

```bash
systemctl status shortlink.service --no-pager --full
journalctl -u shortlink.service -b -n 200 --no-pager
systemctl show shortlink.service -p Result -p ExecMainStatus -p NRestarts
```

优先检查：

- credential 文件缺失/为空；
- EnvironmentFile 拼写或权限；
- 二进制架构、动态库；
- `current` symlink 目标；
- 8080 被占用；
- migration version 不兼容；
- systemd hardening 拒绝了实际需要的文件/系统调用。

`journalctl` 中若只有“permission denied”，结合 unit 的 `ProtectSystem`、`ProtectHome`、`ReadWritePaths` 和文件目录 execute 权限逐层判断，不要直接把服务改为 root。

### 8.2 CPU 高

```bash
top -H -p 应用PID
pidstat -p 应用PID 1 10
```

Go 服务应在安全条件下暴露仅回环可访问的 pprof，或按需临时启用：

```bash
go tool pprof -top http://127.0.0.1:6060/debug/pprof/profile?seconds=30
```

pprof 不得直接开放公网，其中可能泄露路由、调用栈和运行信息。CPU 高要结合 QPS 判断：流量同比增长可能是容量问题；QPS 不变但 CPU 暴涨更像回归、死循环、序列化或 GC 压力。

### 8.3 内存持续上涨

```bash
free -h
ps -o pid,ppid,%cpu,%mem,rss,vsz,etime,cmd -p 应用PID
journalctl -k --since '1 hour ago' | grep -Ei 'oom|killed process'
```

RSS 是当前驻留物理内存，VSZ/虚拟地址空间大不等于实际占用了同样多 RAM。Go 运行时也可能保留已申请内存用于复用，不能只凭 RSS 单点断言泄漏。

结合：

- heap profile；
- goroutine 数量；
- GC 次数和 pause；
- in-use/alloc_objects；
- 缓存大小和无界 map；
- 请求体/响应体是否及时释放；
- 数据库 rows 是否关闭；
- ticker、timer、channel 是否导致 goroutine 无法退出。

### 8.4 goroutine 泄漏

表现可能是 goroutine 数持续上涨、内存慢涨、连接池耗尽、shutdown 无法结束。常见原因：

- 向无人接收的 channel 发送；
- 外部调用没有 timeout/context；
- ticker 未 Stop；
- 后台 goroutine 没监听取消信号；
- `rows.Close()` 或 HTTP body 未关闭；
- 每个请求永久启动一个 goroutine。

使用 goroutine profile 比随意发送 SIGQUIT 更稳妥。Go 默认收到 SIGQUIT 往往会打印栈并退出，生产中不能把它当无损查看命令。

## 9. 磁盘满与 inode 耗尽

```bash
df -hT
df -ih
sudo du -xhd1 /var | sort -h
sudo journalctl --disk-usage
sudo lsof +L1
```

区别：

- `df -h` 满：数据块空间不足；
- `df -i` 满：大量小文件耗尽 inode，即使还有 GB 也不能新建文件；
- `du` 总和明显小于 `df`：可能是已删除但仍被进程打开的文件，`lsof +L1` 可见。

常见来源：

- journald 或应用日志无上限；
- Nginx 日志轮转失败；
- Docker json 日志、旧镜像、构建缓存；
- MySQL binlog/slow log；
- 本机备份未上传却无限保留；
- 临时文件或 core dump。

处理顺序：

1. 找到具体目录和增长来源；
2. 判断能否安全轮转、归档、上传或删除；
3. 修复保留策略；
4. 验证服务恢复写入；
5. 增加磁盘阈值和增长率告警。

不要对正在写入的数据库文件执行 truncate，也不要随机删除 `/var/lib/mysql` 或 Docker volume。

## 10. MySQL：连接、慢查询与锁

### 10.1 先验证可达性

```bash
mysqladmin --defaults-extra-file=/etc/shortlink/diagnose-my.cnf ping
mysql --defaults-extra-file=/etc/shortlink/diagnose-my.cnf \
  -e 'SELECT NOW(), @@hostname, @@version;'
```

把凭据放在 root-only 配置文件，不写到命令行。

### 10.2 连接池耗尽

应用表现：请求等待、504、`too many connections`、pool wait_count 上升。

```sql
SHOW GLOBAL STATUS LIKE 'Threads_connected';
SHOW GLOBAL STATUS LIKE 'Threads_running';
SHOW VARIABLES LIKE 'max_connections';
SHOW FULL PROCESSLIST;
```

不能只增大 `max_connections`：每条连接消耗内存，过多并发查询可能压垮数据库。还要检查：

- Go `MaxOpenConns` 是否合理；
- 查询/事务是否及时结束；
- 是否忘记 `rows.Close()`；
- 慢 SQL 和锁等待；
- 多个应用实例的连接池总和；
- 是否存在无限重试连接风暴。

### 10.3 慢查询

先定位具体 SQL、执行计划和数据分布：

```sql
EXPLAIN ANALYZE
SELECT target_url, status
FROM links
WHERE code = 'abc123';
```

短码查询通常应有唯一索引。即使有索引，也要检查：

- 字段类型/字符集导致隐式转换；
- 查询条件对列做函数运算；
- 统计信息和数据倾斜；
- 返回不必要大字段；
- 热点行锁或大事务；
- 缓存穿透导致数据库 QPS 暴增。

`EXPLAIN ANALYZE` 会真实执行查询，生产中先评估语句成本，不能对未知重型写语句随意执行。

### 10.4 锁等待

```sql
SHOW ENGINE INNODB STATUS\G
SELECT * FROM performance_schema.data_lock_waits\G
```

记录阻塞者、等待者、事务开始时间和业务来源。杀连接前确认事务内容和回滚成本；一个运行很久的大事务被 kill 后，回滚也可能很久。

## 11. Redis：缓存故障不能拖垮 MySQL

本机验证时通过受控 credential 使用 Redis CLI，避免 `redis-cli -a 密码` 把密码放进命令行历史。

关注：

```text
PING
INFO memory
INFO stats
INFO clients
SLOWLOG GET 20
LATENCY DOCTOR
```

关键指标：

- used_memory、maxmemory、evicted_keys；
- keyspace_hits/keyspace_misses；
- connected_clients、blocked_clients；
- instantaneous_ops_per_sec；
- rejected_connections；
- command latency 和 slowlog。

不要在生产使用 `KEYS *`，它可能阻塞 Redis。需要遍历时用 `SCAN`，仍要控制频率。

短链缓存常见问题：

- 缓存穿透：大量不存在短码每次打 MySQL；可短 TTL 缓存不存在结果并配合限流。
- 缓存击穿：热点 key 过期瞬间大量请求回源；可使用互斥重建、逻辑过期或提前刷新。
- 缓存雪崩：大量 key 同时过期或 Redis 故障；TTL 加随机抖动，应用限流/降级。
- 脏缓存：修改/禁用短链后缓存未删除；更新数据库成功后明确失效策略。
- 热 key：单个短码流量极高；监控、局部缓存、复制和容量规划。

Redis 故障时应用应快速 timeout，而不是每个请求等待数十秒。若 MySQL 无法承受全部回源，应有速率限制和降级策略，不能让缓存故障升级为数据库雪崩。

## 12. 短链业务特有故障

### 12.1 跳转到旧目标

按缓存层次检查：

1. `curl -D -` 看状态码和 `Location`；
2. 是否曾返回 301/308，被浏览器长期缓存；
3. CDN/Nginx 是否缓存 redirect；
4. Redis 中是否是旧值；
5. MySQL 当前值；
6. 更新事务和 cache invalidation 是否成功；
7. 是否读到了延迟副本。

用无缓存的命令行请求与隐身窗口对比，可以帮助区分浏览器缓存，但不能代替服务端证据。

### 12.2 某些短码始终 404

检查：

- 路由大小写是否敏感；
- 编码字符集是否一致；
- 是否误去掉前导字符；
- 生成算法是否碰撞但插入失败后仍返回；
- 数据库唯一索引是否存在；
- 负缓存是否在创建成功后失效；
- Nginx rewrite 是否改写路径。

### 12.3 短链创建成功但无法跳转

创建接口的“成功”必须在事务提交后返回。若先写缓存/先返回，再发生数据库提交失败，就会产生短暂或永久不一致。检查 request ID 下的：

- insert 结果和 commit；
- 返回给客户端的 code；
- cache 写入；
- 跳转查询使用的库和环境；
- `PUBLIC_BASE_URL` 是否指向正确域名。

### 12.4 恶意 URL 与滥用

短链服务会被用于钓鱼、恶意软件和开放重定向滥用。至少：

- 只允许明确 scheme，例如 `http`、`https`；
- 拒绝 `javascript:`、`data:` 等危险 scheme；
- 对创建接口鉴权、限流、审计；
- 建立禁用和举报机制；
- 如果服务端抓取网页预览，必须单独防 SSRF，禁止访问环回、私网、metadata 地址并限制重定向；
- 日志和后台展示长 URL 时做好转义与脱敏。

纯 HTTP redirect 本身不等于 SSRF；服务端主动请求目标 URL 才引入 SSRF 风险。

### 12.5 过期时间异常

数据库统一存 UTC，API 明确时区格式，比较时使用同一时间基准。检查：

```bash
timedatectl
date -u -Is
```

不要依赖服务器本地时区字符串做业务比较，也不要把时间同步故障简单“修正”为修改数据库数据。

## 13. Docker / Compose 快速排障

如果使用第 07 章的容器方案：

```bash
docker compose ps -a
docker compose logs --tail=200 app
docker inspect shortlink-app-1 \
  --format 'status={{.State.Status}} exit={{.State.ExitCode}} oom={{.State.OOMKilled}} restarts={{.RestartCount}}'
docker inspect shortlink-app-1 --format '{{json .State.Health}}'
docker network inspect shortlink_data
```

牢记：

- healthcheck 变成 unhealthy 不会被 Docker Engine 单独自动重启；
- restart policy 主要在 PID 1 退出时触发；
- 容器内 `127.0.0.1` 只指向容器自己；
- MySQL/Redis 服务名应使用 Compose DNS；
- `depends_on` 只帮助启动顺序，不保证运行期依赖永远在线；
- `docker compose down` 默认保留命名卷，`down --volumes` 会删除卷；
- `OOMKilled=true` 要结合内存限制和宿主机日志分析，不能只不断加大限制。

容器内服务监听 `127.0.0.1:8080` 会导致端口发布无法连接；容器内应监听 `0.0.0.0:8080`，宿主机发布地址才限制为 `127.0.0.1`。

## 14. 发布失败的决策顺序

### 14.1 什么时候优先回滚

- 错误紧随新版本发布出现；
- 旧版本和当前数据库 schema 仍兼容；
- 核心业务受影响；
- 现场修复需要较长时间；
- 没有证据表明故障来自独立的数据库/网络事件。

回滚后仍要验证创建、跳转、禁用等核心路径，不是只看进程 active。

### 14.2 什么时候不能盲目回滚

- 新 migration 已删除旧版本依赖的列；
- 新版本写入了旧版本无法解析的数据；
- 故障根因是磁盘满、数据库不可用、证书过期；
- 回滚版本本身有已知严重漏洞；
- 多个系统已完成不可逆协议切换。

这时可能需要前滚修复、恢复数据或先处理依赖故障。

### 14.3 发布脚本为什么仍可能失败

- checksum 不匹配；
- artifact 架构错误或缺动态库；
- release 权限错误；
- migration 锁表/失败；
- credential 缺失；
- `current` 指向不存在目录；
- systemd 启动频率限制；
- readiness 设计过重或 timeout 不合理；
- 旧版本已不兼容当前 schema，自动 rollback 也无法恢复。

## 15. 一次完整故障响应

### 阶段一：确认影响

```text
开始时间、用户范围、接口、地区、状态码、延迟、数据正确性
```

### 阶段二：止损

可选动作：回滚、限流、关闭非核心写入口、切只读、隔离坏实例。每个动作要有负责人和验证指标。

### 阶段三：定位

按 DNS → TLS → Nginx → Go → Redis/MySQL 顺序缩短链路，用 request ID 和统一时间线关联证据。

### 阶段四：恢复

确认错误率、延迟、业务 smoke test、资源、数据一致性均恢复。短暂 200 不等于稳定恢复，应观察一个合理窗口。

### 阶段五：复盘

复盘应回答：

- 事实时间线；
- 用户影响；
- 直接根因和促成因素；
- 为什么现有测试/监控未提前发现；
- 哪个动作缩短或延长了恢复；
- 有负责人和截止时间的改进项。

避免只写“某同学操作失误”。更有价值的是：为什么一次操作能绕过评审、没有自动校验、无法快速回滚或没有告警。

## 16. 高频面试问题与准确答法

### 16.1 Linux 与进程

**问：load average 高就是 CPU 满了吗？**

不一定。它反映一段时间内可运行和部分不可中断等待任务的数量。要结合 CPU utilization、I/O wait、运行队列、磁盘和进程状态判断。

**问：`kill` 和 `kill -9` 有什么区别？**

默认 `kill` 发送 SIGTERM，程序可捕获并优雅退出；SIGKILL 不能被捕获或忽略，由内核立即终止。只有进程不响应、且已评估数据和一致性影响时才用 SIGKILL。

**问：僵尸进程是什么？**

子进程已退出，但父进程尚未 wait 回收退出状态。僵尸几乎不占运行内存，但占 PID 表项。应修复父进程回收逻辑，杀僵尸本身通常没有意义。

**问：文件删了为什么磁盘空间没释放？**

进程仍打开该 inode。目录项删除后，直到最后一个 FD 关闭，数据块才释放。用 `lsof +L1` 定位，优先让服务安全 reopen/restart，而不是随意清空 `/proc/PID/fd`。

**问：755 是什么？**

所有者 `rwx=7`，组和其他用户 `r-x=5`。目录的 `x` 表示可穿越，只有读权限却没有 execute 也无法访问其中条目。

### 16.2 systemd

**问：为什么不用 nohup 跑生产 Go 服务？**

systemd 提供开机启动、身份隔离、restart 策略、日志、资源/安全限制、依赖关系和统一状态管理。nohup 只让进程忽略挂断，缺少完整生命周期管理。

**问：active 是否代表服务健康？**

不代表，只说明 systemd 认为主进程正在运行。还要检查 readiness、核心业务、依赖和指标。

**问：`daemon-reload` 与 restart 的区别？**

daemon-reload 让 systemd 重读 unit 定义，不重启进程；restart 真正停止并重新启动服务。

**问：为什么 Go 服务不一定需要 `SuccessExitStatus=143`？**

143 是某些 shell 对 SIGTERM 的编码。Go 捕获 SIGTERM、完成 shutdown 后可正常退出 0。SuccessExitStatus 只改变 systemd 对实际退出状态的成功判定，不提供优雅退出能力。

### 16.3 网络

**问：`0.0.0.0` 与 `127.0.0.1` 有什么区别？**

前者监听所有 IPv4 接口，后者只允许本机 IPv4。Nginx 前的 Go 服务通常监听 127.0.0.1，降低绕过 TLS/限流直接访问的风险。

**问：TIME_WAIT 为什么存在，会占进程 FD 吗？**

它保护 TCP 最后 ACK 重传和旧报文隔离。连接关闭后状态由内核暂存，通常不再占原进程 FD；大量 TIME_WAIT 要结合短连接率、源端口和内核资源判断，不能等同 FD 泄漏。

**问：TIME_WAIT 与 CLOSE_WAIT 哪个更像应用泄漏？**

长期大量 CLOSE_WAIT 更像本地应用收到对端 FIN 后没有 close socket；TIME_WAIT 更多是主动关闭后的正常内核状态。

**问：连接超时与 refused 有何区别？**

超时常见于丢包、防火墙静默丢弃或路由问题；refused 表示目标主机可达，但该端口无监听或主动拒绝。

### 16.4 Nginx 与 TLS

**问：反向代理解决什么？**

为后端提供统一公网入口、TLS、请求头、超时、限流、日志和一定的隔离；业务鉴权和数据逻辑仍在应用。

**问：502 与 504 区别？**

502 通常是无法建立 upstream 连接或收到无效/提前关闭的响应；504 通常是已等待 upstream 但超时。都要用本机 curl、端口、Nginx upstream 日志和应用日志验证。

**问：为什么短链默认不建议 301？**

301/308 可能被客户端和中间层长期缓存，使目标修改、禁用、风控和统计失效。可变短链通常先用 302/307。

**问：如何拿到真实客户端 IP？**

只有直接连接地址或来自明确可信代理的真实 IP 头可信。配置受信任代理地址段，再让 Nginx/Gin解析；不能信任全网提供的 X-Forwarded-For。

### 16.5 Docker

**问：容器和虚拟机区别？**

容器共享宿主机内核，用 namespace/cgroup 隔离进程和资源；虚拟机运行独立 guest kernel。容器启动快、交付一致，但不是绝对安全边界。

**问：镜像、容器、volume 的关系？**

镜像是只读模板，容器是其运行实例，volume 独立保存持久数据。删除容器不应丢数据库，但删除 volume 会。

**问：unhealthy 会自动重启吗？**

Docker Engine 单凭 healthcheck 变为 unhealthy 通常不会自动重启；restart policy 主要响应主进程退出。编排器可依据健康状态替换实例。

**问：为什么 docker 组权限很高？**

控制 daemon 的用户可启动特权容器、挂载宿主机目录等，通常能获得接近 root 的控制力。

### 16.6 Go 服务运维

**问：liveness 与 readiness 区别？**

liveness 判断进程自身是否活着，不应依赖数据库；readiness 判断当前是否适合接流量，可检查关键依赖但必须短超时。依赖故障时可能 live 200、ready 503。

**问：如何优雅发布 Go HTTP 服务？**

捕获 SIGTERM，停止接收新请求，给在途请求有限时间，关闭数据库/缓存资源并退出 0；发布侧等待 readiness，不只看进程启动。

**问：数据库连接池越大越好吗？**

不是。单实例池大小乘实例数不能超过数据库承载；太大会增加内存、调度和查询竞争。应结合 DB 最大连接、查询耗时、并发和 wait 指标压测设置。

## 17. 三道场景题的口述框架

### 场景一：线上突然大量 502

可以这样回答：

1. 确认影响范围、开始时间和是否刚发布。
2. 看 Nginx access/error log 的 upstream 错误。
3. 本机 curl `127.0.0.1:8080/healthz`，看端口监听。
4. 查 systemd 状态、退出码、重启次数和应用日志。
5. 如果紧随发布且旧版兼容，先回滚止损。
6. 若应用健康，再检查 upstream 配置、权限、连接提前关闭。
7. 恢复后验证核心短链创建/跳转并复盘监控缺口。

### 场景二：接口延迟升高但 CPU 不高

1. 看 p95/p99、受影响路由和 request ID。
2. 对比 Nginx request/upstream time，区分入口与后端。
3. 看 Go 连接池 wait、goroutine、外部调用 timeout。
4. 看 MySQL 慢查询、锁等待、连接数；看 Redis 延迟和命中率。
5. 检查磁盘 I/O wait、网络和依赖服务。
6. CPU 不高可能是等待锁、I/O、连接池或网络，不代表程序空闲健康。

### 场景三：内存持续增长最终 OOM

1. 确认是进程 RSS、Go heap 还是 page cache，查内核 OOM 日志。
2. 对比流量、版本和增长速率。
3. 采集 heap/goroutine profile，检查无界缓存、goroutine、body/rows、timer。
4. 短期可回滚或限流止损，不把无限加内存当修复。
5. 修复后用相同压测验证内存是否达到稳定平台，并添加趋势告警。

口述时始终包含“确认影响 → 分层证据 → 可逆止损 → 根因修复 → 验证复盘”，会比堆砌命令更像真实工程师。

## 19. 建议完成的故障演练

所有演练都在本地/测试服务器进行，并提前准备恢复动作。

### 演练一：端口错误导致 502

把测试 Nginx upstream 改到未监听端口，观察 error log、本机 curl 和 502，再恢复配置并 reload。

### 演练二：readiness 与 liveness 分离

停止测试 Redis，确认 live 仍成功、ready 失败，业务请求快速失败或受控回源；恢复 Redis 后验证自动恢复。

### 演练三：systemd crash loop

使用缺少必要配置的测试 unit，让服务触发 StartLimit。查看 `Result`、`ExecMainStatus`、`NRestarts`，修复后 reset-failed。

### 演练四：坏版本自动回滚

发布启动失败的测试 release，观察 symlink、journal、health loop 和旧版本恢复。确认数据库 migration 没有破坏兼容性。

### 演练五：恢复备份

从最新 MySQL 备份恢复到隔离库，用另一端口启动短链服务并验证随机短码。记录真实恢复耗时。

## 20. 学完本章应达到的标准

你应该能在没有“万能重启”的情况下完成：

- 从公网失败逐层缩短到 Nginx、Go、Redis 或 MySQL；
- 准确解释 systemd active、退出码、restart 和 start limit；
- 准确解释 TIME_WAIT 不等于进程 FD 泄漏，并区分 CLOSE_WAIT；
- 用日志和 request ID 判断 502、504、499、404 来源；
- 排查 CPU、内存、磁盘、inode、连接池和慢查询；
- 判断何时回滚、何时回滚会因 schema 不兼容而更危险；
- 完整口述一次故障的确认、止损、定位、恢复和复盘。

真正的面试速查不是背出几十条命令，而是面对一个现象时，能说清下一条最有信息量、风险最小的检查是什么。
