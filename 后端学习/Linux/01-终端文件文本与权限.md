# 01 终端、文件、文本与权限

后端开发的大量工作都发生在终端：进入项目目录、查看配置、搜索日志、复制构建产物、修改权限。真正需要掌握的不是几十条孤立命令，而是四个模型：

1. Shell 如何解析一行命令。
2. Linux 如何组织路径与文件。
3. 标准输入、标准输出和管道如何连接程序。
4. 用户、组与权限如何决定一次访问是否被允许。

本章所有普通实验都放在 `~/linux-lab/01-files`，不要在系统目录里练习删除或递归改权限。

## 1. 终端、Shell 和命令不是一回事

终端是你输入和查看文本的界面；Shell 是读取命令、展开变量、处理重定向并启动程序的解释器；`ls`、`rg`、`go` 则是 Shell 启动的命令。

查看当前 Shell：

```bash
printf '%s\n' "$SHELL"
ps -p $$ -o pid,comm,args
```

Ubuntu 默认交互 Shell 通常是 Bash。本手册的脚本语法以 Bash 为准。

一条命令通常由以下部分组成：

```text
command  option              argument
ls       -la                 /etc
rg       -n --hidden         'listen' .
```

选项调整行为，参数指出操作对象。选项和参数的具体含义必须以该命令的帮助为准：

```bash
ls --help | less
man ls
```

在 `man` 中按 `/pattern` 搜索、`n` 找下一个、`q` 退出。不会某个低频选项时查帮助，比凭记忆猜更可靠。

### 1.1 Shell 会先处理引号和展开

```bash
name='shortlink'
printf '%s\n' "$name"
printf '%s\n' '$name'
```

输出：

```text
shortlink
$name
```

- 双引号允许 `$name`、`$(command)` 展开，同时把展开结果保留为一个参数。
- 单引号把内部内容视为普通字符。
- 不加引号的变量可能被空格拆分，还可能触发通配符展开。

因此在 Shell 脚本中，路径变量通常要写成：

```bash
rm -- "$target"
```

而不是：

```bash
rm $target
```

即便加了引号，删除前仍必须验证目标是否正确。引号只能防止错误拆词，不能阻止你删除一个“内容正确但目标选错”的目录。

## 2. 文件系统是一棵树

Linux 没有 `C:`、`D:` 这样的盘符入口。所有文件系统都挂载到以 `/` 为根的一棵目录树中。

常见目录：

| 路径 | 作用 | 与 Go 部署的关系 |
|---|---|---|
| `/home/<user>` | 普通用户家目录 | 日常代码和实验放这里 |
| `/root` | root 的家目录 | 不作为普通项目目录 |
| `/etc` | 系统和服务配置 | 可放 `/etc/shortlink/shortlink.env` |
| `/opt` | 可选的第三方应用 | 可放 `/opt/shortlink` 二进制与版本 |
| `/var/lib` | 服务持久状态 | 应用需要本地状态时使用 |
| `/var/log` | 系统和服务日志 | systemd 日志主要由 journald 管理 |
| `/run` | 本次启动期间的运行时数据 | PID、socket 等，重启后不保证保留 |
| `/tmp` | 临时文件 | 可能被清理，不放重要数据 |
| `/usr/bin` | 发行版管理的常用命令 | `ls`、`curl` 等通常在这里 |
| `/proc` | 内核提供的进程和系统视图 | `/proc/<PID>` 可观察运行进程 |
| `/dev` | 设备和特殊文件 | `/dev/null`、磁盘、终端等 |

用以下命令查看文件系统和挂载：

```bash
findmnt
df -hT
```

`df` 关注文件系统容量，`du` 关注目录实际占用：

```bash
du -sh ~/linux-lab
du -h --max-depth=1 ~/linux-lab | sort -h
```

二者结果不同不一定是错误；已删除但仍被进程打开的文件、文件系统保留空间等都会造成差异。

## 3. 绝对路径、相对路径和当前位置

绝对路径从 `/` 开始：

```text
/home/honor/code/shortlink/main.go
```

相对路径从当前工作目录开始：

```text
./main.go
../config/app.env
```

几个高频符号：

- `.`：当前目录。
- `..`：上一级目录。
- `~`：当前用户家目录，由 Shell 展开。
- `-`：在 `cd -` 中表示上一次工作目录。

练习：

```bash
mkdir -p ~/linux-lab/01-files/{inbox,archive,logs}
cd ~/linux-lab/01-files
pwd
cd inbox
pwd
cd -
```

`pwd -P` 显示解析符号链接后的物理路径，`pwd -L` 保留逻辑路径：

```bash
pwd -L
pwd -P
```

自动化脚本中不要依赖“调用者刚好在哪个目录”。应明确计算脚本目录或使用绝对路径，第 05 章会进一步处理。

## 4. 查看目录与文件元数据

```bash
ls
ls -la
ls -lh
ls -ld . inbox
```

- `-l` 显示类型、权限、链接数、所有者、组、大小和时间。
- `-a` 包含以 `.` 开头的隐藏文件。
- `-h` 让大小更易读。
- `-d` 查看目录本身，而不是列出目录内容。

典型长格式：

```text
-rw-r----- 1 honor developers 128 Jul 15 10:20 app.env
```

从左到右是：

```text
-       普通文件；d 是目录，l 是符号链接
rw-     所有者权限
r--     所属组权限
---     其他用户权限
1       硬链接计数
honor   所有者
developers 所属组
128     字节数
```

查看更完整的元数据：

```bash
stat inbox
file /bin/ls
```

Linux 文件名区分大小写，`README.md` 与 `readme.md` 是两个名字。以 `.` 开头只表示常规列表默认隐藏，不是安全保护。

## 5. 创建、复制、移动和删除

### 5.1 创建文件和目录

```bash
mkdir data
mkdir -p releases/2026-07-15/bin
touch logs/app.log
printf 'PORT=8080\n' > inbox/app.env
```

`touch` 在文件不存在时创建空文件，存在时主要更新时间戳。创建带内容的文本时，`printf` 比依赖不同实现细节的 `echo` 更可控。

### 5.2 复制

```bash
cp inbox/app.env archive/app.env.bak
cp -a releases archive/releases-copy
```

`cp -a` 递归复制并尽量保留权限、时间戳和符号链接，适合备份目录结构。复制到已存在目标前先查看：

```bash
ls -ld archive archive/app.env.bak
```

交互练习可使用 `cp -i` 询问覆盖，但脚本不能依赖人工确认；可靠脚本应先检查目标、生成唯一版本目录或原子替换。

### 5.3 移动与改名

```bash
mv inbox/app.env inbox/shortlink.env
mv inbox/shortlink.env archive/
```

同一文件系统内移动通常只是更新目录项，速度很快；跨文件系统移动需要复制后删除，失败边界不同。

### 5.4 删除

```bash
rm archive/app.env.bak
rmdir data
```

`rmdir` 只删除空目录。递归删除：

```bash
rm -r releases/2026-07-15
```

`rm -rf` 会递归并忽略大量提示，必须谨慎。安全流程：

```bash
target="$HOME/linux-lab/01-files/archive/releases-copy"
printf 'target=%q\n' "$target"
test -d "$target"
find "$target" -maxdepth 2 -print
# 确认输出无误后，才执行：rm -rf -- "$target"
```

这里的 `--` 表示选项结束，避免名为 `-rf` 一类的文件被解释成选项。处理以连字符开头的名字也可显式写相对路径：

```bash
touch ./-draft
rm -- ./-draft
```

不要对变量可能为空、可能为 `/`、可能越出工作区的路径直接执行递归删除。生产发布脚本还应校验解析后的绝对路径和允许前缀。

## 6. 通配符不是正则表达式

Shell 会在命令运行前展开通配符：

- `*`：任意长度字符，默认不匹配文件名开头的 `.`。
- `?`：任意一个字符。
- `[0-9]`：一个指定范围内的字符。

```bash
touch README.md notes.md logs/app1.log logs/app2.log
printf '%s\n' *.md
printf '%s\n' logs/app*.log
```

危险点是“匹配范围可能比想象大”。执行删除前先用相同通配符配合 `printf` 或 `ls -d --` 查看：

```bash
printf '<%s>\n' ./logs/*.log
```

当没有匹配时，Bash 默认可能把原始模式原样传给命令。脚本若依赖通配符，应明确处理无匹配情况，不要凭空假设。

## 7. 链接：一个文件可以有多个入口

### 7.1 硬链接

```bash
printf 'v1\n' > original.txt
ln original.txt hardlink.txt
ls -li original.txt hardlink.txt
```

两者 inode 相同，指向同一份文件数据。删除其中一个名字，另一个仍可访问。硬链接通常不能跨文件系统，也通常不对目录创建。

### 7.2 符号链接

```bash
mkdir -p releases/v1
printf 'binary placeholder\n' > releases/v1/shortlink
ln -s releases/v1 current
ls -l current
readlink current
readlink -f current
```

符号链接保存的是另一个路径，可跨文件系统，也可指向目录。目标被移动或删除后会成为断链。

发布系统常使用：

```text
/opt/shortlink/releases/20260715-102000/
/opt/shortlink/current -> releases/20260715-102000
```

切换 `current` 可以快速指向新版本，但真正可靠的原子切换、旧进程重启和回滚将在发布章节实现。

## 8. 查看和编辑文本

### 8.1 小文件与大文件

```bash
cat logs/app.log
less logs/app.log
```

`cat` 适合短文件或把内容送入管道；大日志优先用 `less`。在 `less` 中：

- `/error`：向下搜索。
- `?error`：向上搜索。
- `n` / `N`：下一个 / 上一个匹配。
- `G`：文件末尾。
- `g`：文件开头。
- `q`：退出。

查看开头、结尾和持续追加：

```bash
head -n 20 logs/app.log
tail -n 50 logs/app.log
tail -F logs/app.log
```

`tail -F` 会在日志轮转后尝试重新打开同名文件；按 `Ctrl+C` 停止。systemd 服务通常使用 `journalctl -f`，而不是直接 tail 应用文件。

### 8.2 统计和格式

```bash
wc -l logs/app.log
file logs/app.log
nl -ba logs/app.log | less
```

`nl -ba` 给所有行编号，适合讨论配置的具体行。检查不可见字符：

```bash
sed -n '1,10l' deploy.sh
```

若 Shell 脚本第一行出现 `\r$` 或错误中出现 `$'\r'`，通常是 Windows CRLF 换行，应在编辑器中转换为 LF。

### 8.3 编辑器

对新手，`nano` 足够修改服务器上的小配置：

```bash
nano notes.txt
```

底部的 `^O` 表示 `Ctrl+O` 保存，`^X` 表示 `Ctrl+X` 退出。正式修改关键配置前先备份，并使用对应工具做语法检查。不要为了编辑一个文件而把整个目录权限改成可写。

## 9. 搜索文件内容

本机已安装 `ripgrep` 时优先使用 `rg`：

```bash
rg -n 'ERROR|WARN' logs
rg -n -i 'timeout' .
rg -n --hidden -g '!vendor/**' -g '!*.bin' 'localhost' .
```

常用含义：

- `-n`：显示行号。
- `-i`：忽略大小写。
- `--hidden`：包含隐藏文件。
- `-g`：用 glob 包含或排除路径。

`rg` 默认递归、速度快，并会尊重部分忽略规则。查 Git 仓库代码很方便。

服务器最小环境可能没有 `rg`，`grep` 更普遍：

```bash
grep -nE 'ERROR|WARN' logs/app.log
grep -RIn --exclude-dir=.git 'localhost' .
```

固定字符串中含正则特殊字符时，可用：

```bash
grep -Fn '127.0.0.1:8080' app.conf
rg -n -F '127.0.0.1:8080' .
```

不要把日志中的用户输入直接拼接成未经处理的正则表达式。

## 10. 按名称和属性查找文件

`find` 搜索的是目录树：

```bash
find . -maxdepth 3 -type f -name '*.log' -print
find . -type f -size +10M -print
find . -type f -mtime -1 -print
```

`-name '*.log'` 必须加引号，否则 Shell 可能在 `find` 启动前展开模式。

对每个结果执行命令时，优先使用 `-exec`：

```bash
find logs -type f -name '*.log' -exec wc -l {} +
```

若使用 `xargs`，要处理文件名中的空格和换行：

```bash
find logs -type f -name '*.log' -print0 | xargs -0 wc -l
```

删除之前先把 `-delete` 替换为 `-print` 检查范围：

```bash
find logs -type f -name '*.tmp' -print
# 范围确认后才考虑：find logs -type f -name '*.tmp' -delete
```

`locate` 依赖定期更新的索引，可能找不到刚创建的文件；`find` 直接遍历当前目录树，结果更实时。

## 11. 标准流、重定向与管道

每个常规进程启动时通常有三个标准文件描述符：

| 编号 | 名称 | 默认去向 |
|---|---|---|
| `0` | stdin 标准输入 | 终端键盘 |
| `1` | stdout 标准输出 | 终端屏幕 |
| `2` | stderr 标准错误 | 终端屏幕 |

### 11.1 覆盖、追加与错误输出

```bash
printf 'first\n' > logs/demo.log
printf 'second\n' >> logs/demo.log
ls /path/does-not-exist 2> logs/error.log
```

- `>` 在命令运行前截断目标文件。
- `>>` 追加。
- `2>` 只重定向标准错误。

把 stdout 与 stderr 都写入同一文件：

```bash
some-command > logs/all.log 2>&1
```

顺序很重要：`2>&1` 表示让 stderr 指向“此时 stdout 指向的位置”。Bash 也支持：

```bash
some-command &> logs/all.log
```

丢弃不需要的输出：

```bash
some-command > /dev/null 2>&1
```

不要为了让报错“消失”而默认丢弃 stderr。自动化中应保留足够日志，才能判断失败原因。

### 11.2 管道

管道把左侧 stdout 连接到右侧 stdin：

```bash
ps aux | rg '[s]hortlink'
journalctl -u shortlink --since today | rg -i 'error|timeout'
```

管道不会自动传递 stderr，除非显式合并。交互命令中可以逐段验证：

```bash
journalctl -u shortlink --since today --no-pager
journalctl -u shortlink --since today --no-pager | rg -i 'error|timeout'
```

Bash 默认以管道最后一个命令的退出状态作为整个管道状态，可能掩盖前面命令失败。可靠脚本通常启用 `set -o pipefail`，第 05 章会详细讲。

### 11.3 `tee`：既显示又写文件

```bash
printf 'hello\n' | tee logs/tee.log
printf 'again\n' | tee -a logs/tee.log
```

重定向由当前 Shell 执行，所以这条命令通常无权写系统文件：

```bash
sudo printf 'value\n' > /etc/example.conf
```

正确思路是让具有权限的 `tee` 执行打开文件的动作：

```bash
printf 'value\n' | sudo tee /etc/example.conf > /dev/null
```

这只是语法说明，不要创建无用途的 `/etc/example.conf`。修改真实配置前应确认路径、备份和语法验证方式。

## 12. Here document：多行文本与是否展开

未引用的结束标记会展开变量和命令替换：

```bash
name='shortlink'
cat <<EOF
service=$name
generated_at=$(date --iso-8601=seconds)
EOF
```

引用结束标记则保留字面内容：

```bash
cat <<'EOF'
service=$name
generated_at=$(date --iso-8601=seconds)
EOF
```

第二段输出中会保留 `$name` 和 `$(date ...)`。选择哪一种取决于意图：

- 要生成包含当前变量值的配置，使用未引用标记，但必须审视所有 `$` 和反斜杠。
- 要生成脚本模板或原样文本，使用引用标记。

不要一边使用 `<<'EOF'`，一边期待命令替换自动执行；这是常见错误。

## 13. 用户、组与 sudo

确认身份：

```bash
whoami
id
groups
```

Linux 权限判断主要使用数值 UID 和 GID，用户名和组名是便于人阅读的映射。查看账户记录：

```bash
getent passwd "$USER"
getent group sudo
```

`/etc/passwd` 不保存明文密码。密码哈希通常在只有 root 可读的 `/etc/shadow`，也不应随意读取或复制。

`sudo` 以受控方式执行需要提升权限的命令：

```bash
sudo systemctl status ssh
```

原则：

- 日常编辑、编译和 Git 不使用 sudo。
- 只给需要修改系统状态的具体命令提权。
- 不要用 `sudo go build`，否则构建缓存和产物可能归 root 所有。
- 不要长期使用 `sudo -i` 后忘记自己仍是 root。

查看 sudo 是否仍缓存认证：

```bash
sudo -v
sudo -k
```

`sudo -k` 使当前认证时间戳失效，适合在敏感操作后主动结束授权缓存。

## 14. `rwx` 对文件和目录的含义不同

### 14.1 普通文件

- `r`：读取内容。
- `w`：修改内容。
- `x`：把文件作为程序执行。

### 14.2 目录

- `r`：列出目录项名字。
- `w`：在目录中创建、删除或改名目录项。
- `x`：穿过目录，访问其中已知名字的对象。

目录权限最容易被误解。即使文件本身可读，只要路径上的某一级目录没有执行位，也无法访问。反过来，删除文件主要取决于父目录的写和执行权限，而不是文件自身写权限。

用 `namei` 检查整条路径每一级权限：

```bash
namei -l ~/linux-lab/01-files/logs/app.log
```

## 15. 修改权限：符号形式与八进制

符号形式更容易表达意图：

```bash
chmod u=rw,g=r,o= archive/app.env
chmod u+x script.sh
chmod go-rwx secret.env
```

八进制把 `r=4`、`w=2`、`x=1` 相加：

| 数字 | 权限 |
|---|---|
| `7` | `rwx` |
| `6` | `rw-` |
| `5` | `r-x` |
| `4` | `r--` |
| `0` | `---` |

常见合理值：

```bash
chmod 600 secret.env       # 仅所有者读写
chmod 640 app.env          # 所有者读写、同组只读
chmod 644 README.md        # 所有人可读，只有所有者可写
chmod 750 deploy.sh        # 所有者完全权限、同组读和执行
chmod 755 public-dir       # 可遍历和读取，只有所有者可写
```

不要用 `chmod -R 777` 解决权限问题。它会掩盖真正的所有者或目录设计错误，并允许任何本机用户修改文件。正确做法是明确：哪个进程用户需要对哪个路径执行哪种操作。

### 15.1 所有者和所属组

```bash
sudo chown "$USER":developers archive/app.env
sudo chgrp developers archive/app.env
```

递归 `chown -R` 会影响整个子树，部署目录中可能同时存在需要不同权限的配置、二进制和数据目录，不能无脑统一。

团队共享目录可使用 setgid 位，让新文件继承目录所属组：

以下示例假设管理员已经创建了 `developers` 组：

```bash
sudo install -d -o "$USER" -g developers -m 2775 /srv/team-project
```

`2` 表示目录 setgid。新文件最终权限还受创建进程的 `umask` 影响。

## 16. `umask`：新文件权限的默认扣除项

查看当前值：

```bash
umask
umask -S
```

普通文件通常从 `666` 扣除 umask，因为新文本默认不带执行位；目录从 `777` 扣除。例如 `umask 022` 通常得到：

- 新文件：`644`。
- 新目录：`755`。

验证：

```bash
old_umask=$(umask)
umask 027
touch private-file
mkdir private-dir
ls -ld private-file private-dir
umask "$old_umask"
```

预期分别接近 `640` 和 `750`。恢复原 umask 很重要，否则同一终端后续创建文件都会受影响。

systemd unit 可以使用 `UMask=0027` 明确服务创建文件时的默认权限。

## 17. 可执行脚本的三个条件

一个 Shell 脚本能直接执行，通常需要：

1. 正确的解释器行。
2. Linux LF 换行。
3. 执行权限。

示例内容：

```bash
#!/usr/bin/env bash
set -euo pipefail
printf 'hello from %s\n' "$(hostname)"
```

保存为 `hello.sh` 后：

```bash
chmod u+x hello.sh
./hello.sh
```

若不设置执行位，也可显式让 Bash 读取：

```bash
bash hello.sh
```

`./hello.sh` 中的 `./` 表示当前目录。Linux 通常不会默认从当前目录搜索程序，这是为了避免误执行同名恶意文件。

## 18. Go 服务的目录与权限设计

开发阶段无需复杂目录：

```text
~/code/shortlink/
├── cmd/
├── internal/
├── go.mod
└── README.md
```

正式部署时，职责应分开：

```text
/opt/shortlink/current/shortlink        root 部署，服务用户只读和执行
/etc/shortlink/shortlink.env            root 管理，shortlink 组只读
/var/lib/shortlink/                     服务用户可写（确有本地状态时）
```

一个合理的所有权示例：

```text
-rwxr-x--- root shortlink /opt/shortlink/current/shortlink
-rw-r----- root shortlink /etc/shortlink/shortlink.env
drwxr-x--- shortlink shortlink /var/lib/shortlink
```

这样即使应用进程被利用，它通常也不能替换自己的二进制或修改配置；它只对业务确实需要写入的状态目录拥有写权限。

不要让 Go 服务以 root 运行来绕过权限设计。监听 `1024` 以上端口（如 `8080`）不需要 root，公网 `80/443` 由 Nginx 处理。

## 19. 综合实验：建立一个安全的短链工作区

### 19.1 创建结构和样例文件

```bash
cd ~/linux-lab/01-files
mkdir -p shortlink/{bin,config,data,logs}
printf 'APP_ADDR=127.0.0.1:8080\nLOG_LEVEL=info\n' > shortlink/config/app.env
printf '2026-07-15T10:00:00+08:00 INFO request_id=r1 path=/healthz status=200\n' > shortlink/logs/app.log
printf '2026-07-15T10:00:01+08:00 WARN request_id=r2 path=/api/links status=429\n' >> shortlink/logs/app.log
printf '2026-07-15T10:00:02+08:00 ERROR request_id=r3 mysql timeout\n' >> shortlink/logs/app.log
```

### 19.2 设置权限

```bash
chmod 750 shortlink
chmod 750 shortlink/{bin,config,data,logs}
chmod 640 shortlink/config/app.env
chmod 640 shortlink/logs/app.log
find shortlink -maxdepth 2 -printf '%M %u:%g %p\n'
```

注意：这里用当前用户模拟服务所有者，只练权限含义；真实部署会创建独立 `shortlink` 系统用户。

### 19.3 搜索和统计

```bash
rg -n 'WARN|ERROR' shortlink/logs
rg -o 'status=[0-9]+' shortlink/logs/app.log | sort | uniq -c
wc -l shortlink/logs/app.log
```

预期能找到两条异常级别日志，文件共三行。

### 19.4 验证重定向

```bash
{
  printf 'generated_at=%s\n' "$(date --iso-8601=seconds)"
  printf 'file_count=%s\n' "$(find shortlink -type f | wc -l)"
  printf 'total_size=%s\n' "$(du -sh shortlink | awk '{print $1}')"
} | tee shortlink/report.txt
```

检查报告：

```bash
cat shortlink/report.txt
```

### 19.5 制造并定位权限故障

移除配置目录的执行位：

```bash
chmod u-x shortlink/config
cat shortlink/config/app.env
```

如果当前用户仍因其他权限或特殊环境能够读取，可换一个权限更严格的实验目录；常规情况下会得到 `Permission denied`。检查路径：

```bash
namei -l shortlink/config/app.env
```

恢复：

```bash
chmod u+x shortlink/config
cat shortlink/config/app.env
```

这个实验说明：目标文件拥有读权限还不够，路径上的目录必须允许穿过。

## 20. 权限故障的排查顺序

看到 `Permission denied`，按证据检查：

```bash
id
namei -l /path/to/target
stat /path/to/target
findmnt -T /path/to/target
```

依次回答：

1. 当前进程实际 UID/GID 是谁？
2. 路径每一级目录是否有执行权限？
3. 目标文件的所有者、组和模式是什么？
4. 文件系统是否只读挂载，或来自 Windows/网络文件系统？
5. 是否存在 ACL、AppArmor 或 systemd 沙箱等额外限制？

如果普通 `ls -l` 无法解释，可继续检查 ACL：

```bash
getfacl /path/to/target
```

`getfacl` 若未安装，可通过 `sudo apt install acl` 安装。不要在没有理解现有 ACL 的情况下用 `setfacl -b` 清空。

## 21. 高频错误与正确判断

### `No such file or directory`

不一定只表示目标不存在。脚本存在却报这个错误时，还可能是 shebang 指向不存在的解释器，或首行含 CRLF 导致解释器路径末尾多了 `\r`。

```bash
ls -l ./script.sh
head -n 1 ./script.sh
file ./script.sh
```

### `Is a directory` / `Not a directory`

说明某一级对象类型与命令预期不同。使用：

```bash
file path
namei -l path
```

### 文件删除后磁盘空间没有释放

可能仍有进程保持文件描述符：

```bash
sudo lsof +L1
```

应确认是哪个服务和文件，再通过正确方式重启或让进程重新打开日志；不要随意杀掉未知进程。

### `sudo` 后生成的项目文件无法编辑

检查所有者：

```bash
find . -maxdepth 2 -user root -ls
```

如果确实是误用 sudo 生成，针对明确项目路径修复所有者。不要对家目录或整个磁盘盲目递归 `chown`。

### `chmod +x` 后脚本仍不能执行

继续检查 shebang、换行符、所在文件系统是否带 `noexec` 挂载选项：

```bash
head -n 1 script.sh
file script.sh
findmnt -T script.sh
```
