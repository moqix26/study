# 00 学习路线与环境选择

Linux 不是某个孤立的软件，而是一整套运行环境：内核负责进程、内存、文件系统和网络，用户空间提供 Shell、systemd、软件包、日志与各种服务。学习它最有效的方法，是先明确“当前实验需要模拟什么”，再选择环境。

本章完成后，你应该能够：

- 解释 WSL2、虚拟机和云服务器的差异。
- 为日常学习、部署演练和真实上线选择合适环境。
- 安装或初始化 Ubuntu 24.04 LTS，并完成一次环境体检。
- 分清 Windows 路径、Linux 路径、宿主机与来宾机。
- 知道公网机器最基本的安全边界。

## 1. 先建立三层视角

以后排查 Linux 问题时，可以先判断问题属于哪一层：

```text
你的 Go/Gin 程序
  ├─ 配置、代码、数据库连接、HTTP 路由
Linux 用户空间
  ├─ Bash、systemd、journald、Nginx、APT、SSH
Linux 内核
  ├─ 进程、内存、文件系统、TCP/IP、设备
```

例如：

- `permission denied` 通常先看文件权限、用户身份或挂载方式。
- `address already in use` 通常先看内核中哪个进程占用了端口。
- 服务重启后环境变量丢失，通常看 systemd unit，而不是改 Go 代码。
- 域名能解析但请求超时，要继续区分安全组、防火墙、监听地址和应用状态。

Linux 学习不是背命令，而是用命令观察这三层的状态。

## 2. 三种环境的本质区别

### 2.1 WSL2：日常开发环境

WSL2 在 Windows 中运行一个真实 Linux 内核，并提供与 Windows 的文件和网络互通。它启动快、占用相对低，很适合：

- Bash、文件、文本、权限等日常命令。
- Git 和 Go 开发。
- Shell 脚本。
- 本机运行 Gin、MySQL、Redis 或 Docker 进行开发。
- 在 Windows 编辑器与 Linux 工具链之间协作。

WSL2 目前支持 systemd，但它仍不是一台完全独立的远程服务器。关机、IP、端口转发、DNS、Windows 防火墙和磁盘挂载都可能受宿主机影响。因此本手册不把 WSL2 作为 systemd、SSH、防火墙和正式部署的唯一实验环境。

### 2.2 虚拟机：可反复破坏的独立服务器

VirtualBox、VMware 或 Hyper-V 中的 Ubuntu 虚拟机拥有自己的虚拟磁盘、网卡、用户和启动流程。它适合：

- systemd 服务和开机启动。
- SSH 登录。
- UFW、防火墙与监听地址实验。
- Nginx、Docker Compose 和完整部署。
- 制造故障后使用快照恢复。

虚拟机最大的学习价值是“边界清楚”：宿主 Windows 是一台机器，Ubuntu VM 是另一台机器。你可以更真实地理解 IP、端口、远程用户和服务生命周期。

### 2.3 云服务器：真实公网环境

云服务器在操作体验上像远程虚拟机，但多了真实公网因素：

- 云厂商安全组在 Linux 防火墙之外再过滤一次流量。
- 公网 IP 会受到自动扫描和登录尝试。
- 域名、DNS、TLS 证书可以真实生效。
- 磁盘、带宽、公网 IP 和快照可能产生费用。

云服务器适合最后的真实部署，不适合第一次练习危险命令。先在 VM 中跑通，再上云。

## 3. 推荐的环境组合

对 Go 后端学习，推荐两个环境并行：

| 任务 | 首选环境 | 原因 |
|---|---|---|
| Go/Gin 编码与单元测试 | WSL2 | 启动快，开发工具方便 |
| 文件、文本、Shell 命令 | WSL2 | 练习成本最低 |
| systemd 与 journald | VM | 启动流程和服务边界更接近服务器 |
| SSH 和网络实验 | VM | IP、网卡和远程连接更清晰 |
| Nginx 与完整部署 | VM，之后云服务器 | 先低风险复现，再验证公网链路 |
| DNS 和 HTTPS | 云服务器 | 需要真实域名和公网入口 |

如果电脑资源有限，只使用 WSL2 也可以开始学习前几章；当学到部署时，再租一台按量计费云服务器或准备轻量 VM。

## 4. 安装 WSL2 + Ubuntu 24.04

以下命令在 **管理员 PowerShell** 中运行。先查看可安装发行版：

```powershell
wsl --list --online
```

列表中若包含 `Ubuntu-24.04`，可安装：

```powershell
wsl --install -d Ubuntu-24.04
```

安装完成后按提示重启 Windows，首次打开 Ubuntu 时创建 Linux 用户名和密码。这个密码用于 `sudo`，输入时终端不会显示星号，这是正常行为。

确认 WSL 状态：

```powershell
wsl --status
wsl --list --verbose
```

典型结果会显示发行版版本为 `2`：

```text
  NAME            STATE           VERSION
* Ubuntu-24.04    Running         2
```

如果已有发行版但仍是 WSL 1，可以转换：

```powershell
wsl --set-version Ubuntu-24.04 2
```

命令中的发行版名称必须以 `wsl --list --verbose` 的实际结果为准。

### 4.1 在 Linux 中确认 systemd

进入 Ubuntu 后运行：

```bash
ps -p 1 -o pid,comm,args
systemctl is-system-running
```

若 PID 1 是 `systemd`，说明 systemd 已启用。`systemctl is-system-running` 可能返回 `running`，也可能在 WSL 中返回 `degraded`；后者不一定代表无法使用，需要结合失败 unit 判断：

```bash
systemctl --failed
```

如果旧版 WSL 没有启用 systemd，可以检查 `/etc/wsl.conf`：

```ini
[boot]
systemd=true
```

保存后，在 PowerShell 关闭并重新启动 WSL：

```powershell
wsl --shutdown
```

新安装的现代 WSL 通常无需手动配置。即使 systemd 可用，本手册仍建议把完整服务部署放到 VM 或云服务器练习。

## 5. 准备 Ubuntu 24.04 虚拟机

### 5.1 建议配置

用于 Go、Nginx、MySQL、Redis 与 Docker 的练习 VM，建议至少：

- CPU：2 核。
- 内存：4 GiB；若同时运行多个容器，建议 6～8 GiB。
- 磁盘：40 GiB，动态分配即可。
- 系统：Ubuntu Server 24.04 LTS 64 位。
- 网络：第一次使用 NAT；需要局域网其他设备直连时再考虑桥接。

Ubuntu Server 没有桌面环境并不是缺点。后端服务器主要通过终端、SSH 和日志管理，纯命令行反而更贴近目标环境。

### 5.2 NAT 与桥接不要混淆

NAT 模式下，VM 可以借宿主机访问互联网，并获得一段虚拟网络中的私有 IP。宿主机能否直接访问来宾机，取决于虚拟化软件的网络实现和规则；很多常见 NAT 网络允许宿主访问来宾 IP，但不能把它当成所有平台都成立的规则。

应当实际验证：

```bash
# VM 内查看地址
ip -brief address
ip route
```

然后在 Windows PowerShell 中测试：

```powershell
Test-NetConnection <VM_IP> -Port 22
```

若宿主无法直连，可检查虚拟化软件的 NAT 设置或增加端口转发。外部局域网设备通常不能直接通过 NAT 访问 VM，除非配置转发。

桥接模式让 VM 更像局域网中的独立机器，会从局域网获得地址。它便于其他设备访问，但在校园网、公司网、公共 Wi-Fi 中可能被限制，也扩大了暴露面。不要为了“看起来更真实”而默认桥接。

### 5.3 快照策略

建议建立三个快照：

1. `clean-install`：系统刚安装并更新。
2. `base-tools`：SSH、Go 和基础工具准备完成。
3. `before-deploy`：开始 Docker、Nginx 或数据库部署前。

快照不是数据备份，但很适合学习时快速恢复误改的系统配置。真实项目数据仍需独立备份。

## 6. 初始化云服务器

云服务器一旦拥有公网 IP，就不要使用以下做法：

- 不要给 root 设置弱密码并开放密码登录。
- 不要把 SSH 暴露后长期依赖用户名密码。
- 不要开放 `3306`、`6379` 给整个互联网。
- 不要把数据库口令写进公开仓库或命令截图。
- 不要因为“暂时测试”就关闭所有防火墙。

创建实例时优先选择：

- Ubuntu 24.04 LTS 官方镜像。
- SSH 密钥登录。
- 只允许你当前 IP 访问 SSH；若 IP 经常变化，可先开放 SSH 后尽快加固。
- 安全组初始只开放 `22/tcp`。部署 Nginx 时再添加 `80/tcp` 和 `443/tcp`。

首次登录后确认用户身份和登录来源：

```bash
whoami
id
who
last -n 5
```

云厂商默认用户可能是 `ubuntu`，也可能不同，以镜像说明为准。不要假设首次登录一定是 root。

## 7. Ubuntu 环境体检

以下检查适用于 WSL2、VM 和云服务器。每条命令都在回答一个具体问题。

### 7.1 我是谁，在哪台机器上

```bash
whoami
id
hostnamectl
pwd
```

关注：

- 当前用户是否为普通用户。
- 用户属于哪些组，是否有 `sudo`。
- 主机名是否容易区分，例如 `go-dev`、`go-vm`、`shortlink-prod`。
- 当前目录是否符合预期。

不要在 root 身份下完成所有日常操作。root 的误操作不会被普通权限拦截。

### 7.2 系统和内核是什么版本

```bash
cat /etc/os-release
uname -a
uname -m
```

Ubuntu 24.04 的输出应包含类似：

```text
PRETTY_NAME="Ubuntu 24.04.x LTS"
VERSION_CODENAME=noble
```

`uname -m` 常见为 `x86_64` 或 `aarch64`。下载 Go、Docker 或其他二进制时，架构必须匹配。

### 7.3 时间是否正确

```bash
date --iso-8601=seconds
timedatectl
```

时间错误会导致 TLS 证书校验、日志对齐、令牌有效期等问题。服务器通常保留 UTC 或设置明确时区；无论选择哪个，应用日志应带时区或统一使用 UTC。

在中国大陆学习机上可设置时区：

```bash
sudo timedatectl set-timezone Asia/Shanghai
```

生产团队应统一约定，不要在不同机器上混用无标识的本地时间。

### 7.4 网络、路由与 DNS 是否正常

```bash
ip -brief address
ip route
getent hosts example.com
curl -I --max-time 10 https://example.com
```

这些命令分别检查：

1. 网卡是否有地址。
2. 是否有默认路由。
3. 系统解析器是否能解析域名。
4. TCP、TLS 与 HTTP 请求能否完成。

只执行 `ping` 不能证明 HTTP 正常；一些主机或网络会屏蔽 ICMP，但允许 HTTPS。

### 7.5 磁盘和内存是否足够

```bash
df -hT
free -h
lsblk -f
```

关注根分区 `/` 的剩余空间。Docker 镜像、数据库数据和 journald 日志都可能逐渐占满磁盘。`free` 中的 `available` 通常比 `free` 列更能反映还能分配多少内存，因为 Linux 会主动使用空闲内存做缓存。

### 7.6 PID 1 与服务管理器

```bash
ps -p 1 -o pid,comm,args
systemctl is-system-running
```

VM 和云服务器中 PID 1 应通常是 systemd。某些容器没有 systemd，这时不能照搬 `systemctl` 命令。

## 8. 安装基础工具

Ubuntu 的软件包索引需要先更新：

```bash
sudo apt update
```

然后安装本手册常用工具：

```bash
sudo apt install -y \
  ca-certificates curl git jq less nano python3 \
  ripgrep tree unzip zip \
  dnsutils iproute2 lsof procps \
  netcat-openbsd
```

说明：

- `ripgrep` 的命令名是 `rg`，用于快速搜索文本。
- `iproute2` 提供 `ip`、`ss` 等现代网络命令。
- `procps` 提供 `ps`、`top`、`free` 等进程工具。
- `dnsutils` 提供 `dig`、`nslookup`，但常规解析验证优先使用遵循系统配置的 `getent hosts`。
- `netcat-openbsd` 提供 `nc`，后续用于端口测试。

不要看到教程就立即运行 `apt full-upgrade`。学习机可以在理解影响后更新；生产服务器升级内核和关键组件前，应先看变更、备份并安排维护窗口。

## 9. Windows 与 WSL 文件放在哪里

在 WSL 中，Windows 的 `F:\study` 通常映射为：

```text
/mnt/f/study
```

可以读取：

```bash
ls -la /mnt/f/study
```

但 Linux 项目长期放在 `/mnt/c`、`/mnt/f` 等 Windows 文件系统挂载下，可能遇到：

- 大量小文件操作更慢。
- Linux 权限语义与 Windows ACL 不能完全等同。
- 文件监听、符号链接和大小写行为出现差异。
- Windows 编辑器生成 CRLF，导致 Shell 脚本出现 `^M` 或解释器错误。

推荐把实际 Go 仓库放在 WSL 的 Linux 文件系统，例如：

```bash
mkdir -p ~/code
cd ~/code
```

Windows 可以通过资源管理器访问：

```text
\\wsl$\Ubuntu-24.04\home\<YOUR_USER>\code
```

发行版名称以本机实际名称为准。学习资料可以继续保存在 `F:\study`，代码是否迁移则根据你的编辑器和工作流决定。

### 9.1 换行符建议

Shell 脚本必须使用 LF。可以在仓库的 `.gitattributes` 中约定：

```gitattributes
*.sh text eol=lf
```

检测脚本是否混入 CRLF：

```bash
file deploy.sh
sed -n '1l' deploy.sh | head
```

若首行末尾显示 `^M` 或 `\r$`，需要在编辑器中改为 LF。不要用 `chmod +x` 试图解决换行符问题；它们是不同故障。

## 10. 终端与会话的基本概念

你在终端看到的提示符可能类似：

```text
honor@go-dev:~/code$
```

它通常包含用户名、主机名和当前目录。`$` 表示普通用户提示符，root 常显示 `#`。文档中的 `$`、`#` 只是提示，不应复制。

关闭终端可能同时终止其前台进程。开发时可以用多个终端标签页；正式服务不要靠“开着一个终端”维持，而应交给 systemd 或容器编排管理。

通过 SSH 登录远程机器后，务必先确认：

```bash
printf 'user=%s host=%s dir=%s\n' "$(whoami)" "$(hostname)" "$PWD"
```

许多误删和误重启都源于“以为自己还在本机”。可以给开发、测试和生产使用不同主机名与 Shell 提示颜色，但颜色只能辅助，不能代替确认。

## 11. 第一个实验：环境报告

目标是生成一份不包含密码、令牌和私钥的环境报告。

先创建安全实验目录：

```bash
mkdir -p ~/linux-lab/00-env
cd ~/linux-lab/00-env
```

逐条运行并理解输出：

```bash
{
  printf '== identity ==\n'
  id
  printf '\n== release ==\n'
  grep -E '^(PRETTY_NAME|VERSION_CODENAME)=' /etc/os-release
  printf '\n== kernel ==\n'
  uname -srmo
  printf '\n== time ==\n'
  date --iso-8601=seconds
  printf '\n== address ==\n'
  ip -brief address
  printf '\n== route ==\n'
  ip route
  printf '\n== disk ==\n'
  df -hT /
  printf '\n== memory ==\n'
  free -h
  printf '\n== init ==\n'
  ps -p 1 -o pid,comm,args
} | tee environment-report.txt
```

这里使用未加引号的普通命令块，并通过管道交给 `tee`；变量和命令会在执行时正常展开。生成后检查：

```bash
less environment-report.txt
```

成功标准不是输出完全相同，而是：

- 能认出自己的用户、系统版本和 CPU 架构。
- 至少有回环地址 `127.0.0.1`，联网环境通常还有一个非回环地址。
- 默认路由存在。
- 根分区有足够空间。
- 知道 PID 1 是不是 systemd。

### 11.1 主动验证写权限

在家目录中写文件应成功：

```bash
touch ~/linux-lab/00-env/can-write
ls -l ~/linux-lab/00-env/can-write
```

普通用户直接在 `/etc` 创建文件应失败：

```bash
touch /etc/should-not-exist
```

典型输出：

```text
touch: cannot touch '/etc/should-not-exist': Permission denied
```

这是正常的安全边界，不要为了让命令成功就给整个 `/etc` 改权限。后续需要修改系统配置时，只对具体命令使用 `sudo`。

## 12. 第二个实验：确认请求经过哪些环境

在 WSL2 或 VM 中启动一个临时 HTTP 服务：

```bash
mkdir -p ~/linux-lab/00-env/www
printf 'hello from %s\n' "$(hostname)" > ~/linux-lab/00-env/www/index.html
cd ~/linux-lab/00-env/www
python3 -m http.server 8000 --bind 127.0.0.1
```

另开一个同环境终端：

```bash
curl -i http://127.0.0.1:8000/
```

预期看到 `HTTP/1.0 200 OK` 和主机名。这个实验只绑定回环地址，因此仅允许同一 Linux 环境访问。按 `Ctrl+C` 停止服务。

如果把监听地址改成 `0.0.0.0`，代表在所有 IPv4 接口上监听，但能否从 Windows、局域网或公网访问，还要依次受 WSL/VM 网络、Windows 防火墙、云安全组和 Linux 防火墙影响。**监听所有接口不等于应该向公网开放。**

正式结构中，Go 服务通常监听 `127.0.0.1:8080`，由同机 Nginx 接收公网 `80/443` 后反向代理。只有跨机器或容器网络确有需要时，才扩大监听范围，并配套访问控制。

## 13. 环境选择决策表

遇到新实验时，可以快速判断：

```text
只是练 Bash、文件、Go 编译？
  └─ 是：WSL2

需要独立 IP、SSH、systemd、UFW 或重启验证？
  └─ 是：Ubuntu VM

需要真实域名、Let's Encrypt、互联网访问？
  └─ 是：云服务器

命令可能破坏系统或导致断网？
  └─ 先在有快照的 VM 验证
```

同一条命令在不同环境中效果可能不同。记录实验时，至少写清：操作系统版本、环境类型、命令、关键输出和最终结论。

## 14. 常见环境问题

### WSL 中命令突然失效

先在 PowerShell 查看状态：

```powershell
wsl --list --verbose
wsl --shutdown
```

重新进入后再检查网络和 DNS。不要一上来重装发行版；先区分是单个进程、发行版还是 Windows 网络问题。

### `sudo` 密码输入没有显示

Linux 终端默认不回显密码，也不显示星号。正常输入并回车即可。连续输错后应确认键盘布局和你输入的是 Linux 用户密码，而不是 Microsoft 账户密码。

### VM 有 IP 但 Windows 访问不到

按顺序检查：

1. 服务是否运行。
2. 服务监听的是 `127.0.0.1` 还是 VM 网卡地址/`0.0.0.0`。
3. VM 内防火墙是否允许。
4. 虚拟网卡模式和 NAT 转发是否正确。
5. Windows 到 VM IP 的路由是否存在。

不要直接关闭所有防火墙来“验证”。可以临时添加精确规则，测试后再撤销。

### 云服务器本机 curl 成功，外部仍超时

这通常说明应用进程本身可能正常，继续检查监听地址、Linux 防火墙、云安全组、公网 IP 和路由。第 03 章会完整拆解这条链路。

### `systemctl` 提示系统未以 systemd 启动

先运行：

```bash
ps -p 1 -o comm=
```

如果 PID 1 不是 systemd，你可能处于不带 systemd 的容器、旧版 WSL 或特殊环境。不要把 `systemctl` 错误当成目标服务本身的错误。
