## 2. Go 语言（25 项）

### 2.1 基础与类型

- ⬜ **变量声明**：`:=` 短声明 vs `var`；包级变量初始化顺序
- ⬜ **零值**：各类型零值；nil slice vs 空 slice JSON 差异
- ⬜ **string 与 []byte**：不可变、共享底层、转换成本
- ⬜ **rune 与 UTF-8**：遍历字符串、`range` 得到 rune
- ⬜ **数组 vs 切片**：值拷贝 vs 切片头；`append` 扩容由运行时决定，小容量常快速增长、大容量逐步放缓，不能死背固定 2 倍
- ⬜ **map**：非线程安全；`delete`；遍历时删除规则
- ⬜ **struct 嵌入**：Promotion；与方法集
- ⬜ **interface**：隐式实现；nil interface 陷阱（type/value 双 nil）
- ⬜ **type switch / assertion**：安全断言 `ok` 模式
- ⬜ **defer**：LIFO；返回值命名与 defer 修改
- ⬜ **error 与 panic**：错误包装 `%w`；何时 panic vs error
- ⬜ **iota 与 const**：枚举位掩码

### 2.2 并发与内存

- ⬜ **goroutine**：栈初始 2KB；与线程对比
- ⬜ **GMP 模型**：G/M/P 职责；work stealing 一句话
- ⬜ **channel**：无缓冲同步 vs 有缓冲；关闭后 receive 规则
- ⬜ **select**：随机公平；`default` 非阻塞
- ⬜ **sync.Mutex / RWMutex**：不可重入；defer Unlock
- ⬜ **sync.WaitGroup**：Add 要在 Go 前；Done 配对
- ⬜ **context**：取消传播；`WithTimeout` 在 HTTP 中的用法
- ⬜ **竞态**：`-race`；map 并发写 panic
- ⬜ **内存逃逸**：返回局部变量指针；`-gcflags=-m` 入门
- ⬜ **GC**：三色标记；STW 简化理解；GOGC

### 2.3 工程与标准库

- ⬜ **模块**：`go mod`；replace；版本选择 MVS
- ⬜ **testing**：表驱动；`t.Parallel`；benchmark `-bench`
- ⬜ **net/http**：Handler 接口；Server 与 Client
- ⬜ **JSON**：tag `omitempty`；自定义 `MarshalJSON`

## 3. MySQL（12 项）

- ⬜ **InnoDB**：聚簇索引；主键选择对性能影响
- ⬜ **B+ 树**：为何 MySQL 用 B+ 而非 B 树
- ⬜ **最左前缀**：联合索引 `(a,b,c)` 查询条件顺序
- ⬜ **覆盖索引**：Using index；减少回表
- ⬜ **EXPLAIN**：type、key、rows、Extra 各代表什么
- ⬜ **事务 ACID**：InnoDB 如何保证
- ⬜ **隔离级别**：脏读/不可重复读/幻读；MVCC 一句话
- ⬜ **间隙锁 / 临键锁**：RR 下防幻读
- ⬜ **索引失效**：函数、隐式转换、 `%like`、or 不当
- ⬜ **慢 SQL 排查**：慢日志、EXPLAIN、索引优化
- ⬜ **分库分表**：垂直/水平；全局 ID（雪花）
- ⬜ **短链表设计**：`short_code` 唯一索引；长 URL 长度；分片键（见 [08 设计](../系统设计/08-短链服务设计.md)）

## 4. Redis（12 项）

- ⬜ **单线程模型**：为何还快；6.0 多 IO 线程
- ⬜ **五种基础类型**：String/Hash/List/Set/ZSet 场景
- ⬜ **过期策略**：惰性 + 定期；内存淘汰 LRU/LFU
- ⬜ **持久化**：RDB vs AOF；混合持久化
- ⬜ **主从复制**：全量 + 增量；读写分离
- ⬜ **哨兵 / Cluster**：高可用与槽位
- ⬜ **缓存穿透**：布隆过滤器；空值缓存（08 短链）
- ⬜ **缓存击穿**：互斥锁；逻辑过期
- ⬜ **缓存雪崩**：TTL 随机；多级缓存
- ⬜ **Cache Aside**：先更 DB 再删缓存；延迟双删
- ⬜ **分布式锁**：SET NX EX；Lua 解锁；Redlock 争议
- ⬜ **短链缓存**：`code→URL`；302 前查 Redis（[11 章](./11-短链服务项目实战下.md)）

## 8. 项目场景题（短链 STAR）

用 [11 章项目](./11-短链服务项目实战下.md) + [08 设计](../系统设计/08-短链服务设计.md) 准备：

| 场景 | Action（做了什么） | Result 必须拿什么证明 | 不能直接照抄的说法 |
|------|--------------------|--------------------------|----------------------|
| **302 跳转优化** | Cache Aside、负缓存、TTL 抖动、`singleflight` | 固定环境下的冷/热缓存 QPS、P95/P99、命中率 | “P99 < 50ms”“性能提升 300%” |
| **发号唯一** | 选定 Counter/随机码/号段方案，DB 唯一索引兜底 | 并发创建测试、冲突重试测试、故障边界说明 | 没实现就说“雪花 + Base62” |
| **点击统计** | 有界队列或 MQ/Redis Stream，消费者确认与幂等 | 重复消息、消费者重启、积压恢复测试 | “完全不丢不重” |
| **可靠性** | Redis 故障回源、DB 故障返回 503、超时与降级 | 故障演练步骤、错误率和恢复记录 | “高可用”“生产级” |
| **工程化** | 单测、集成测试、`-race`、CI、迁移、日志与指标 | CI 链接、测试报告、覆盖核心分支 | 只写“测试覆盖完善” |
| **部署** | 多阶段镜像、非 root、健康检查、迁移、Nginx/TLS | 一键启动命令、在线演示或部署截图 | “云原生”“容器化集群” |

**开放题话术**：

- 「介绍你的短链项目」→ 业务 + 架构图（Client→App→Redis/MySQL/MQ）+ 难点一条
- 「如何防缓存穿透」→ 布隆 + 空值 + 限流（08 章）
- 「如何估算存储」→ 日增 URL 数 × 字节 × 保留年限（08 章）

### 8.1 项目写进简历的最低门槛

下面 8 项未完成前，只能说“课程/练习项目”，不包装成成熟实战：

1. 从空环境按 README 能启动，并能跑完注册、创建、跳转、禁用/过期、统计主链路。
2. 数据库使用版本化迁移；关键唯一索引和联合索引有说明。
3. 核心包有单元测试，MySQL/Redis 有集成测试，CI 跑 `go test -race ./...`。
4. Redis 故障、DB 故障、统计消费者重启时，系统行为有明确且已验证的结果。
5. 有结构化日志、请求 ID、Prometheus 指标和独立的 liveness/readiness。
6. 有可复现压测脚本，报告机器、Go 版本、数据量、并发、P50/P95/P99、错误率。
7. 至少完成一次“基线 → pprof → 修改 → 回归 → 对比”的真实优化。
8. 简历上的每一个数字都能在仓库报告或脚本中找到来源。

### 8.2 AI 辅助开发时，你必须掌握什么

AI/Cursor/Codex 可以生成脚手架、CRUD、配置和测试样板，但面试所有权由下面三条决定：

- **核心链路 100% 能讲**：请求经过哪些层、数据存在哪、失败时怎样返回。
- **关键取舍能辩护**：为什么 302、为什么该短码方案、为什么统计最终一致、为什么不拆微服务。
- **能独立修改**：现场增加一个字段、改缓存 TTL、补错误分支或定位一个 failing test。

不要说“代码都是我从零手写的”；可以诚实表达为“我使用 AI 辅助实现重复代码，但架构、验收、调试和性能实验由我完成并能解释”。

### 8.3 简历描述模板（数据必须实测后填写）

```text
- 使用 Go/Gin、MySQL、Redis 实现短链服务，支持鉴权、短链生命周期管理、302 跳转与异步点击统计；
- 为跳转热路径设计 Cache Aside、负缓存、TTL 抖动与 singleflight，在【机器/并发/数据集】下将 P99 从【实测值】降至【实测值】，缓存命中率达到【实测值】；
- 基于【Redis Stream/RabbitMQ/有界 worker】解耦点击统计，通过 event_id 幂等、重试与死信处理消费者重启和重复投递；
- 建立单元/集成/E2E 测试与 CI，使用 Prometheus、结构化日志和 pprof 完成一次可复现性能优化，并通过 Docker Compose 部署。
```

未实现或未测量的技术与数字必须删除，而不是保留括号占位后直接投递。

### 8.4 五分钟项目介绍结构


## 12. 高频算法与数据结构（Go 手撕）

| 题型 | 频率 | 模板位置 |
|------|------|----------|
| 两数之和 / 哈希 | 高 | map 补角 |
| LRU | 高 | map + 双向链表 |
| 反转链表 | 高 | 三指针 |

15. **设计** 估算 5 年 1 亿 URL 存储量级思路？

### 13.1 自测参考答案

1. interface 存 `(dynamic type, dynamic value)`；typed nil 场景中动态类型非 nil、动态值为 nil，因此整个接口不等于 nil。
2. panic: send on closed channel。
3. return 先算返回值，再 LIFO 执行 defer，defer 可改命名返回值。
4. MVCC 快照读；当前读用间隙锁。
5. 不能利用最左前缀对 `b,c` 做高效定位；优化器仍可能选择扫描该索引，但通常不能得到期望的范围查找收益，应结合 `EXPLAIN` 判断。
6. 穿透：key 不存在打穿 DB；击穿：热点 key 过期瞬间打 DB。
7. 内存操作、IO 多路复用、无锁单线程无竞争。
8. 302 每次经短链服务可统计；301 浏览器/CDN 缓存跳过。
9. 先确认谁在主动关闭连接以及是否存在连接风暴；客户端复用连接/连接池、服务端正确 Keep-Alive、扩展端口与实例容量。不要在没理解影响时随意缩短 TIME_WAIT 或修改内核参数。
10. O(1) 事件通知，无 1024  fd 上限瓶颈，减少线性扫描。
11. 检测回拨等待/报错；NTP 同步；成熟库内置处理。
12. 业务唯一键、Redis 去重 set、状态机已处理则 skip。
13. 镜像小、无编译器、安全攻击面小。
14. Client→（CDN）→App→布隆→Redis→（miss）MySQL→302→异步 MQ 统计。
15. 单条记录字节 × 日增 × 365 × 5；加索引与副本倍数；见 08 章估算表。
