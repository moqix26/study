# Go 后端系统设计学习路线图

> 这套资料服务于一个明确目标：从“能写 Go 接口”走到“能用数据和取舍解释一个后端系统”。
>
> 学习方式采用项目驱动。第一个完整 Case 是短链服务，不再先学完一整套组件再开始项目。

---

## 1. 定位与分层

系统设计不是中间件名词比赛，也不是脱离代码画大图。对初学者，它应当回答三类问题：

1. 当前需求最简单、正确的实现是什么？
2. 数据量或故障发生变化后，哪里先出问题？
3. 为解决这个具体问题，引入的新复杂度是否值得？

### 1.1 现在必学

完成 Go 基础和第一个 Gin CRUD 后，优先掌握：

- HTTP 请求、响应、状态码和超时
- MySQL 表、索引、事务的基本边界
- 需求范围与非功能目标
- 平均 QPS、峰值 QPS、存储量的粗估
- 单体服务的读路径、写路径和状态来源
- 日志、错误率、P95/P99 延迟等基本观测
- 用压测结果而不是固定数字判断瓶颈

这部分对应 [01-系统设计方法论与面试框架](./01-系统设计方法论与面试框架.md) 的基础层。

### 1.2 项目再学

短链项目出现真实需求后，再加入：

- Redis Cache Aside、空值缓存、TTL 与失效
- 本地限流、并发上限、超时、重试和熔断
- 幂等、异步统计、消息队列
- 数据库读写分离、分片和归档
- 故障演练、容量验收与降级

对应章节是 [02](./02-限流熔断与降级.md)～[05](./05-数据库扩展与读写分离.md)。

### 1.3 面试进阶

在至少完成一个可运行项目并做过压测后，再系统学习：

- CAP、复制、一致性和分布式事务
- 秒杀、Feed、跨地域和多活
- 复杂容量规划与故障恢复目标
- 多方案对比和 15～45 分钟设计表达

对应 [06](./06-分布式一致性与CAP.md)～[10](./10-面试专题与知识点总表.md)。

## 2. 开始系统设计前的门槛

不要把“暂时不学”理解为落后。基础没落地时，架构名词很容易变成背诵。

### 2.1 最低门槛

开始 01 前，至少能够：

- 用 Go 写结构体、方法、错误处理和基础并发
- 用 `net/http` 或 Gin 写两个以上 REST 接口
- 解释一次请求从路由到数据库的调用链
- 用 MySQL 完成增删改查
- 写至少一个单元测试
- 知道 `context.Context` 用来传递取消和截止时间

若尚未满足，继续 Go 和 Gin 主线即可，不需要提前刷完整系统设计资料。

### 2.2 进入缓存与韧性章节的门槛

开始 02、03 前，建议已经：

- 完成短链创建与跳转接口
- 为短码建立唯一索引
- 能说明数据库是当前真相源
- 跑过一次简单压测并看到延迟或吞吐数据

这样学限流和缓存时，每个组件都有明确保护对象。

## 3. 项目驱动主线

推荐顺序不是“先背完 01～06，再做 Case”，而是围绕短链迭代。

```text
Go 基础与并发
  ↓
Gin + 内存版 API
  ↓
MySQL 持久化版短链
  ↓
01：需求、SLO、估算、API、模型、HLD
  ↓
压测单体基线，找到真实瓶颈
  ↓
03：为跳转读路径加入缓存
  ↓
02：为创建/跳转接口加入超时、限流和过载保护
  ↓
04：点击统计异步化
  ↓
05：数据增长后再讨论读副本、分片和归档
  ↓
06：用已遇到的一致性问题整理理论
  ↓
07 秒杀 → 09 Feed，作为第二、第三类综合题
```

短链被放在第一个 Case，因为它具备：

- 清晰的写路径：长 URL 生成短码并持久化
- 清晰的读路径：短码查询并返回跳转
- 可验证的唯一性和幂等问题
- 典型的读多写少特征
- 可以逐步加入缓存、限流、异步统计和分片
- 初版不需要同时掌握所有分布式组件

## 4. 八个项目里程碑

### M0：可运行服务

目标：服务能启动、退出、测试和观测。

产出：

- `/livez` 存活检查与 `/readyz` 就绪检查
- 配置从环境变量读取
- 请求日志包含方法、路径、状态码、耗时和 request ID
- 收到退出信号后优雅关闭

验收：

- `go test ./...` 通过
- `go test -race ./...` 无数据竞争
- 服务退出时不丢正在处理的请求

### M1：MySQL 短链最小版本

核心接口：

```http
POST /api/v1/links
GET  /r/{code}
GET  /api/v1/links/{id}
```

最低不变量：

- `code` 全局唯一
- 创建成功后能够稳定跳转
- 无效短码返回明确的 404
- 数据库写成功前不能返回创建成功
- 原始 URL 需要做格式和长度校验

### M2：使用 01 做第一次设计

写出一页设计说明：

1. 做什么、不做什么
2. 延迟、可用性和一致性目标
3. 读写 QPS 与数据量估算
4. API 与核心表
5. 单体读写路径
6. 当前瓶颈及暂不引入的组件

### M3：建立性能基线

不要先假设 Redis 必需。先测：

- 创建接口吞吐和 P95/P99
- 跳转接口吞吐和 P95/P99
- MySQL 查询次数与慢查询
- CPU、内存、goroutine 和连接池使用量

记录测试环境、数据规模、请求模型和压测命令，否则数字不可比较。

### M4：缓存跳转映射

当读路径确实需要优化时，引入 [03 缓存架构](./03-缓存架构设计.md)：

- Redis Cache Aside
- 不存在短码的短 TTL 空值缓存
- 热点回源合并 `singleflight`
- Redis 故障时受控回源，而不是全量打 MySQL
- 更新或删除短链后的可靠失效

### M5：流量与依赖保护

结合 [02 限流熔断与降级](./02-限流熔断与降级.md)：

- 创建接口按用户或 IP 限速
- 跳转接口设置并发上限
- MySQL、Redis 调用传递 deadline
- 只对可安全重试的错误做有限重试
- 过载时返回明确的 429 或 503

### M6：点击统计异步化

点击统计不是跳转正确性的前置条件，可作为 [04 MQ](./04-消息队列架构设计.md) 的练习：

- 跳转先返回，统计事件异步处理
- 消费者按事件 ID 幂等
- 允许统计存在可说明的延迟
- 队列不可用时明确选择丢弃、落本地或降级

### M7：容量扩展与理论复盘

只有当估算或测试证明单库存在瓶颈时，才进入 [05](./05-数据库扩展与读写分离.md)：

- 索引与 SQL 优化
- 归档过期数据
- 读副本及复制延迟
- 按短码或用户进行分片

随后用 [06](./06-分布式一致性与CAP.md) 解释已经遇到的复制延迟、缓存失效和异步事件一致性。

## 6. 每轮学习的固定闭环

每引入一个组件，都完成下面六步：

1. **现象**：当前数据说明哪里有问题？
2. **目标**：希望哪个指标改善到什么程度？
3. **最小方案**：能否先用索引、批处理或配置解决？
4. **新故障**：组件挂掉、变慢、数据过期时会怎样？
5. **观测**：如何知道它有效或正在失效？
6. **验收**：测试结果是否达到目标，复杂度是否值得？

例如加入 Redis 前，不只问“命中率能否到 90%”，还要问：

- miss 会给 MySQL 增加多少 QPS？
- Redis 超时时 goroutine 会不会堆积？
- 热点 key 失效是否产生回源风暴？
- 删除缓存失败后旧数据最多存在多久？

## 7. 核心不变量

这些原则贯穿所有章节：

### 7.1 正确性先于扩展

- 先定义真相源，再讨论缓存和副本
- 先定义事务边界，再讨论异步化
- 先保证幂等，再打开重试
- 先保证资源有上限，再扩大吞吐

### 7.2 高层路径先于组件

一张设计图至少要能回答：

- 请求从哪里进入？
- 哪个服务负责业务规则？
- 状态写到哪里？
- 失败在哪里返回？
- 异步任务由谁继续？

如果这些问题没回答，Redis、MQ 和分库只是图标。

### 7.3 数字必须带上下文

任何 QPS、延迟或容量结论都要同时记录：

- 硬件和实例数
- 数据规模和请求比例
- 响应大小
- 并发模型
- 可接受错误率
- 测试持续时间

“MySQL 能扛多少 QPS”“多少行必须分表”都没有脱离上下文的固定答案。

### 7.4 失败必须有边界

- 外部调用必须有 deadline
- 队列和连接池必须有容量上限
- 重试必须有次数、时间和预算上限
- 缓存回源必须有限流或并发合并
- 降级必须说明牺牲了什么

## 8. Go 落地工具箱

| 目标 | 优先工具 |
|---|---|
| HTTP 服务 | `net/http`、Gin |
| 取消与超时 | `context.Context` |
| 本地速率限制 | `golang.org/x/time/rate` |
| 回源合并 | `golang.org/x/sync/singleflight` |
| Redis | `github.com/redis/go-redis/v9` |
| MySQL | `database/sql` 或项目已选 ORM |
| 指标 | Prometheus client |
| 跟踪 | OpenTelemetry，项目进阶再加 |
| 压测 | Vegeta、k6 或 hey |
| 并发检查 | `go test -race ./...` |
| 性能分析 | benchmark、pprof |

工具不是强制清单。项目已选同类库时，优先保持一致，不为追求名词频繁换栈。

## 9. 最低观测基线

系统设计不能只交架构图。每个版本至少能看到：

### 请求指标

- 请求总量和各状态码数量
- P50、P95、P99 延迟
- 当前在途请求数
- 429、503 和超时数量

### 依赖指标

- MySQL 查询延迟、错误和连接池等待
- Redis 命中、miss、错误和超时
- MQ 堆积、消费延迟和重试

### Go 运行时

- goroutine 数量
- heap、GC 和 CPU
- 阻塞或泄漏趋势

验收以自己的基线和 SLO 为准，不使用“命中率必须 90%”“Redis 必须 2ms”这类无上下文常数。

## 10. 学习节奏建议

不规定必须四周学完。以里程碑是否通过为准。

| 阶段 | 推荐投入 | 可验证产出 |
|---|---:|---|
| Go/Gin 基础 | 按主线课程 | 内存版 API + 测试 |
| MySQL 短链 | 3～5 天 | 创建、跳转、唯一索引 |
| 01 方法论 | 1～2 天 | 一页设计说明 + 架构图 |
| 基线压测 | 0.5～1 天 | 可复现测试记录 |
| 03 缓存 | 2～3 天 | Cache Aside + 故障实验 |
| 02 韧性 | 2～3 天 | deadline + 限流 + 过载测试 |
| 04～06 | 随项目逐章 | 异步统计与一致性说明 |
| 07/09 | 面试阶段 | 两个新场景迁移练习 |

如果每天只有一小时，每次只做一个闭环动作：实现、测试、画图、估算或复盘，不要同时开多章。
