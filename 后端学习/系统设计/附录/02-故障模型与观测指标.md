# 故障模型、恢复与观测指标

## 1. 最重要的一句话

**超时只表示调用方没有及时得到答案，不表示操作一定失败。**

请求可能处于：

1. 根本没到服务端；
2. 到达但尚未执行；
3. 已执行但响应丢失；
4. 已提交但调用方超时；
5. 部分副作用成功，其他失败。

因此重试前必须回答：

- 操作是否幂等；
- 是否有 Idempotency-Key；
- 能否查询最终状态；
- 是否需要补偿或对账；
- 总 deadline 还剩多少。

## 2. 超时预算

上游 deadline 应大于各阶段预算之和，但下游不能各自使用完整上游超时。

~~~text
request deadline
  = gateway budget
  + service processing
  + dependency calls
  + retry reserve
  + response reserve
~~~

Go 中使用 context 传播取消；不要在请求结束后继续使用已经取消的 context 完成共享回源任务，也不要无界创建后台 goroutine。

## 3. 重试

可重试的典型情况：

- 明确的临时网络错误；
- 429/503 且服务给出 Retry-After；
- 乐观锁冲突并且业务允许重新读取；
- 消费者处理的可恢复依赖故障。

通常不应自动重试：

- 参数错误；
- 权限错误；
- 明确的业务冲突；
- 非幂等外部副作用；
- deadline 已无剩余。

重试至少需要：

- 指数退避；
- jitter；
- 最大次数；
- 总 deadline；
- retry budget；
- 每次尝试的可观测记录。

## 4. 常见故障窗口

| 窗口 | 风险 | 常用闭环 |
|------|------|----------|
| DB 提交后、发 MQ 前崩溃 | 永久漏事件 | Outbox/CDC |
| MQ 已发、标记已发前崩溃 | 重复发布 | 消费端 Inbox 幂等 |
| 消费业务提交后、ACK 前崩溃 | 重复消费 | 同事务去重与业务更新 |
| 更新 DB 后、删缓存失败 | 读旧值 | Outbox/CDC、TTL、版本 |
| 读旧 DB 后、写操作删缓存、旧读再回填 | 旧值复活 | 版本化快照、条件写 |
| 锁过期后旧持有者继续写 | 新旧持有者并发 | fencing token、DB 条件更新 |
| 请求超时但操作已成功 | 重试重复执行 | 幂等键、状态查询 |

## 5. RPO 与 RTO

- RPO：故障后最多允许丢多少数据。
- RTO：故障后多久必须恢复服务。

示例：

| 子系统 | RPO | RTO |
|--------|-----|-----|
| 短链映射 | 接近 0 | 分钟级 |
| 点击近似统计 | 可少量丢失 | 小时级也可能接受 |
| 秒杀订单 | 0 或可对账恢复 | 分钟级 |
| Feed 派生时间线 | 可重建 | 分钟到小时 |

不同数据不能统一套“高可用”三个字。

## 6. 四类观测信号

### 6.1 流量

- QPS、并发、请求大小；
- 热点 key/用户/租户；
- 允许、拒绝、降级数量；
- 队列生产速率。

### 6.2 延迟

- HTTP P50/P95/P99；
- DB、Redis、MQ 分阶段延迟；
- 队列 oldest age 和 consumer lag；
- outbox 最老未处理事件年龄。

### 6.3 错误

- 业务错误与依赖错误分开；
- timeout、cancel、retry exhausted；
- cache fill 失败；
- publisher nack、unroutable；
- DLQ 数量与重放结果。

### 6.4 饱和度

- CPU、内存、GC、goroutine；
- DB 连接池等待和利用率；
- Redis 内存、eviction、hot key；
- 磁盘、IOPS、复制 lag；
- worker queue depth。

## 7. 高基数纪律

Prometheus label 不应直接放：

- user_id；
- order_id；
- short_code；
- URL；
- request_id。

这些值进入采样日志或 trace。指标 label 使用有限集合，例如 route template、status、result。

## 8. 故障演练流程

1. 写假设：故障时系统应该怎样。
2. 固定环境和负载。
3. 注入单一故障。
4. 记录时间线、指标和用户结果。
5. 恢复并测量 RTO。
6. 对账是否丢失、重复或不一致。
7. 写出实际与预期差异。
8. 修复后用相同条件复测。

## 9. 最小故障矩阵

| 依赖 | 超时 | 不可用 | 返回旧值 | 恢复后 |
|------|------|--------|----------|--------|
| MySQL | 快速失败/取消 | 核心写不可用 | 副本可能旧 | 连接和事务恢复 |
| Redis | 短超时 | 受控回源或降级 | 缓存可能旧 | 防止回源风暴 |
| MQ | publish 结果未知 | Outbox 重试 | 消费延迟 | 追平与对账 |
| 第三方 API | 总 deadline | 熔断/降级 | 结果需查询 | 幂等重试 |

没有故障矩阵的“高可用设计”通常只是组件清单。
