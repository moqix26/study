# 系统设计术语与延伸阅读

## 1. 核心术语

| 术语 | 准确定义 |
|------|----------|
| SLI | 实际测量指标，例如成功率、P99 |
| SLO | 对 SLI 的目标，例如月成功率 99.9% |
| SLA | 对外承诺及可能的违约责任 |
| invariant | 无论并发和故障如何都必须成立的业务条件 |
| idempotency | 同一逻辑操作重复执行，最终效果与一次相同 |
| linearizability | 操作表现得像在某一瞬间按单副本顺序生效 |
| serializability | 并发事务结果等价于某个串行顺序 |
| eventual consistency | 停止更新后，各副本最终收敛 |
| read-your-writes | 一个会话能读到自己刚完成的写 |
| backpressure | 下游处理不过来时向上游传递减速信号 |
| load shedding | 过载时主动拒绝部分请求保护核心服务 |
| fencing token | 单调递增令牌，防止过期锁持有者继续写 |
| Outbox | 业务写和待发布事件在同一数据库事务落盘 |
| Inbox | 消费端在本地事务中记录已处理业务事件 |
| RPO | 故障允许的数据丢失窗口 |
| RTO | 故障允许的恢复时间 |

## 2. 容易混淆的概念

### 2.1 Rate、Concurrency、Quota

- Rate limit：单位时间接纳多少。
- Concurrency limit：同时处理多少。
- Quota：一个周期内总共允许消费多少。
- Backpressure：让生产速率适配消费能力。

### 2.2 ACK 与业务成功

Broker ACK、Publisher Confirm、Consumer ACK 分别发生在不同链路，不能互相替代。

### 2.3 缓存一致与数据库一致

缓存通常是派生数据。强一致业务字段应以事务数据库或具有明确一致性协议的存储为准。

### 2.4 CAP

CAP 讨论网络分区发生时，受影响操作在一致性和可用性之间的取舍。它不是给整个产品永久贴 CP/AP 标签，也不是任意三项“选两项”。

## 3. 产品和版本提示

- RabbitMQ 新系统优先了解 quorum queue，而不是只背旧镜像队列。
- Redis Sentinel/Cluster 使用异步复制，不能简单等同于严格 CP 或 AP。
- RedisBloom 是 Redis 模块/Redis Stack 能力，不是“某个 Redis 版本天然内置”。
- Spring readOnly 事务不会天然把查询路由到只读副本；Go 的 database/sql 也需要显式 writer/reader 设计。
- 中间件吞吐必须带消息大小、持久化、副本、确认、批量和硬件条件。

### 4.3 阅读纪律

1. 先读本仓库给出的业务问题。
2. 遇到产品具体语义，再查对应版本的官方文档。
3. 把官方保证与自己的部署配置分开。
4. 不把博客中的固定 QPS、行数和延迟当作通用定律。
5. 面试表述应说明假设和取舍，不背品牌结论。
