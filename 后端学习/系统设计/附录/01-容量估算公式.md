# 容量估算公式与单位检查

> 目的：得到数量级并驱动架构选择，不是预测未来到个位数。

## 1. 先写时间窗口

任何请求量都必须说明：

- 每秒、每分钟还是每天；
- 平均值还是峰值；
- 峰值持续多久；
- 活跃存量还是当天新增；
- 外部请求还是系统内部放大后的操作。

## 2. 请求与并发

~~~text
average_qps = requests_per_day / 86400
peak_qps = average_qps × peak_factor
in_flight ≈ qps × average_latency_seconds
~~~

最后一条是 Little 定律的简化应用，表示平均在途请求，不等于 TCP 连接数，也不等于线程数。

Go HTTP 服务可能通过 keep-alive、HTTP/2 和 goroutine 复用连接；线程池大小不能直接用 QPS × RT 得到。

## 3. 存储

~~~text
logical_bytes = records × average_record_bytes
provisioned_bytes =
    logical_bytes
    × index_and_page_factor
    × replication_factor
    × backup_and_headroom_factor
~~~

需要分别记录：

- 主数据；
- 二级索引；
- 日志和事件；
- 临时迁移空间；
- 副本；
- 备份。

不要把“逻辑数据 9TB”和“部署需要 30TB”混成一个没有解释的数字。

## 4. 带宽

~~~text
bytes_per_second = qps × average_payload_bytes
bits_per_second = bytes_per_second × 8
~~~

同时考虑：

- 请求和响应；
- TLS、HTTP、压缩开销；
- 服务间调用；
- MQ、复制和缓存回填；
- 跨地域流量。

## 5. 缓存

~~~text
cache_memory =
    cached_items
    × average_key_value_bytes
    × metadata_fragmentation_factor
    × replica_factor
~~~

Redis 的真实占用不仅是 value 长度。需要用 MEMORY USAGE、采样和实际序列化结果校准。

缓存命中率也不能单独看：

~~~text
origin_qps = total_qps × (1 - hit_rate)
~~~

当 total_qps 很大时，即使命中率 99%，剩余 1% 也可能打垮数据库。

## 6. 队列

~~~text
backlog_growth_per_second = producer_rate - consumer_rate
drain_time = backlog / (consumer_rate - producer_rate)
~~~

只有 consumer_rate 大于 producer_rate 时，积压才会被追平。

还要估算：

- 单条消息大小；
- 最大停机窗口；
- 保留时间；
- 分区/队列数量；
- 重试和 DLQ 放大。

## 7. 随机空间与碰撞

空间大小：

~~~text
M = alphabet_size ^ code_length
~~~

生日近似：

~~~text
P(any collision) ≈ 1 - exp(-n(n-1)/(2M))
expected_collision_pairs ≈ n(n-1)/(2M)
~~~

对随机短码：

- 不能用“空间很大”代替唯一约束；
- 发生过任一碰撞的概率可能很高；
- 单次插入冲突率仍可能足够低，从而通过重试工作；
- 结论必须同时说明 n、M、唯一索引和重试上限。

## 8. 分片

若有 D 个库、T 张表：

~~~text
slot_count = D × T
slot = hash(key) % slot_count
db = slot / T
table = slot % T
~~~

错误示例：

~~~text
db = key % 4
table = key % 16
~~~

两者相关，只会得到 16 种组合，而不是 64 种。

## 9. 故障余量

容量设计至少考虑一个实例或一个可用区故障：

~~~text
required_capacity_after_failure
    <= remaining_capacity × target_utilization
~~~

不要长期把正常利用率压到 100%。CPU、连接池、线程/goroutine、磁盘和下游限额都需要余量。
