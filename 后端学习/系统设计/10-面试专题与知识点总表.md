## 2. 系统设计答题框架

### 2.1 七步顺序

| 步骤 | 时间 | 必答内容 |
|---|---:|---|
| 1. 澄清需求 | 2min | 用户、核心功能、规模、一致性、延迟、是否允许异步 |
| 2. 容量估算 | 2min | 峰值 QPS、存储、带宽、内部放大 |
| 3. API | 1min | 核心资源、请求/响应、状态码、幂等 |
| 4. 数据模型 | 2min | 权威表、唯一约束、索引、分片键 |
| 5. 高层架构 | 3min | 请求流、写流、异步流、权威数据源 |
| 6. 深挖难点 | 4min | 选 2～3 个最影响不变量的点 |
| 7. 故障与观测 | 2min | 降级、重试、对账、指标、取舍 |

### 2.2 开场模板

```text
我先确认核心用户路径和不做的范围，再给出一组可调整的规模假设。
接着我会估算峰值 QPS、存储和内部读写放大，定义业务不变量，
然后设计 API、权威数据模型和高层架构，最后深入一致性、热点与故障恢复。
```

### 2.3 每一层都问三个问题

1. 它维护哪条业务不变量？
2. 它故障时用户看到什么？
3. 用什么指标证明它正常？

答不出这三问，通常只是堆组件。

## 3. 容量估算速查

### 3.1 常用公式

```text
平均 QPS = 日请求量 / 86400
峰值 QPS = 平均 QPS × 峰值系数
突发 QPS = 窗口内请求数 / 窗口秒数
带宽 = QPS × 单次请求或响应大小
逻辑存储 = 条数 × 单条逻辑字节
实际容量 = 逻辑存储 × 索引/副本/备份/碎片系数
内部放大 = API QPS × 每请求下游操作数
```

### 3.2 估算检查

- “同时点击”必须写清是 1 秒、2 秒还是 1 分钟。
- 日新增与日活跃不能混用。
- 500B × 182.5 亿约为 9.1TB，不是 91TB。
- Redis 容量要计 key、对象、allocator、复制和预留。
- Feed、搜索、聚合题必须算内部 fan-out/fan-in。
- 压测数字只有记录机器、配置、数据集和原始结果后才能写进简历。

### 3.3 数量级不确定时

可以说：

```text
我先用这组假设完成设计；如果真实峰值高一个数量级，
我会优先重新验证热点、分片和下游写预算，而不是直接加所有组件。
```

## 4. 组件选型速查

| 问题 | 首选思路 | 先问的边界 |
|---|---|---|
| 热点读 | Cache Aside、对象缓存 | 权威数据在哪、能否读旧值 |
| 随机不存在 key | 负缓存；必要时 Bloom | DB 预算、Bloom 完整性 |
| 热 key 回源 | singleflight、本地缓存 | 单实例还是多实例 |
| 突发写 | 队列削峰 | 消息进入队列前如何可靠 |
| 重复消息 | inbox/唯一键/状态机 | 幂等键是什么 |
| 顺序消息 | 同业务 key 分区 | 需要全局顺序还是局部顺序 |
| 写库与发消息 | transactional outbox | 延迟能否接受 |
| API 防刷 | 网关 + 服务限流 | Redis 故障时放行还是拒绝 |
| 分布式 ID | 随机 ID、号段、Snowflake | 是否可枚举、是否要排序 |
| 深分页 | 复合 cursor | 排序是否稳定、有无同值 |
| 派生数据 | 可重建缓存/索引 | 源数据和重建窗口 |
| 实时通知 | SSE/WebSocket | 只是提示还是权威消息 |

### 4.1 Redis 数据结构

| 需求 | 结构 | 典型命令 |
|---|---|---|
| 映射缓存 | String | `GET/SET` |
| 计数 | String | `INCR` |
| 唯一集合 | Set | `SADD/SISMEMBER` |
| 排行榜/时间线 | ZSet | `ZADD/ZRANGE` |
| 近似 UV | HyperLogLog | `PFADD/PFCOUNT` |
| 可靠轻量事件 | Stream | `XADD/XREADGROUP/XACK` |

### 4.2 不要默认使用的组件

- 微服务：模块化单体能满足时先不用。
- 分库分表：先证明单库索引和容量不够。
- Bloom：先证明随机 miss 确实打穿 DB。
- MQ：如果没有异步、削峰或解耦价值，不要为了关键词加入。
- Kubernetes：一台主机和 Docker 就能展示时，不影响项目质量。
- 分布式锁：优先考虑唯一约束、状态条件和原子命令。

## 5. 一致性、CAP 与可靠性

### 5.1 CAP 正确说法

CAP 不是日常状态下简单“三选二”。当网络分区发生时，对受影响操作需要在：

- 保持一致但拒绝/等待部分请求；
- 保持可用但允许短暂不一致；

之间取舍。

不同业务可以不同选择：

| 数据 | 倾向 | 原因 |
|---|---|---|
| 订单与库存 | 一致性优先 | 不能超卖或重复扣款 |
| 短链映射 | MySQL 权威，缓存最终一致 | 跳转可用性与禁用时效需明确 |
| 点击统计 | 可用性优先 | 允许少量延迟或近似 |
| Feed timeline | 最终一致 | 源帖子持久，timeline 可重建 |
| 屏蔽/权限 | 一致性优先 | 不能因缓存故障泄露内容 |

### 5.2 常见可靠性模式

| 模式 | 解决什么 |
|---|---|
| 唯一索引 | 并发创建、重复消费的最终防线 |
| 状态条件更新 | 支付/取消、启用/禁用竞态 |
| Idempotency-Key | 客户端超时重试返回同一结果 |
| Inbox | 消费端识别重复事件 |
| Outbox | DB 提交与发布事件之间的双写窗口 |
| DLQ | 有限重试后的人工修复入口 |
| 对账 | 发现异步链路长期不收敛 |
| Tombstone/version | 防旧缓存回填覆盖新状态 |

### 5.3 “结果未知”必答

以下操作都可能返回结果未知：

- DB commit 时连接断开；
- MQ publish 后 confirm 丢失；
- Redis 写成功但响应超时；
- 第三方支付成功但回调超时。

处理原则：

1. 重试必须复用同一个业务幂等键；
2. 先查询权威状态；
3. 不能在结果未知时做无条件反向补偿；
4. 用唯一约束和状态机让重复操作安全。

## 7. 三大 Case 对比

| 维度 | 秒杀 | 短链 | Feed |
|---|---|---|---|
| 主要瓶颈 | 瞬时资格写热点 | 跳转读热点 | fan-out/fan-in |
| 权威数据 | MySQL 订单/库存 | MySQL 链接映射 | MySQL 帖子/关系 |
| Redis | 资格与事件 | 映射缓存、限流 | timeline 派生数据 |
| 异步 | 订单消费者/补偿 | 点击统计 | fan-out |
| 一致性 | 库存强约束 | 映射权威、统计最终一致 | timeline 最终一致 |
| 幂等 | request/event/order | 创建/统计 event | outbox/fan-out task |
| 典型游标 | 订单列表 | 链接列表 | score + member + as_of |
| 故障重点 | 发送未知、支付取消 | Redis 降级、旧缓存 | outbox lag、重建 |
| 适合当前项目 | 否，后期进阶 | **是，主项目** | 否，后期进阶 |

## 8. 短链项目答辩证据

### 8.1 15 分钟讲解顺序

1. 产品范围：注册、创建、管理、302 跳转、统计。
2. V1：模块化单体、MySQL 权威、随机 Base62、完整 API。
3. 正确性：唯一索引、Idempotency-Key、业务去重、权限。
4. V2：Cache Aside、负缓存、singleflight、Redis 可降级。
5. 写一致性：version + tombstone + cache outbox。
6. 统计：有界队列的近似语义，或 Stream 的重试与幂等。
7. 观测：日志、Prometheus、健康检查、故障演练。
8. 证据：测试、压测原始结果、pprof 或恢复时间线。

### 8.2 高频项目追问

| 追问 | 回答核心 | 项目证据 |
|---|---|---|
| 为什么不用 Redis INCR 生成码 | Redis 不成为创建正确性依赖；随机码不连续难枚举 | Go 10 短码 ADR |
| 随机码会碰撞吗 | 会；DB 唯一索引 + 有限重试是最终保障 | 冲突集成测试 |
| 幂等和去重区别 | 同请求重放 vs 同业务 URL 是否复用 | idempotency/dedup 表 |
| 为什么 302 | 便于每次回源和统计；项目配 `no-store` | 跳转响应测试 |
| Redis 挂了怎么办 | 短超时后回源 MySQL；限流按接口降级 | 故障演练 |
| 更新后为何不只 DEL | 旧 miss 可能延迟回填；version CAS 拒绝旧值 | 竞态集成测试 |
| 统计是否 exactly-once | V2 近似；Stream 入流后可幂等重试，但入流前仍可能丢 | README 语义说明 |
| 为什么不用每请求 goroutine | 无界 goroutine 会耗尽资源且无法优雅关闭 | 有界队列测试 |
| Bloom 必须吗 | 不是；先用负缓存和指标证明需要 | V3 可选项 |
| 如何证明性能 | 固定环境、脚本、原始数据、P95/P99/资源指标 | benchmark 报告 |
| 单机项目算高并发吗 | 不冒充生产规模；展示正确模型、故障边界和真实实验 | README/ADR |
| 最大遗憾是什么 | 给出真实未做项与下一步，不编造 | Roadmap |

### 8.3 简历描述规则

可以写：

- 设计并实现；
- 通过测试验证；
- 在某固定环境实测；
- 故障演练恢复时间为多少。

不能写：

- 没有压测却声称“支持 10 万 QPS”；
- 只使用组件就声称“高可用”；
- 内存队列却声称“消息零丢失”；
- 单实例 singleflight 却声称“集群只回源一次”。

## 11. 常见红旗

- 一上来就画 Redis、Kafka、分库分表，没有需求与估算。
- 把平均 QPS 当峰值。
- 把缓存当权威数据源。
- 说“MQ 保证 exactly-once”却没有业务幂等。
- DB 提交后普通发 MQ，却没看到双写窗口。
- 结果未知时直接反向补偿。
- Redis Cluster 多 key Lua 没有同槽 hash tag。
- 完整 Snowflake ID直接作为 ZSet score。
- cursor 只有 timestamp，没有同值 tie-breaker。
- 只说“Redis 挂了回源 DB”，不做超时、限流和 DB 预算。
- 用静态阈值代替压测和容量预算。
- 声称零丢失、强一致、高可用，却没有边界和证据。
- Prometheus label 放 user_id、URL、tweet_id 等高基数字段。
- 项目讲解全是技术名词，没有用户路径和失败行为。

## 12. 高频口述答案骨架

### 12.1 缓存一致性

```text
MySQL 是权威。读走 Cache Aside；写先提交 DB，再发布失效或新快照。
仅 DEL 仍有旧 miss 延迟回填竞态，因此重要状态用 version/tombstone，
并由 outbox 重试传播。若要求立即封禁，就要牺牲可用性查询强一致撤销信息。
```

### 12.2 消费幂等

```text
消息传输采用至少一次。消费者用全局 event_id 或业务唯一键写 inbox，
inbox、业务更新和状态变更放同一 DB 事务；commit 成功后才 ACK。
重复消息读到既有 outcome 后安全返回。
```

### 12.3 Redis 故障

```text
先区分 Redis 是正确性依赖还是可降级缓存。
缓存故障用短超时、singleflight、限流后回源；
限流或可靠 Stream 故障要按接口定义 fail-open/fail-closed，
并监控 DB QPS、错误率、恢复时间和积压。
```

### 12.4 分片

```text
先根据容量和延迟证明需要分片，再选最常见访问路径的 shard key。
同时列出被牺牲的查询、全局唯一键、扩容迁移和跨分片事务。
如果辅助索引或 ES 解决列表查询，要说明一致性和回源边界。
```
