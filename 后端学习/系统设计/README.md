# 系统设计：Go 后端项目驱动路线

> **入口原则**：先做出正确的单体，再根据指标引入复杂组件。  
> **当前主 Case**：短链服务。秒杀与 Feed 属于后期综合练习。  
> **资料用途**：学习设计推理、故障边界与取舍，不用于虚构项目规模。

---

## 1. 这套资料解决什么问题

系统设计不是背“Redis + MQ + 分库分表”，而是完成一条可验证的推理链：

~~~text
业务目标与不变量
  → 容量和 SLO 假设
  → API、数据模型、读写路径
  → 单机正确版本
  → 找出真实瓶颈
  → 选择最小复杂度的扩展
  → 定义故障、观测、回滚和验收
~~~

学完后，你应能：

- 先问需求，而不是先报中间件名字；
- 区分正确性层和性能层；
- 解释超时、重试、重复、丢失和结果未知；
- 画出读写路径及每个故障窗口；
- 用指标说明为什么需要缓存、MQ、副本或分片；
- 把自己的 Go 短链项目讲清，而不是照背大厂架构。

## 4. 每道设计题的统一框架

1. **需求与范围**：谁用、核心功能、明确不做什么。
2. **不变量与 SLO**：哪些数据绝不能错，哪些允许延迟。
3. **容量估算**：时间窗口、活跃数据集、读写放大、单位闭环。
4. **API 与数据模型**：围绕访问模式和事务边界设计。
5. **高层架构和读写路径**：先画最小正确版本。
6. **瓶颈与扩展**：每增加一个组件，都给触发指标和代价。
7. **故障与恢复**：超时、重复、丢失、结果未知、补偿和对账。
8. **观测与验收**：指标、日志、追踪、测试、压测和演练证据。

这不是唯一的行业模板。面试时可根据题目调整 API、模型和高层图的先后顺序，但不能缺少不变量、读写路径与故障边界。

## 5. 组件选择纪律

| 组件 | 什么时候引入 | 主要代价 |
|------|--------------|----------|
| Redis | 读热点、限流状态或可重建派生数据 | 一致性、故障回源、内存成本 |
| MQ/Stream | 写路径可异步、需要削峰或事件扇出 | 重复、顺序、堆积、恢复 |
| 读副本 | 读负载成为主库瓶颈且可接受复制延迟 | 读旧、故障切换 |
| 分片 | 单库在延迟、容量、恢复或写吞吐上达到实测瓶颈 | 路由、迁移、跨分片查询 |
| Bloom | 大量不存在 key 已被证明确实打穿存储 | 构建、更新、重建和健康降级 |
| CDN | 边缘命中能显著降低回源且统计语义可接受 | 失效、缓存一致性、统计缺口 |

禁止用固定行数或“某组件能扛十万 QPS”作为唯一理由。性能数字必须带硬件、数据、配置和测试方法。

不要为了“笔记完整”重复抄 FAQ、速记卡和模板。不会的地方回到具体不变量和故障窗口。

~~~powershell

.\99-工具与验证\verify_system_design.ps1 -Run
~~~

验证范围：

- UTF-8、H1、代码围栏；
- 本地 Markdown 链接；
- 禁止残留的扩写模板标记；
- Go 实验编译与运行；
- 关键数学和路由示例的运行结果。

验证通过不代表所有架构结论永远正确；产品版本、云服务和中间件特性仍需查官方文档。

## 8. 当前最小行动

如果正在学 Gin：

1. 暂不通读 02～07、09～10；
2. 完成 Gin users API；
3. 开 MySQL 与 Redis；
4. 阅读 [08 短链](./08-短链服务设计.md) 的 V1；
5. 跟 [Go/10](../Go/10-短链服务项目实战上.md) 自己实现；
6. 项目遇到具体瓶颈时，再回专题章节查阅。

系统设计的目标不是让项目看起来更复杂，而是让每个复杂度都有证据、有边界、可恢复。
