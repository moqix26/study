# 短链服务设计：从可用 V1 到可证明的扩展

> **定位**：本系列第一个完整 Case。先设计一个真实可实现的 Go 短链服务，再讨论缓存、异步统计、分片和 CDN。
> **项目实现**：[Go/10 短链项目上](../Go/10-短链服务项目实战上.md) 与 [Go/11 短链项目下](../Go/11-短链服务项目实战下.md)。
> **原则**：MySQL 保存正确性事实；Redis、队列、Bloom 和 CDN 都是可替换的性能层。

---

## 1. 学习分层

| 层级 | 现在需要掌握 | 暂时不要硬加 |
|------|--------------|--------------|
| V1 正确可用 | API、权限、随机短码、唯一索引、过期/禁用、幂等、302 跳转、测试 | Redis、MQ、Bloom、分片 |
| V2 简历级 | Cache Aside、负缓存、singleflight、原子限流、有界统计、指标、故障演练 | 多地域、全球 CDN |
| V3 扩展 | 可靠事件流、消费幂等、缓存版本、Outbox、真实压测 | 没有证据的“亿级高并发” |
| 面试进阶 | 固定逻辑槽、冷热数据、边缘统计、多地域取舍 | 把设计图冒充已实现成果 |

当前推荐顺序：

~~~text
需求与不变量
  → V1 单体 + MySQL
  → 测试证明正确
  → V2 Redis 与观测
  → V3 可靠统计
  → 最后才讨论分片、Bloom、CDN
~~~

## 2. 需求与不变量

### 2.1 核心功能

1. 用户注册、登录。
2. 创建自动短码或自定义短码。
3. 查询、分页、更新、禁用、启用、软删除自己的链接。
4. 访问公开短码并重定向到原始 URL。
5. 查看点击累计值及统计延迟说明。

### 2.2 非功能目标

| 目标 | V1/V2 建议 |
|------|-------------|
| 正确性 | 同一短码永远只对应一条历史链接；删除后不复用 |
| 隔离 | 管理 API 必须按 user_id 限定所有权 |
| 可用性 | Redis 故障时，核心映射仍可回源 MySQL |
| 延迟 | 先测单机基线，再设置 P95/P99 目标 |
| 一致性 | 映射以 MySQL 为准；缓存和统计允许明确的最终一致 |
| 安全 | 只接受 http/https，限制长度、保留字和创建速率 |

### 2.3 必须一直成立的不变量

- short_code 在数据库中唯一，且大小写规则明确。
- 用户不能查看或修改其他用户的资源。
- 过期、禁用、删除的链接不能继续跳转。
- 同一 Idempotency-Key 与相同请求重放时不重复创建。
- 同一 Idempotency-Key 配不同请求体时必须冲突。
- 缓存不能覆盖比自己更新的数据库版本。
- 点击统计失败不能拖死跳转核心路径。
- 所有“已实现”“提升多少”必须有测试或压测证据。

## 3. 容量估算：先把单位闭环

### 3.1 不要从 DAU 直接猜 QPS

短链系统至少需要区分三个量：

| 量 | 含义 |
|----|------|
| daily_new_links | 每天新增多少映射 |
| active_links | 当前仍会被访问的存量短链 |
| clicks_per_active_link | 活跃短链每天平均被点击多少次 |

日跳转量应由活跃数据集推导：

~~~text
daily_redirects = active_links × clicks_per_active_link
average_qps = daily_redirects / 86400
peak_qps = average_qps × peak_factor
~~~

不能只用“当天新增短链 × 每条点击数”，否则会漏掉历史存量链接。

### 3.2 示例估算

假设：

- 每日新增 1000 万条；
- 保留 5 年且暂不归档；
- 活跃短链 5000 万条；
- 每条活跃链日均 20 次跳转；
- 峰值系数 5。

~~~text
创建平均 QPS = 1000万 / 86400 ≈ 116
创建峰值 QPS ≈ 580

日跳转量 = 5000万 × 20 = 10亿
跳转平均 QPS ≈ 11574
跳转峰值 QPS ≈ 57870
~~~

这些只是面试假设，不是项目成绩。真实项目必须用固定环境压测。

### 3.3 存储估算

若单条映射连同必要索引粗估为 500B：

~~~text
总条数 = 1000万 × 365 × 5 = 182.5亿
原始逻辑容量 = 182.5亿 × 500B ≈ 9.1TB
~~~

实际规划还要乘：

- InnoDB 页和索引放大；
- 副本数；
- 备份保留；
- 迁移和故障余量。

因此可以得到数十 TB 的 provisioned capacity，但不能把 9.1TB 直接写成 91TB。

### 3.4 短码空间与碰撞

Base62 空间：

| 长度 | 空间 |
|------|------|
| 7 | 62^7 ≈ 3.52 万亿 |
| 8 | 62^8 ≈ 218.34 万亿 |

随机取码时，“整个生命周期是否出现过至少一次碰撞”和“单次插入的碰撞概率”是两件事。

生日近似：

~~~text
P(any collision) ≈ 1 - exp(-n(n-1)/(2M))
~~~

7 位空间中放入 10 亿条数据时，预期碰撞对约 14.2 万，至少一次碰撞的概率近乎 100%。这不代表随机方案不可用，而是说明：

1. 必须有数据库唯一索引；
2. 冲突后必须重新生成；
3. 需要限制重试次数并告警；
4. 不能写“概率极低，所以无需兜底”。

## 4. API 与资源边界

推荐与 Go 项目保持一致：

| Method | Path | 鉴权 | 说明 |
|--------|------|------|------|
| POST | /api/v1/auth/register | 否 | 注册 |
| POST | /api/v1/auth/login | 否 | 登录 |
| POST | /api/v1/links | 是 | 创建链接 |
| GET | /api/v1/links | 是 | 游标分页 |
| GET | /api/v1/links/:id | 是 | 详情 |
| PATCH | /api/v1/links/:id | 是 | 更新 URL 或过期时间 |
| POST | /api/v1/links/:id/disable | 是 | 禁用 |
| POST | /api/v1/links/:id/enable | 是 | 启用 |
| DELETE | /api/v1/links/:id | 是 | 软删除 |
| GET | /api/v1/links/:id/stats | 是 | 查询统计 |
| GET | /r/:code | 否 | 公开跳转 |

设计要点：

- 管理接口使用资源 id，不用 short_code 代替所有者边界。
- Repository 查询直接带 id + user_id + deleted_at 条件。
- 非所有者访问返回 404，避免泄露资源是否存在。
- 列表固定按 created_at DESC, id DESC，cursor 同时编码两个字段。
- 更新使用 version 或 If-Match，避免静默覆盖。
- Idempotency-Key 与“是否复用同一长链”是两个不同需求。

完整字段和迁移见 [Go/10 §2～§3](../Go/10-短链服务项目实战上.md)。

## 5. 短码生成方案

### 5.1 V1 推荐：密码学随机 Base62

流程：

~~~text
crypto/rand 生成固定 8 位 Base62
  → 直接 INSERT
  → 唯一索引成功：返回
  → duplicate key：重新生成，最多有限次
  → 其他错误：立即失败
~~~

不要先 SELECT 再 INSERT。并发下，检查与插入之间仍有竞争；唯一索引才是最终裁决者。

优点：

- 不依赖 Redis；
- 不连续，较难枚举；
- 多实例天然可生成。

代价：

- 会发生理论碰撞；
- 必须识别唯一键冲突；
- 需要有限重试和碰撞指标。

### 5.2 Counter + Base62

自增 ID 再 Base62：

- 无随机碰撞；
- 码长可变；
- 可枚举；
- Redis/号段服务成为创建依赖；
- 恢复时必须保证序号不倒退。

它适合做学习对照或经过设计的集中 ID 服务，不应在 Redis 故障时临时切换算法。

### 5.3 自定义短码

1. 校验字符、长度和保留词；
2. 直接 INSERT；
3. 唯一冲突返回 409；
4. 不自动替用户改名；
5. 删除后也不复用。

若支持多个短域名，唯一键应从 short_code 扩展为 domain_id + short_code。

## 7. 跳转语义：统计模式和边缘模式必须分开

### 7.1 项目精确回源模式

~~~http
HTTP/1.1 302 Found
Location: https://example.com/original
Cache-Control: no-store
~~~

特点：

- 每次跳转经过服务；
- 可以在服务端投递点击事件；
- 原站压力更高；
- 适合当前项目和可验证统计。

### 7.2 超大规模边缘缓存模式

~~~http
HTTP/1.1 302 Found
Location: https://example.com/original
Cache-Control: public, s-maxage=60
~~~

特点：

- CDN 命中时不会经过源站；
- 源站点击数不再完整；
- 统计必须依赖 CDN 日志、边缘 Worker 或明确采样；
- 统计会有延迟、丢失和对账边界。

不能同时宣称“CDN 缓存 302”和“每次都经过源站、统计完整”。

### 7.3 301、302、307

| 状态码 | 适用说明 |
|--------|----------|
| 301 | 永久跳转，客户端和中间层更可能长期缓存；不利于源站统计和目标变更 |
| 302 | 短链常用；缓存行为仍由响应头控制，不是天然绝不缓存 |
| 307 | 保留原 HTTP 方法；公开跳转接口本来就是 GET 时通常没有额外价值 |

## 8. V2：可降级的读路径

~~~mermaid
sequenceDiagram
    participant U as Browser
    participant S as Redirect Service
    participant R as Redis
    participant D as MySQL

    U->>S: GET /r/code
    S->>R: GET snapshot
    alt hit
        R-->>S: active / disabled / expired snapshot
    else miss or Redis error
        S->>D: SELECT by short_code
        D-->>S: row or not found
        S->>R: best-effort fill
    end
    S-->>U: 302 or 404
~~~

### 8.1 缓存内容

缓存的是带 version 的快照，而不是只有 long_url：

~~~json
{
  "version": 7,
  "status": "active",
  "original_url": "https://example.com",
  "expires_at": "2026-12-31T00:00:00Z"
}
~~~

禁用、删除、过期也应有短 TTL tombstone，防止旧值继续回填。

### 8.2 防穿透与击穿

- 不存在短码：短 TTL 负缓存。
- 同一实例并发 miss：singleflight 合并回源。
- TTL：基础值加随机抖动。
- Redis 故障：短超时后受控回源 MySQL。
- 回源必须有并发上限，不能让 Redis 故障变成 DB 雪崩。

singleflight 只合并单实例请求。没有压测证据时，不急着上分布式锁。

### 8.3 写后失效

简单的 UPDATE DB → DEL cache 仍有旧读回填窗口。V2/V3 可采用：

1. DB 每次变更 version + 1；
2. 同一事务写 cache_outbox；
3. worker 发布高 version 快照或 tombstone；
4. Redis Lua 拒绝低 version 覆盖高 version；
5. TTL 作为最终兜底。

MySQL 提交后、缓存更新前仍有最终一致窗口。必须在 README 说明，而不是宣称缓存强一致。

## 9. Bloom Filter：可选优化，不是正确性真相

Bloom 的语义：

- 返回“不存在”：在过滤器完整且更新健康的前提下，可认为不在集合中；
- 返回“可能存在”：仍需查缓存或数据库；
- 存在假阳性，不应存在算法层面的假阴性。

但业务系统会因为构建或更新失败产生“工程假阴性”：

~~~text
DB INSERT 已提交
  → 进程崩溃
  → Bloom 尚未 add
  → 若 Bloom false 直接 404，真实链接被误杀
~~~

因此只有满足以下条件时，Bloom 才能作为硬拒绝：

1. 已从完整快照构建；
2. 增量更新有 Outbox/CDC；
3. 有构建版本和健康状态；
4. 故障时自动降级为查 DB；
5. 支持重建和容量告警。

当前项目优先使用负缓存。只有压测证明随机不存在码确实打穿数据库，再考虑 Bloom。

## 10. 点击统计：先定义允许丢多少

### 10.1 V2 近似统计

使用固定 worker + 有界队列：

- Handler 非阻塞尝试入队；
- 队列满则丢弃并增加 dropped 指标；
- worker 聚合后批量写 MySQL；
- 关机按 deadline 排空；
- 进程崩溃时未落盘事件会丢。

这种方案只能叫“近似统计”。

### 10.2 V3 可靠消费

~~~mermaid
sequenceDiagram
    participant A as Redirect API
    participant Q as Redis Stream or MQ
    participant W as Worker
    participant M as MySQL

    A->>Q: publish event_id + short_code
    Q-->>A: accepted
    W->>Q: read batch
    W->>M: transaction: inbox unique + aggregate update
    M-->>W: commit
    W->>Q: ACK
~~~

关键边界：

- event_id 是业务幂等键，不能只使用 broker 的 delivery id；
- inbox 去重和 click_count 更新必须在同一事务；
- DB 提交后、ACK 前崩溃会重投，但不会重复累计；
- 事件尚未成功进入队列时仍可能少记；
- poison message 需要有限重试、DLQ、告警和可重放流程。

“至少一次 + 幂等”只有在去重记录和业务副作用具有原子边界时，才得到恰好一次效果。

## 11. 数据模型与分片

V1 核心模型至少包含：

- users；
- short_links；
- idempotency_records；
- 可选 link_dedup_keys；
- V2/V3 的 cache_outbox、click_event_dedup。

详细 SQL 见 [Go/10 §2.3](../Go/10-短链服务项目实战上.md) 和 [Go/11 §2、§4](../Go/11-短链服务项目实战下.md)。

### 11.1 什么时候才分片

不要按“单表超过固定行数”自动分片。观察：

- P95/P99 查询延迟；
- 索引和工作集是否能被内存容纳；
- IOPS、连接池、锁等待；
- 备份、DDL 和恢复时间；
- 增长率及未来容量；
- 热点是否集中在单个 short_code。

### 11.2 固定逻辑槽

需要水平分片时，优先固定逻辑槽：

~~~text
slot = hash(short_code) % 1024
physical_shard = slot_mapping[slot]
~~~

扩容时迁移部分 slot，并通过 CDC、校验和灰度切读完成闭环。不要以为一致性哈希能自动解决关系型数据库迁移、事务和索引问题。

管理端按 user_id 查询与跳转端按 short_code 查询是两种访问模式，可能需要：

- 可推导路由的资源 ID；
- 最终一致的路由索引；
- 独立管理查询库；
- 低频跨分片查询。

## 12. 安全与滥用治理

短链天然是开放重定向产品，安全目标不是“消灭重定向”，而是控制滥用：

- 只接受 http/https；
- 拒绝非法控制字符和超长 URL；
- 自定义短码保留系统路径；
- 用户、IP、设备多维限流；
- 钓鱼/恶意域名检测和申诉；
- 禁用链接的快速传播与缓存失效；
- 不在日志中记录 JWT、密码和完整敏感 query；
- Prometheus label 不使用 short_code，避免高基数；
- 管理 API 的 JWT 与所有者校验；
- 创建配额和批量接口审计。

若未来增加“抓取网页标题/预览”，才会引入服务端请求，此时必须单独防 SSRF、内网地址和重定向绕过。

## 13. 可观测与故障演练

### 13.1 最小指标

~~~text
http_requests_total{route,method,status}
http_request_duration_seconds{route,method}
link_cache_requests_total{result}
link_singleflight_shared_total
rate_limit_decisions_total{route,result}
click_events_total{result}
click_queue_depth
click_consumer_lag
db_query_duration_seconds{operation}
outbox_pending_events
~~~

### 13.2 健康检查

| Endpoint | 语义 |
|----------|------|
| /livez | 进程是否存活，不访问外部依赖 |
| /readyz | 是否能承接核心流量；MySQL 是硬依赖，Redis 可按降级策略报告 degraded |
| /metrics | Prometheus 抓取，生产限制访问范围 |

### 13.3 必做演练

| 故障 | 期望 |
|------|------|
| Redis 停止 | 回源 MySQL，DB QPS 上升但受控，不出现请求无限挂起 |
| stats worker 停止 | 跳转正常，lag 告警 |
| MySQL 断开 | 缓存命中可按策略继续；miss 和管理接口快速失败 |
| 旧快照延迟回填 | 低 version 被拒绝 |
| 统计队列灌满 | 302 不阻塞，drop 指标增加 |

演练记录必须包含假设、步骤、实际结果、差异和修复。

## 14. 面试时如何逐层展开

15 分钟回答顺序：

1. 确认创建、跳转、管理、统计范围。
2. 明确映射正确、统计可最终一致。
3. 估算新增量、活跃量、峰值 QPS 和逻辑容量。
4. 给出 API 和核心表。
5. 先画 V1 单体 + MySQL。
6. 读热点出现后加 Redis、负缓存和 singleflight。
7. 统计从有界近似版升级为可靠事件流。
8. 数据量真实到达瓶颈后再讲逻辑槽、归档和 CDN。
9. 最后说明故障、观测、安全与取舍。

常见红旗：

- 一上来就 Kafka、分库分表；
- 把 Redis 当正确性真相源；
- 认为 302 天然不缓存；
- 用概率代替唯一约束；
- 把示例容量写成自己的压测成绩；
- 宣称 exactly-once，却没有原子 inbox；
- Bloom 更新失败时仍直接拒绝 DB 查询。
