## 1. STAR 法则精讲

### 1.1 四要素

| 字母 | 含义 | 时长占比 | 要点 |
|------|------|----------|------|
| **S** Situation | 背景 | 15% | 谁、什么系统、规模 |
| **T** Task | 任务/问题 | 15% | 你要解决什么 **可衡量** 问题 |
| **A** Action | 行动 | 50% | **你** 做了什么（技术细节） |
| **R** Result | 结果 | 20% | 数字、对比、学到什么 |

### 1.2 好 STAR vs 坏 STAR

| 坏 | 好 |
|----|-----|
| 「做了一个 KV 项目」 | 「单机 KV，WAL 持久化，kill -9 后 replay 恢复」 |
| 「用了 epoll」 | 「epoll LT + 线程池，Get QPS 从 8k→35k」 |
| 「我们组做的」 | 「我负责 WAL 与 LRU 模块，CMake+gtest」 |

### 1.3 三分钟口述结构

```text
[0:00-0:20] S：项目定位 + 技术栈一句
[0:20-0:40] T：瓶颈/目标（数字）
[0:40-2:20] A：2～3 个技术行动（epoll/WAL/LRU）
[2:20-3:00] R：QPS/恢复/测试 + 反思
```

### 1.4 费曼检验

向非 C++ 同学讲一遍：若对方能复述「先写日志防丢数据」，则合格。

### 2.1 Situation 模板

- 「在自学/课程 **C++ 后端** 方向时，为补齐 **Infra 网络+存储** 能力，我实现了 …」

### 2.2 Task 模板

- 「初始版 **单线程阻塞** TCP，压测 QPS 仅 **X**，且 **kill -9 丢数据**。」
- 「需要在 **内存受限** 下支持 **1 万 key**，并保证 **崩溃可恢复**。」

### 2.3 Action 模板（Infra 高频动词）

| 动词 | 例句 |
|------|------|
| 设计 | 设计 length-prefix 协议解决粘包 |

- 「Release 压测 **Get QPS X**（N 并发），P99 **Y ms**。」

- 「gtest **Z** 用例，核心模块覆盖率 …（若有）」

## 3. KV-Store 完整 STAR（背诵版）

### 3.1 中文版（约 450 字）

**S**：我独立完成了一个 C++17 单机 KV-Store，作为 Infra 方向作品集，采用 TCP 自定义二进制协议，部署于 Linux/WSL。  

**T**：第一版单线程阻塞模型压测 Get QPS 约 8000，且进程被 kill 后内存数据全部丢失，无法满足「高性能 + 可恢复」目标。  

**A**：  
1. **网络**：改为 epoll LT 事件循环负责 accept/读帧，业务提交 **线程池**，避免 IO 阻塞 worker。  
2. **存储**：内存层 **unordered_map + LRU** O(1) 淘汰；**WAL** 先 append 日志再更新内存，启动 replay。  
3. **工程**：CMake 组织多目标；gtest 覆盖 LRU 淘汰、WAL 重放；客户端压测记录 QPS/P99。  

**R**：Release -O2 下 Get QPS 提升至 **3.5 万**（8 线程 client，示例数字请换实测）；`kill -9` 后重启 key 仍可 Get；形成 GitHub 可展示仓库与 README 压测表。

### 3.2 英文 elevator pitch（30s）

"I built a single-node C++17 KV store with epoll, thread pool, LRU eviction, and WAL for crash recovery. After moving from blocking IO to epoll plus worker threads, GET throughput reached about 35k QPS in our benchmark, and data survives kill -9 thanks to WAL replay."

### 3.3 1 分钟极简版

KV-Store：epoll 接客 + 线程池 + LRU + WAL。解决 QPS 低和数据丢失。压测 QPS X，kill 恢复 OK。

## 4. mini-http STAR（备选项目）

### 4.1 完整版

**S**：跟随路线完成 mini-http，单线程 TCP 返回 HTML/JSON。  
**T**：压测 RPS 约 800，Valgrind 报 fd 与 string 泄漏。  
**A**：RAII 封装 client fd；预生成静态 200 响应；perf 定位热点在字符串拼接；access.log（11 章）。  
**R**：Release RPS 1200+，memcheck 0 lost；掌握 HTTP 解析与性能分析流程。

### 4.2 与 KV-Store 如何选

| 场景 | 推荐主项目 |
|------|------------|
| Infra/后端/storage | KV-Store |
| 网络入门/demo | mini-http |
| 两个都有 | 简历都写，面试 **深讲 KV** |

### 5.2 KV-Store bullet 示例（三选一风格）

**偏网络**：

- 基于 **epoll LT + 线程池** 的 TCP KV 服务，自定义 length-prefix 协议；单机 Get QPS **35k**（8 线程压测，Release -O2）

**偏存储**：

- 实现 **LRU** O(1) 淘汰与 **WAL** 持久化，Put 先写日志再更新内存；**kill -9** 重启后 replay 恢复

**偏工程**：

- **CMake** 构建 server/client/tests；**gtest** 单测 LRU/WAL；**ctest** 本地 CI，README 含架构图与压测

## 6. Infra / 基建岗话术

### 6.1 岗位关注点

| 公司类型 | 爱问 |
|----------|------|
| 云/数据库 | 存储、WAL、并发、延迟 |
| AI Infra | epoll 网关、批调度、gRPC（19 章） |
| 中间件 | 协议、连接管理、资源上限 |
| 量化（部分） | 延迟、内存、无锁 |

### 6.2 自我介绍（90 秒模板）

「面试官好，我是 XXX，XX 学校 XX 专业。我主攻 **C++ 后端/Infra**，系统学习过内存模型、STL、并发与 epoll 网络编程。  

近期代表作是 **KV-Store**：用 **epoll + 线程池** 做接入，**LRU + WAL** 做存储，CMake 和 gtest 做工程化，单机 Get QPS 达到 **X**。  

之前还做过 **mini-http** 性能优化，熟悉 perf/Valgrind。希望能在贵司 **基础架构/存储/推理接入** 方向继续深入。谢谢。」

### 6.3 深挖预备答（10 题）

1. **为何 epoll 不用 ET？** LT 实现简单；ET 需读尽，教学/首版选 LT。  
2. **WAL 为什么先日志？** 崩溃时可 replay 已提交写。  
3. **LRU 线程安全？** 引擎 mutex；进阶分片锁。  
4. **fsync 每条太慢？** 可 batch，说明丢失窗口。  
5. **线程池大小？** 接近 `hardware_concurrency`，IO 密集可略大。  
6. **与 Redis 区别？** 学习项目，无 RESP/单线程事件库等。  
7. **最大技术难点？** 半包粘包 + 并发写 WAL 顺序。  
8. **如何压测？** 多线程 client，统计 QPS/延迟。  
9. **下一步？** snapshot 压缩 WAL、分片。  
10. **为什么选 C++？** 控制内存与延迟，Infra 栈主流。

## 7. 游戏岗话术（可选）

### 7.1 项目迁移

| KV-Store 点 | 游戏说法 |
|-------------|----------|
| LRU | 资源缓存 / 纹理热数据 |
| 线程池 | 任务调度 / Job System 简化版 |
| 内存池（34 章） | 对象池减少 GC 式卡顿 |
| OOP+vtable | 实体组件/技能系统 |

### 7.2 STAR 示例（对象池）

**S**：Demo 中大量子弹对象 new/delete 卡顿。  
**T**：帧时间尖刺 >5ms。  
**A**：固定块 **内存池** + placement new 复用。  
**R**：帧时间稳定，尖刺消失（Profiler 截图）。

### 7.3 游戏常考 C++（链 33 章）

- 虚函数与多态、智能指针、move、STL 容器选型
- 数据驱动：配置表 + 哈希加载
- **不要** 强行把 KV-Store 说成游戏引擎
