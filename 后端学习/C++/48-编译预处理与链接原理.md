## 2. 预处理阶段

编译器第一步：**预处理器**处理以 `#` 开头的指令，输出纯 C++ 文本（`.i`）。

```cpp
#define MAX(a,b) ((a) > (b) ? (a) : (b))
#define LOG(msg) std::cerr << __FILE__ << ':' << __LINE__ << ' ' << msg << '\n'

#ifdef DEBUG
    #define DBG(x) x
#else
    #define DBG(x)
#endif
```

| 指令 | 作用 |
|------|------|
| `#include` | 文本插入 |
| `#define` | 宏替换 |
| `#ifdef/#ifndef/#endif` | 条件编译 |
| `#pragma once` | 非标准但广泛支持的 include guard |
| `#error` | 编译期报错 |

**查看预处理结果**：

```bash
g++ -E main.cpp -o main.i
cl /P main.cpp
```

## 3. 宏：能力与陷阱

```cpp
#define SQUARE(x) ((x) * (x))
// 错误：#define SQUARE(x) (x * x)  // 宏不替括号会出错

#define CONNECT(a, b) a##b   // 令牌粘贴
```

经典陷阱：

```cpp
int a = 5;
std::cout << SQUARE(a + 1);  // 若缺括号 → 错误展开
```

**现代 C++**：用 `constexpr`、`inline`、`template` 替代宏；宏仅保留：`#ifdef` 平台分支、`__FILE__` 等。

### 3.1 函数式宏 vs inline 函数

| 宏 | inline 函数 |
|----|-------------|
| 无类型检查 | 有类型 |
| 多次求值副作用 | 单次求值 |
| 无作用域 | 有作用域 |

## 4. Include Guard 与头文件

```cpp
#ifndef PROJECT_UTILS_H
#define PROJECT_UTILS_H

void foo();

#endif
```

或：

```cpp
#pragma once
```

**Primer 规则**：
- 头文件放**声明**；源文件放**定义**
- 模板定义通常放头文件（实例化需要完整定义）
- 不要在头文件 `using namespace std;`

---
## 5. 翻译单元（Translation Unit）

每个 `.cpp`（经预处理后）是一个**翻译单元**，独立编译成 `.o`/`.obj`。

```text
main.cpp  ──→  main.o  ─┐
utils.cpp ──→ utils.o  ─┼──→  linker ──→ a.out / app.exe
foo.cpp   ──→  foo.o   ─┘
```

编译四阶段：**预处理 → 编译 → 汇编 → 链接**（前四步常合并说「编译」）。

```bash
g++ -c utils.cpp    # 只到 .o
g++ main.o utils.o -o app
```

## 6. One Definition Rule（ODR）

**ODR 核心**：整个程序中，非 inline 的函数/变量/类**最多一个定义**；声明可有多份。

违规示例：

```cpp
// header.h — 错误！
int global_counter = 0;  // 每个 include 的 .cpp 都有一份定义
```

正确：

```cpp
// header.h
extern int global_counter;  // 声明

// one.cpp
int global_counter = 0;     // 唯一定义
```

| 实体 | 头文件 | 源文件 |
|------|--------|--------|
| 函数 | 声明 | 一个定义 |
| inline 函数 | 定义可放头 | — |
| 模板 | 定义放头 | 可选显式实例化 |
| 全局变量 | `extern` 声明 | 一处定义 |
| `const` 常量 | `constexpr`/`inline` 变量(C++17) | — |

## 7. static 与 extern 链接属性

```cpp
static int file_local = 0;   // 内部链接：仅本 TU 可见
extern int shared;           // 外部链接：其他 TU 可链接
```

| 关键字 | 文件作用域 | 类成员 |
|--------|------------|--------|
| `static` | 内部链接 | 静态成员 |
| `extern` | 声明外部符号 | — |
| 匿名 namespace | 内部链接（C++ 推荐替代 static） | — |

```cpp
namespace {
    void helper() {}  // 仅本 TU
}
```

## 8. 声明与定义再辨析

```cpp
int add(int, int);           // 声明
int add(int a, int b) { return a + b; }  // 定义
```

- 调用前需**声明**可见（通常 `#include` 头文件）
- 链接器需要**定义**恰好一份（ODR）

## 9. 符号表与 Name Mangling

C++ 因重载、namespace 对符号**改名（mangling）**：

```bash
nm main.o
g++ -c foo.cpp && nm foo.o | c++filt
```

`extern "C"` 阻止 C++ mangling，供 C 库互操作：

```cpp
extern "C" void c_api(int x);
```

Windows `__declspec(dllexport)` / Linux visibility 控制导出符号。

## 10. 链接器 ld 做什么

链接器任务：
1. 合并各 `.o` 的 **sections**（`.text`, `.data`, `.bss`）
2. 解析**符号引用**：未定义符号必须在某 `.o` 或库中找到定义
3. 重定位：把符号地址填回指令
4. 生成可执行文件或共享库

**静态库** `libfoo.a`：`.o` 归档；链接时**按需**抽取成员。  
**动态库** `libfoo.so`/`.dll`：运行时加载；减小体积、可热更新。

```bash
g++ main.cpp -L. -lfoo -o app
```

## 11. 常见链接错误

| 错误 | 原因 |
|------|------|
| `undefined reference to foo` | 缺定义或未链接含 foo 的 .o/库 |
| `multiple definition of foo` | 违反 ODR |
| `duplicate symbol` | 头文件定义全局变量 |
| `.o` 架构不匹配 | 32/64 位混链 |

排查步骤：
1. `nm -C` 看符号是否存在 `U`（未定义） vs `T`（定义）
2. 确认 CMakeLists 是否链接目标库
3. 检查 `extern "C"` 与 C++ 声明是否一致

## 14. FAQ

**Q：#pragma once  portable 吗？**  
主流编译器支持；极老或特殊构建需 guard 宏。

**Q：inline 函数放头文件 ODR 安全吗？**  
C++ 允许多 TU 有相同 inline 定义，链接器合并。

**Q：模板为何放头文件？**  
实例化点需要完整定义；否则链接 undefined reference to `vector<int>::push_back`。
