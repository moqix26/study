## §1 为什么需要「编译」？

> 生活类比：菜谱不能直接上桌，必须经过烹饪（编译链接）才成为可吃的菜（可执行文件）。

### 术语三件套

| 维度 | 内容 |
| --- | --- |
| **定义** | 编译型语言先把整份源码翻译成机器码，再一次性运行。 |
| **生活类比** | 菜谱不能直接上桌，必须经过烹饪（编译链接）才成为可吃的菜（可执行文件）。 |
| **为什么重要** | 换来运行速度与编译期类型检查；代价是多一步构建。 |
| **本章用到** | §1 |

### 最小可运行例子

```cpp
#include <iostream>
int main() {
    std::cout << "Hello\n";
    return 0;
}
```

**逐行读代码**：
1. `#include <iostream>` 在编译前由预处理器展开。
2. `main` 是程序入口。
3. `return 0` 向系统报告成功。

#### 常见错误

1. 以为安装 VS Code 就等于能运行 C++——还需要编译器。
2. 双击 `.cpp` 期望运行——操作系统只认可执行格式。

#### 循序渐进练习

保存为 `hello.cpp`，先不编译，思考：扩展名 `.cpp` 传达了什么信息？

#### 本节小结

- 源码是给人和编译器看的「配方」，不是给 CPU 直接执行的指令。
- 编译失败多数是语法问题，不是电脑故障。

## §2 四阶段构建流水线

```mermaid
flowchart LR
    A[hello.cpp] --> B[预处理]
    B --> C[编译]
    C --> D[汇编]
    D --> E[链接]
    E --> F[可执行文件]
```

| 阶段 | 命令（g++） | 产物 | 做了什么 |
| --- | --- | --- | --- |
| 预处理 | `g++ -E hello.cpp -o hello.i` | .i | 展开 #include、处理 #define |
| 编译 | `g++ -S hello.i -o hello.s` | .s | 语法/语义检查，生成汇编 |
| 汇编 | `g++ -c hello.s -o hello.o` | .o/.obj | 汇编指令→机器码 |
| 链接 | `g++ hello.o -o hello` | exe/无后缀 | 合并目标文件与库 |

**生活类比**：预处理=备料；编译=写工序；汇编=切配；链接=摆盘出菜。

#### 循序渐进练习
对 Hello World 依次执行四步，用 `dir`/`ls` 比较文件大小。

## §4 g++ 一步编译与常用标志

> 生活类比：一条命令让四阶段在幕后连贯完成，像「一键出餐」。

### 术语三件套

| 维度 | 内容 |
| --- | --- |
| **定义** | `g++ -std=c++17 -Wall -Wextra hello.cpp -o hello` 生成可执行文件。 |
| **生活类比** | 一条命令让四阶段在幕后连贯完成，像「一键出餐」。 |
| **为什么重要** | `-Wall` 开启警告；`-g` 保留调试信息；`-o` 指定输出名。 |
| **本章用到** | §4 |

### 最小可运行例子

```cpp
// hello.cpp
#include <iostream>
int main() {
    std::cout << "Hello, world!\n";
    return 0;
}
```

**逐行读代码**：
1. `-std=c++17` 指定本教程语言标准。
2. `-o hello` 避免 Linux 默认 `a.out`。
3. Windows 运行 `hello.exe`；Unix 用 `./hello`。

#### 常见错误

1. `'g++' 不是内部或外部命令`：未安装或未加入 PATH。
2. Linux 下直接输入 `hello` 而非 `./hello`：找不到文件。

#### 循序渐进练习

加上 `-Wall -Wextra` 重新编译；故意留未使用变量，修到零警告。

#### 本节小结

- 学习阶段建议始终：`-std=c++17 -Wall -Wextra`。

## §5 Hello World 逐行精读

> 生活类比：第一次完整阅读 C++ 程序，像逐句念菜谱。

### 术语三件套

| 维度 | 内容 |
| --- | --- |
| **定义** | 标准结构：`#include`、命名空间、`main`、语句、`return`。 |
| **生活类比** | 第一次完整阅读 C++ 程序，像逐句念菜谱。 |
| **为什么重要** | 搞清每一行的角色，后面章节才不至于怕符号。 |
| **本章用到** | §5 |

### 最小可运行例子

```cpp
#include <iostream>
int main() {
    std::cout << "Hello, world!" << std::endl;
    return 0;
}
```

**逐行读代码**：
1. `std::cout`：`std` 命名空间中的字符输出流。
2. `<<` 插入运算符，把右侧数据送入流。
3. `std::endl` 换行并刷新缓冲区；可用 `'\n'` 替代。
4. 双引号内是字符串字面量；单引号用于单个字符（第92章）。

#### 常见错误

1. 漏写分号：`expected ';'`。
2. 把 `main` 写成 `Main`：链接阶段怪异错误。
3. 使用中文弯引号：非法字符。

#### 循序渐进练习

删掉 `#include <iostream>` 再编译，抄写报错并用自己的话解释。

#### 本节小结

- `main` 有且仅有一个；`return 0` 表示成功退出。

## §8 多文件与链接初体验

> 生活类比：大项目像多道菜组合宴席：各 `.cpp` 先做目标文件，链接器再合并。

### 术语三件套

| 维度 | 内容 |
| --- | --- |
| **定义** | 头文件声明接口，源文件实现；`#include` 把声明带给调用方。 |
| **生活类比** | 大项目像多道菜组合宴席：各 `.cpp` 先做目标文件，链接器再合并。 |
| **为什么重要** | 理解链接错误是进阶必备。 |
| **本章用到** | §8 |

### 最小可运行例子

```cpp
// add.h
#pragma once
int add(int, int);
// add.cpp
#include "add.h"
int add(int a, int b) { return a + b; }
// main.cpp
#include <iostream>
#include "add.h"
int main() {
    std::cout << add(2, 3) << "\n";
    return 0;
}
```

**逐行读代码**：
1. `#pragma once` 防止头文件重复包含。
2. 分步：`g++ -c add.cpp` `g++ -c main.cpp` `g++ add.o main.o -o app`。

#### 常见错误

1. 只编译 main.cpp 不链 add.cpp：`undefined reference to add`。
2. 头文件与实现签名不一致。

#### 循序渐进练习

实现上述三文件项目，故意去掉 add.cpp 的链接，观察链接错误。

#### 本节小结

- 声明在 .h，定义在 .cpp，调用方 `#include` 头文件。

## FAQ

**必须学命令行吗？**

建议至少走通一次；IDE 按钮背后是同样命令。

**a.out 是什么？**

Linux 默认输出名；用 -o 自定义。

**.cpp 与 .cc？**

都是 C++ 源扩展名。

**可以没有 main 吗？**

不可以，可执行程序需要入口。

**编译很慢？**

Hello World 瞬间完成；大项目用增量 -c。

**中文乱码？**

源 UTF-8；终端配置或先输出英文。

**看不懂英文报错？**

记 error、expected、undeclared 关键词。

**头文件与源文件？**

.h 声明，.cpp 实现。

本章建立「源码→可执行文件」全景图，掌握 g++ 一步/分步构建，逐行读懂 Hello World，
并学会读报错与跨平台常识。请完成闭卷自测后进入
[第92章 · 数据类型与变量完全基础](92-数据类型与变量完全基础.md)。

## 深度阅读：分文件编译中的符号可见性

只有被声明且定义的符号才能被链接器解析。

若 main 调用 `int mul(int,int);` 但无 mul.cpp，链接报 undefined reference。

```cpp
g++ -c main.cpp 成功；g++ main.o -o app 失败。
```

**实践**：补全 mul.cpp 或去掉调用。

---

## 深度阅读：预处理器宏与条件编译初窥

`#define PI 3.14` 纯文本替换；`#ifdef DEBUG` 可选编译调试代码。

宏在预处理阶段处理，无类型检查——初学少用大宏。

```cpp
#ifdef DEBUG
cerr<<"dbg\n";
#endif
```

**实践**：用 -DDEBUG 编译观察差异。

---

## 深度阅读：可执行文件与脚本返回值

Shell 用退出码判断程序是否成功；自动化测试依赖 return 0。

CI 流水线编译后运行测试 exe，非零 return 即失败。

```cpp
return 42;
```

**实践**：写脚本调用并检查 ERRORLEVEL。

---

## 深度阅读：编辑器、编译器、调试器三角

编辑器写字；编译器翻译；调试器在 -g 下按行停住。

三者分工不同，缺编译器则 IDE 只能当记事本。

```cpp
g++ -g -O0 app.cpp -o app
```

**实践**：gdb/lldb 下一章深入，本章先 -g 编译。

## 深度阅读：从源码到运行的五分钟清单

保存→编译→看警告→运行→对照预期→改代码。

养成「零警告」习惯，比以后改坏习惯容易。

```cpp
g++ -Wall -Wextra -std=c++17
```

**实践**：打印贴在显示器旁。

## 深度阅读：iostream 为何需要链接

cout 实现在标准库中，你的 .o 只有调用点，链接器把库函数地址填上。

所以「只编译不链接」得不到完整程序。

**实践**：用 nm 看 .o 中 cout 为 U（未定义）。
