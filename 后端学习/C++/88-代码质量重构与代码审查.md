## §1 代码异味（Code Smells）

| 异味 | 识别信号 | 常用重构 |
|------|----------|----------|
| 长函数 Long Method | 函数 > 40 行；多层嵌套 | 提取函数 Extract Method |
| 大类 God Class | 一个类承担过多职责 | 提取类 Extract Class |
| 重复代码 Duplicated Code | 复制粘贴块 | 提取函数/模板 |
| 过长参数列表 Long Parameter List | >4 个参数 | 引入参数对象 Parameter Object |
| 散弹式修改 Shotgun Surgery | 改一处动多处 | 搬移方法 Move Method |
| 依恋情结 Feature Envy | 频繁访问别类数据 | 搬移函数 |
| 数据泥团 Data Clumps | 总是一起出现的字段 | 提取类/value object |
| 基本类型偏执 Primitive Obsession | 用 int/string 代替类型 | 引入 NewType |
|  switch 泛滥 | 同一 switch 多处 | 多态 Replace Conditional with Polymorphism |
| 注释过多 Comments | 用注释解释烂代码 | 重命名+提取函数 |

## §2 重构手法详解

### 2.1 提取函数 Extract Method

```cpp
// Before
void process(Request& req) {
  // validate 20 lines...
  // parse 20 lines...
}

// After
void validate(Request& req);
void parse(Request& req);
void process(Request& req) {
  validate(req);
  parse(req);
}
```

### 2.2 内联函数 Inline Method

函数体过薄且名字无增益时 inline；或相反：过短多次调用可内联减开销（谨慎）。

### 2.3 引入参数对象 Introduce Parameter Object

```cpp
// Before
void draw(int x, int y, int w, int h, Color c, int border);

// After
struct Rect { int x,y,w,h; };
struct DrawStyle { Color c; int border; };
void draw(const Rect& r, const DrawStyle& s);
```

### 2.4 以多态取代条件 Replace Conditional with Polymorphism

见 [43 章](43-继承与多态模型完全指南.md) 状态模式。

### 2.5 搬移方法 Move Method

函数更常用别类数据时，移到那个类。

### 2.6 提取接口 Extract Interface

依赖抽象 `IWriter` 而非 `FileWriter`（DIP）。

### 2.7 封装字段 Encapsulate Field

public 字段改 private + getter；线程安全另议。

### 2.8 替换算法 Substitute Algorithm

更清晰/更快的 std::algorithm 替代手写循环。

## §3 重构安全网：测试先行

1. 重构 **不改外部行为**；先 green test。
2. 一步一提交；每步可 revert。
3. 无测试时先 ** characterization test ** 锁行为。
4. 链 [27 章 gtest](27-Google-Test与单元测试工程.md)：`TEST(Foo, Bar)`。

## §4 SOLID 与 C++ 落地

### S 单一职责 SRP

- **定义**：类只有一个变更理由
- **C++ 例**：HttpServer 又解析又写 DB → 拆分 Parser/Handler/Repo

### O 开闭 OCP

- **定义**：对扩展开放对修改关闭
- **C++ 例**：用策略接口加新压缩算法不改调用方

### L 里氏替换 LSP

- **定义**：子类可替父类
- **C++ 例**：Square 继承 Rectangle 改宽高语义破坏 LSP

### I 接口隔离 ISP

- **定义**：不强迫依赖无用接口
- **C++ 例**：胖接口 `IWorker` 拆成 `IRead`/`IWrite`

### D 依赖倒置 DIP

- **定义**：依赖抽象非细节
- **C++ 例**：高层依赖 `ILogger` 非 `SpdlogLogger`

## §5 clang-tidy 工程化

```bash
cmake -DCMAKE_EXPORT_COMPILE_COMMANDS=ON -B build
clang-tidy -p build src/*.cpp --checks='-*,bugprone-*,modernize-*,performance-*'
```

| Check | 含义 |
|-------|------|
| `modernize-use-override` | 显式 override 关键字 |
| `modernize-use-nullptr` | NULL → nullptr |
| `performance-move-const-arg` | 可 move 处提示 |
| `bugprone-use-after-move` | move 后使用 |
| `readability-identifier-naming` | 命名规范 |
| `cppcoreguidelines-owning-memory` | 原始 owning 指针 |

## §6 cppcheck 补充

```bash
cppcheck --enable=all --std=c++17 -I include src/
```
与 clang-tidy 互补：未初始化、越界、资源泄漏等。

## §8 Code Review 清单

1. [ ] 逻辑正确性：边界、空指针、错误码
2. [ ] 资源：RAII、无泄漏、异常安全
3. [ ] 并发：锁顺序、数据竞争、原子
4. [ ] API：命名、const 正确性、 noexcept 承诺
5. [ ] 性能：不必要拷贝、reserve、算法复杂度
6. [ ] 测试：新行为有测试；回归
7. [ ] 安全：输入校验、整数溢出
8. [ ] 可读性：函数长度、魔法数命名
9. [ ] 风格：clang-format 一致
10. [ ] 文档：公共 API 注释、CHANGELOG

## §11 FAQ

1. **重构会不会引入 bug？** 小步+测试；无测试先补 characterization test。
2. **inline 一定更快吗？** 不一定；LTO 可能内联；过大函数 inline 增代码体积。
3. **SOLID 是否过度设计？** 小脚本可简化；Infra/长期维护项目收益大。
4. **clang-tidy 误报？** 用 NOLINT 注释或调整 check；理解规则意图。
5. **Review 太费时？** 小 PR、清单化、自动化静态分析先过滤。
6. **与 87 章关系？** 读开源见异味；本章系统改。
7. **参数对象 vs struct？** 参数对象强调语义聚合+校验；普通 struct 也可。
8. **何时不重构？** deadline 紧且无测试；先标记 TODO+issue。
9. **God Class 如何拆？** 按职责 Extract Class；配合 SRP。
10. **cppcheck 与 ASan？** 静态 vs 动态；都进 CI 最佳。

8. characterization test 是什么？

10. God Class 常用哪种重构？

### 参考答案

1. 长函数/大类/重复/长参数/散弹式修改等。
2. 长函数拆分用提取；过薄无意义函数可内联。
3. 参数过多、语义分散、难以扩展。
4. 子类可替换父类且不破坏契约。
5. `cmake -DCMAKE_EXPORT_COMPILE_COMMANDS=ON`。
6. 逻辑/资源/并发/API/性能/测试/安全/可读/风格/文档。
7. 重构不改行为；优化改性能可能改实现。
8. 锁定现有行为的测试，重构前写入。
9. 派生函数显式加 override。
10. Extract Class 或按 SRP 拆分。

| 原则 | 追问 |
