## 2. 对象内存布局基础

### 2.1 成员顺序与 padding

```cpp
#include <cstddef>
#include <iostream>

struct PodLike {
    char c;    // offset 0
    // 3 bytes padding
    int i;     // offset 4
};  // sizeof 通常 8

class WithVptr {
public:
    virtual void f() {}
    int x = 0;
};

int main() {
    std::cout << "PodLike: " << sizeof(PodLike) << '\n';
    std::cout << "WithVptr: " << sizeof(WithVptr)
              << " align=" << alignof(WithVptr) << '\n';
}
```

**规则速记**：成员顺序决定 padding；有 virtual → 插入 **vptr**（通常对象开头）；空类 `sizeof==1`。

```cpp
class WithVptr { public: virtual void f() {} int x = 0; };
// 64 位典型： [ vptr | x | padding ]
```

## 3. 虚函数表（vtable）与 vptr

### 3.1 编译器模型（Itanium ABI 概念，主流 Linux/macOS）

```text
类 Animal 有 virtual void speak()
  → 编译器生成 Animal::vtable[] = { &Animal::speak, &Animal::~Animal, ... }
  → 每个 Animal 对象头部含 vptr → 指向 Animal::vtable

class Dog : public Animal { void speak() override; }
  → Dog::vtable[] = { &Dog::speak, &Dog::~Dog, ... }
  → Dog 对象 vptr → Dog::vtable
```

```cpp
struct Base {
    virtual void f() { puts("Base::f"); }
    virtual ~Base() = default;
    int bx = 1;
};

struct Derived : Base {
    void f() override { puts("Derived::f"); }
    int dy = 2;
};

// 典型布局（64 位，示意）：
// Derived 对象: [ vptr | bx | dy ] + 可能的 padding
```

### 3.2 动态绑定

```text
Base* p = new Derived;
p->f()  →  读 p 的 vptr  →  查 Derived::vtable  →  调 Derived::f
```

**开销**：间接加载 + 间接跳转；LTO/devirtualization 可能优化。

### 3.3 虚析构

```cpp
Base* p = new Derived;
delete p;  // 若 ~Base 非 virtual → 只析构 Base 部分 → 泄漏/UB
```

**面试金句**：多态基类指针 `delete` 时，**析构函数必须 virtual**，否则 Derived 资源泄漏。

### 3.4 构造/析构期间 vptr

对象构造顺序：**基类 → 成员 → 派生**。在基类构造函数里调用虚函数，**绑定到基类版本**（vptr 尚未完全指向最终类型）。析构同理——勿在基类析构里调虚函数 expecting 派生行为。

## 4. RTTI 与 dynamic_cast

### 4.1 typeid 与 type_info

```cpp
#include <typeinfo>
#include <iostream>

struct Poly { virtual ~Poly() = default; };
struct Concrete : Poly {};

void demo(const Poly& p) {
    std::cout << typeid(p).name() << '\n';  // 运行时：实际动态类型
}
```

- **前提**：类至少有一个 **虚函数**（通常虚析构即可）才有多态 RTTI。
- `typeid(*ptr)` 对多态对象查 **动态类型**；对非多态为静态类型。

### 4.2 dynamic_cast

```cpp
Poly* p = new Concrete;
auto* c = dynamic_cast<Concrete*>(p);  // 成功
Poly* q = new Poly;
auto* bad = dynamic_cast<Concrete*>(q); // 失败 → nullptr

// 引用版失败抛 std::bad_cast
```

**实现概念**：vtable 旁常挂 **type_info 指针**（ABI 相关）；`dynamic_cast` 沿继承图 walk，比 `static_cast` 慢。

| cast | 运行时检查 | 典型用途 |
|------|------------|----------|
| `static_cast` | 否 | 已知安全上行/数值转换 |
| `dynamic_cast` | 是 | 下行多态、交叉 cast（MI） |
| `reinterpret_cast` | 否 | 底层比特（危险） |
| `const_cast` | 否 | 去 const（谨慎） |

### 4.3 关闭 RTTI

`-fno-rtti` 可减小二进制；`dynamic_cast` / `typeid` 受限。游戏/嵌入式常见；Infra 服务端一般 **保留 RTTI** 便于调试。

## 5. 多重继承（MI）

### 5.1 非虚多重继承布局

```cpp
struct A { virtual void fa() {} int a; };
struct B { virtual void fb() {} int b; };
struct C : A, B { int c; };

// C 对象示意（两 vptr）：
// [ A 子对象: vptr_A | a ] [ B 子对象: vptr_B | b ] [ c ]
```

```cpp
C obj;
A* pa = &obj;           // 通常与 C* 同地址
B* pb = &obj;           // 地址可能 != C*（指针调整 thunks）
C* pc = static_cast<C*>(pa); // 已知从 A 子对象还原
```

**面试必画**：`B*` 与 `C*` 的 **地址差**（offset to top）。

### 5.2 菱形继承与虚继承

```cpp
struct Base { int v; virtual void f() {} };
struct Left : virtual Base { };
struct Right : virtual Base { };
struct Bottom : Left, Right { };

// 无 virtual 继承 → Base 两份 → 歧义
// virtual 继承 → 共享一份 Base，布局更复杂（vbptr 等）
```

```text
Bottom 对象（概念）：
  Left 部分 + Right 部分 + 共享 Base + vbptr 链
```

**工程建议**：Infra 代码 **少 MI**；接口用组合 + 单一继承；必须 MI 时优先 **virtual public** 基类。

## 6. 空基类优化（EBO）

```cpp
struct Empty { };
struct Holder {
    Empty e;   // sizeof(Holder) 可能 == 1（无 EBO 时 Empty 占 1）
    int x;
};

template<class Tag, class Base = Empty>
struct Compact : Base {  // 空 Base → EBO：Base 不占额外字节
    int value;
};

static_assert(sizeof(Compact<int>) == sizeof(int));  // 常见编译器
```

| 类型 | 无 EBO | 有 EBO |
|------|--------|--------|
| `std::function` 小对象 | 仍可能堆分配 | 与空 deleter 组合省空间 |
| 自定义 allocator 空类 | 每个容器 +1B？ | 零大小嵌入 |

**注意**：EBO 是 **编译器优化**，非标准强制；同类型多 Empty 基类可能不全部 EBO。

## 7. CRTP 与 devirtualization（速览）

```cpp
template<class Derived>
struct CRTPBase {
    void interface() { static_cast<Derived*>(this)->impl(); }
};
```

**CRTP** 编译期绑定、可内联；热路径可用 `final` + LTO 帮助编译器 **去虚化** 虚调用。

## 12. 闭卷自测

1. 何时对象含 vptr？
2. vtable 存什么？每对象几份？
3. 为什么多态基类需要 virtual 析构？
4. `dynamic_cast` 需要什么前提？
5. 菱形继承为何常用 virtual 继承？
6. EBO 解决什么问题？
7. 构造 Base 子对象时调虚函数绑到哪？
8. MI 下 `B* pb = &c_obj` 与 `C* pc = &c_obj` 地址关系？
9. `-fno-rtti` 影响哪些运算符？
10. 虚函数调用相对直接调用的主要开销？

<details>
<summary>自测参考答案</summary>

1. 类或其基类声明了 **virtual 成员函数**（含虚析构）。
2. **函数指针表**（含 typeinfo 等 ABI 扩展）；**每类一份**，所有对象共享。
3. `delete Base*` 时若析构非 virtual，只执行 Base 析构，**Derived 成员泄漏**。
4. 多态类型（通常基类有虚函数）；指针/引用转换时 **运行时检查** 继承关系。
5. 避免 **Base 子对象重复** 导致的歧义与双倍成员。
6. **空类基类不占空间**，紧凑封装 policy（如 allocator、deleter）。
7. 绑定到 **当前正在构造的类** 的版本（Base），非最终 Derived。
8. **可能不相等**；`B*` 指向 B 子对象起始，相对 `C*` 有 offset（thunk/adjust）。
9. **`dynamic_cast`、`typeid`（多态）** 等 RTTI 特性受限或不可用。
10. **两次间接**（读 vptr → 读 vtable 条目 → 跳转），且难内联。

</details>

## 13.1 MSVC vs Itanium ABI

|  | Itanium (Clang/GCC Linux) | MSVC |
|---|---|---|
| vtable 指针位置 | 对象起始（单继承） | 对象起始 |
| 多重继承 | secondary vptr + thunks | 类似，细节不同 |
| RTTI | vtable 负偏移 typeinfo | 类似概念 |
| name mangling | Itanium ABI 规则 | MSVC 自有规则 |

## 13.2 多重继承内存布局图

class B1 { virtual void f(); int b1; };

class D : public B1, public B2 { int d; };

D 对象: [vptr_B1][b1][vptr_B2][b2][d]
B2* p = &d_obj; 地址可能 != D* 地址（adjustor thunk）

## 13.3 虚继承与 vbptr

菱形继承：`virtual public B` → **vbptr** 指向 vbase offset table，保证唯一 Base 子对象。
`sizeof(D)` 含 vbptr + vptr + 成员 + padding。

---

## 13.4 RTTI type_info

每个多态类对应 `type_info`；`typeid(*p)` 查 vtable 关联 typeinfo。
`-fno-rtti` 关闭以减小二进制，失去 `dynamic_cast` 运行时检查。

---

## 13.5 dynamic_cast 实现

运行时沿继承图 walk，比对 typeinfo；跨类转换失败返回 **nullptr**（指针）或 **bad_cast**（引用）。
需 **多态类型**（有虚函数）才可用运行时 dynamic_cast。

```cpp
// 编译器生成 typeinfo 与 hierarchy 信息
Base* pb = ...;
if (Derived* pd = dynamic_cast<Derived*>(pb)) {
    pd->derived_only();
} else {
    // 转换失败
}
// static_cast 不做运行时检查 —— MI 下可能编译通过但 UB
```

## 13.5.1 虚继承布局 ASCII 图

```
class A { virtual void a(); };
class B : virtual public A {};
class C : virtual public A {};
class D : public B, public C {};

D: [B 子对象][C 子对象][D 成员][共享 A 子对象（唯一）]
     vbptr_B → vbase offset
     vbptr_C → vbase offset
```

## 13.5.2 thunk 与 adjustor

MI 下 `B* → D*` 可能需要 **adjustor thunk**：修正 this 指针偏移后再跳转到真实函数。
面试：看到 vtable 条目是 thunk 而非直接函数地址，说明存在多重继承调整。
