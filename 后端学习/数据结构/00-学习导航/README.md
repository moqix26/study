# 学习导航与使用说明

## 1. 这套资料解决什么问题

数据结构回答“数据怎样组织”，算法回答“操作怎样进行”。学习时最容易陷入两个极端：

- 只背模板，不知道为什么正确；
- 只看原理，真正编码时边界条件全错。

本库采用同一条学习闭环：

```text
场景问题 → 数据组织方式 → 核心不变量 → 操作步骤
       → 复杂度 → 三语言实现 → 测试 → 变种题
```

## 2. 八阶段路线

| 阶段 | 内容 | 过关标准 |
|---:|---|---|
| 1 | 复杂度、数组、链表 | 能分析循环与递归；能处理空表、头尾节点 |
| 2 | 栈、队列、哈希 | 能说明 LIFO/FIFO；能解释冲突与负载因子 |
| 3 | 二叉树、BST、堆 | 能写遍历；能说清 BST 与堆的有序性差异 |
| 4 | 排序、查找 | 能手写归并/快排/二分并分析稳定性 |
| 5 | 图遍历、拓扑 | 能根据稀疏/稠密选存储；能处理环与多连通分量 |
| 6 | MST、最短路 | 能说明各算法前提，避免把负权图交给 Dijkstra |
| 7 | AVL、红黑树、并查集、Trie | 能讲不变量、旋转/压缩与典型场景 |
| 8 | 滑窗、回溯、分治、DP | 能从暴力解法推导状态与剪枝 |

## 3. 每次学习一个主题的步骤

### 3.1 第一次：建立模型

只关注四个问题：

1. 它解决什么场景？
2. 数据在内存或逻辑上怎样组织？
3. 必须始终保持什么不变量？
4. 查询、插入、删除分别要付出什么代价？

### 3.2 第二次：手动跟踪

选 5～8 个数据，在纸上或注释中走一遍：

- 指针怎样变化；
- 队列什么时候入队出队；
- 树旋转前后高度怎样变化；
- 图算法中的 `visited`、入度、距离数组怎样变化。

### 3.3 第三次：运行三语言代码

不要求同时熟练三门语言。建议：

- 主语言完整手敲；
- 第二语言对照类型、容器与内存管理差异；
- 第三语言能读懂并成功运行。

### 3.4 第四次：主动修改

至少完成一种改动：

- 改输入数据并预测输出；
- 增加一个操作或测试；
- 故意制造边界错误再修复；
- 把递归改为迭代，或反过来。

## 4. 三语言对照重点

| 维度 | C++ | Java | Go |
|---|---|---|---|
| 动态数组 | `vector` | `ArrayList` / 数组 | slice |
| 哈希表 | `unordered_map` | `HashMap` | `map` |
| 队列 | `queue` / `deque` | `ArrayDeque` | slice + head 或自建队列 |
| 堆 | `priority_queue` | `PriorityQueue` | `container/heap` |
| 内存 | 值、指针、RAII | 对象引用 + GC | 值、指针 + GC |
| 泛型 | template | generics | type parameters / 具体类型 |

学习自定义结构时不要只调用标准容器。文档会在“理解实现”与“工程调用”之间明确区分。

## 5. 复杂度书写约定

- `n`：元素数量；
- `V`：图的顶点数；
- `E`：图的边数；
- `h`：树高；
- `k`：键值范围、桶数或字母表大小；
- 空间复杂度默认指算法使用的**额外空间**；
- 递归必须计入调用栈。

## 6. 树与图为什么要精读

线性结构中，通常只需关注前后相邻元素；树与图同时存在多条分支，错误更隐蔽：

- 树要维护结构不变量、高度或颜色；
- 图可能有环、多个连通分量、方向和权重；
- 同一个算法在不同前提下可能正确，也可能完全错误。

因此树与图文档不仅给模板，还会解释：

- 不变量为什么足以保证正确性；
- 每次旋转、松弛、入队、合并到底改变了什么；
- 算法适用前提不满足时会怎样失败；
- 如何选择邻接矩阵/邻接表、BFS/DFS、Prim/Kruskal、Dijkstra/Floyd。
