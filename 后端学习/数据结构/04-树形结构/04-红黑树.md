# 红黑树

## 学习目标

- 理解红黑树的五条颜色不变量，以及它们如何限制树高。
- 掌握插入后“叔叔为红”和“叔叔为黑”两大修复分支。
- 理解旋转只改变结构、变色只改变黑高分配，两者如何配合恢复不变量。
- 能运行并验证真正的红黑树插入、查找和遍历实现，而不是调用标准容器代替。
- 知道删除中的“双黑”含义及主要修复思路。

## 1. 原理讲解：红黑树不变量

红黑树首先是一棵二叉搜索树，并为每个节点增加红/黑颜色。普通 BST 只维护键的顺序，无法阻止单调输入形成长链；红黑树用颜色约束控制根到叶路径的相对长度，在不要求像 AVL 那样严格等高的情况下，仍保证最坏 `O(log n)`。

本文使用一个共享的黑色哨兵 `NIL` 表示所有空孩子。真实叶子节点的左右孩子也指向 `NIL`，而不是普通空指针。这样“孩子颜色”“从节点到叶子的黑高”等逻辑在边界处仍有统一定义。

红黑树必须同时满足以下规则：

1. 每个节点非红即黑。
2. 根节点是黑色。
3. 所有 `NIL` 叶子是黑色。
4. 红节点的两个孩子都是黑色，即不能出现连续红节点。
5. 从任一节点到其所有后代 `NIL` 的路径包含相同数量的黑节点，这个数量称为黑高。

第 1 条规定颜色状态完整；第 2 条让整棵树有统一起点；第 3 条把缺失孩子也纳入黑高计算；第 4 条禁止连续红层；第 5 条要求任一节点的左右方向具有相同黑色骨架。真正限制高度的是第 4、5 条的配合，其余规则让定义和修复过程保持一致。

黑高可以约定“是否计入起点节点”，不同教材写法略有差异。为了与下面的节点数推导对齐，本文令 `bh(x)` 统计从 `x` 开始但不计 `x` 本身、直到后代 `NIL` 且计入 `NIL` 的黑节点数。后面的验证函数对黑色当前节点返回“孩子返回值 + 1”；因此父节点拿到的左右返回值，正好是它两个分支的黑高。若采用“计入起点”的约定，数值只会整体平移一个常数，`O(log n)` 结论不变。

颜色不是键值大小的一部分。旋转负责改变结构但保持 BST 中序顺序；重新着色不改变结构，却重新分配路径上的黑节点。插入和删除修复正是把这两类局部操作组合起来恢复全部不变量。

与 AVL 不同，红黑树不要求左右子树高度差不超过 1。它允许更宽松的形状，以减少更新时的结构调整。

`NIL` 哨兵在实现中通常是单个共享对象：颜色恒为黑，左右孩子可指向自身。旋转和修复逻辑可以直接读取 `node.left.color`，无需对每个空孩子写额外分支。代价是所有位置都必须统一使用哨兵，不能有些地方用 `NIL`、有些地方用普通空指针。

## 2. 高度为何是 `O(log n)`

先忽略红节点，把每个红节点与它的黑色父节点合并，可以把红黑树理解为一棵 2-3-4 树：一个黑层可能对应 2、3 或 4 个分支。由于所有根到叶路径黑高相同，黑色骨架是平衡的。

若某节点黑高为 `bh`，它的子树至少有 `2^bh - 1` 个内部节点，所以 `bh <= log2(n+1)`。又因为红节点不能连续，每两个相邻黑节点之间至多夹一个红节点，普通树高 `h <= 2*bh`，于是：

`h <= 2*log2(n+1) = O(log n)`。

因此查找、插入和删除的下降路径都有严格的最坏 `O(log n)` 上界。

上面两个不等式分别来自不同不变量：

1. **黑高迫使节点数指数增长。** 黑高为 `bh` 的最小子树，其左右子树都至少具有黑高 `bh-1`。设最少内部节点数为 `M(bh)`，则 `M(bh) >= 1 + 2*M(bh-1)`，展开后得到至少 `2^bh-1` 个节点。
2. **红节点不能把路径无限拉长。** 一条路径上若有 `bh` 个黑节点，任意两个黑节点之间至多出现一个红节点，否则会产生红红冲突。因此红节点数不超过黑节点数量的同阶上界，总路径长度至多约为黑高的两倍。

例如某节点到所有 `NIL` 的路径都有 3 个黑层，那么最短路径可以几乎全黑，最长路径可以在相邻黑层间各插一个红节点，但不能出现 `黑-红-红-黑`。所以最长路径最多约是最短黑色骨架的两倍，而不会像普通 BST 那样任意增长到 `n`。

把红节点向上与黑父节点合并的 2-3-4 树视角也能解释同一事实：黑节点自身对应一个 2-节点；带一个红孩子可对应 3-节点；带两个红孩子可对应 4-节点。所有叶子处于相同黑高，说明对应的多路树所有叶子在同一层。红黑树的变色与旋转，可以理解为多路节点的分裂和重新组织。

## 3. 插入：先按 BST，再修复红红冲突

新节点按 BST 规则作为叶子插入，并染成红色。选择红色的原因是：红节点不增加任一路径的黑节点数，所以第 5 条黑高不变量暂时不变。可能被破坏的只有第 2 条（新节点成为根）或第 4 条（父子连续红）。

若父节点是黑色，无需修复。若父节点是红色，则祖父一定存在且为黑色，接下来观察叔叔颜色。

“祖父一定为黑”来自红节点不能有红孩子：当前父节点已经是红色，因此其父亲若存在只能是黑色。叔叔是祖父的另一个孩子。插入修复每轮只需查看新节点、父、祖父、叔叔四个局部角色。

插入过程可以按以下顺序理解：

1. 按 BST 规则下降到 `NIL` 位置；
2. 建立红色新节点，其左右孩子均指向黑色 `NIL`；
3. 若新节点是根，染黑后结束；
4. 若父为黑，所有性质已经成立；
5. 若父为红，根据叔叔颜色和“直线/折线”形状修复；
6. 循环结束后再次强制根为黑，处理变色传播到根的情况。

### 3.1 叔叔为红：变色并向上传播

把父亲和叔叔染黑，祖父染红。这样经过祖父的每条路径黑节点数保持一致，但祖父可能又与它的父亲形成红红冲突，因此把祖父当作新节点继续向上检查。最后强制根为黑色。

假设局部结构如下，`P` 是父、`U` 是叔、`G` 是祖父：

```text
        G(B)                         G(R)
       /    \                       /    \
    P(R)    U(R)      变色       P(B)    U(B)
    /
  X(R)
```

变色前，从 `G` 经过 `P` 或 `U` 向下的每条路径都包含 `G` 这一层黑色。变色后 `G` 变红，但 `P、U` 都增加一层黑色，所以每条路径的黑节点总数仍不变；同时 `X-P` 的红红冲突消失。唯一的新风险是 `G` 变红后可能与更高处的红色父节点冲突，因此问题向上传播，而不是在当前局部继续旋转。

若 `G` 恰好是根，最后把它重新染黑。根由红变黑会让所有根到叶路径同时增加一个黑节点，不会造成路径之间不一致。

### 3.2 叔叔为黑：旋转与变色后结束

- 若新节点与父、祖父呈“直线”（LL 或 RR），对祖父做一次反方向旋转，并交换父与祖父的颜色角色。
- 若呈“折线”（LR 或 RL），先对父节点旋转把折线变直，再按直线情况处理。

旋转保持 BST 中序顺序；父染黑、祖父染红则恢复黑高并消除红红冲突。这个分支完成后无需继续向上传播。

以父节点位于祖父左侧为例，右侧情况完全镜像。

**LL 直线：** 新节点 `X` 是父 `P` 的左孩子，叔叔 `U` 为黑。先把 `P` 染黑、`G` 染红，再对 `G` 右旋：

```text
        G(B)                         P(B)
       /    \                       /    \
    P(R)    U(B)      修复        X(R)   G(R)
    /                                      \
  X(R)                                     U(B)
```

旋转前，`X-P` 连续红；旋转后 `P` 成为局部根且为黑，两个红孩子 `X、G` 不再相邻。考虑黑高：原本所有路径都经过黑色 `G`；修复后所有路径都经过黑色 `P`，只是承担“这一层黑色”的节点从 `G` 换成 `P`，所以局部黑高不变。

**LR 折线：** `P` 是 `G` 的左孩子，但 `X` 是 `P` 的右孩子。先对 `P` 左旋，把 `X` 提升，使结构变成 LL；随后把“当前节点”更新为原父节点，再执行 LL 的染色和右旋：

```text
      G(B)                 G(B)                  X(B)
     /                    /                     /    \
   P(R)      左旋 P     X(R)      右旋 G      P(R)  G(R)
      \                 /
      X(R)             P(R)
```

RR 与 RL 分别是上述两种情况的左右镜像。实现时最容易出错的是折线预旋后仍使用旧的节点角色；标准循环会在预旋后重新赋值 `node/parent`，再进入直线修复。

标准插入修复至多进行两次旋转，但变色可能沿树向上多次发生，总时间仍为 `O(log n)`。

### 3.3 示例序列 `{41, 38, 31, 12, 19, 8}` 的完整修复

这也是后面三种语言示例使用的插入序列：

1. 插入 41：它成为根，最终染黑，得到 `41(B)`。
2. 插入 38：38 是 41 的红色左孩子，父 41 为黑，无需修复。
3. 插入 31：形成 `41(B) <- 38(R) <- 31(R)`，叔叔为黑色 `NIL`，属于 LL。把 38 染黑、41 染红并右旋 41，得到根 38，左右为 `31(R)、41(R)`。
4. 插入 12：12 是 31 的红孩子，此时父 31 与叔 41 都是红色。把 31、41 染黑，祖父 38 染红；38 是根，再强制染黑。
5. 插入 19：路径为 `38 -> 31 -> 12 -> 19`，父 12 红、叔为黑，形成 LR。先左旋 12，把 19 提升；再把 19 染黑、31 染红并右旋 31。此时 38 的左子树根是 19，孩子为 `12(R)、31(R)`。
6. 插入 8：8 成为 12 的红色左孩子。父 12 与叔 31 都红，于是二者染黑、祖父 19 染红。19 的父 38 是黑，修复结束。

最终结构是：

```text
          38(B)
         /     \
      19(R)    41(B)
      /   \
   12(B) 31(B)
   /
 8(R)
```

中序键为 `8,12,19,31,38,41`，任意根到 `NIL` 的路径黑高相同，也没有连续红节点。这个推演同时展示了叔叔黑 LL、叔叔红传播和叔叔黑 LR 三类修复。

## 4. 删除修复思想

删除先按 BST 规则找到要真正移走的节点。若移走红节点，黑高不变；若用红孩子替换黑节点，只需把孩子染黑。困难情况是移走黑节点且替代位置也是黑色 `NIL`，该位置可理解为暂时带有一个额外的“黑色”，称为**双黑**。

“双黑”通常不是节点真实保存的第三种颜色，而是推理状态：与其他兄弟路径相比，替代位置所在路径少了一个黑节点，因此把它视为携带一个待补偿的额外黑色。修复目标是通过旋转和变色把这份差额消除，或一路上移到根后自然消失。

若被删节点有两个孩子，BST 删除会选择后继或前驱，真正从树中移走的往往是替代节点。是否需要双黑修复，要看**真正移走节点的原颜色**，而不是最初搜索到的键所在节点颜色。

修复围绕双黑节点的兄弟展开：

1. **兄弟为红**：父必黑。旋转父节点并交换父兄颜色，把问题转换成“兄弟为黑”。
2. **兄弟为黑，两个侄子都黑**：兄弟染红，双黑上移到父节点；若父原来是红色，把父染黑即可结束。
3. **兄弟为黑，近侄子红、远侄子黑**：旋转兄弟并调整颜色，转化为下一种情况。
4. **兄弟为黑，远侄子红**：围绕父节点旋转，让兄弟继承父颜色，父与远侄子染黑，双黑消除。

“近/远”相对于双黑节点而言，左右情况互为镜像。删除代码长的根本原因不是比较逻辑，而是需要覆盖这些镜像颜色状态。实现时采用黑色 `NIL` 能显著减少空指针特判。

下面固定双黑位置 `X` 是父节点 `P` 的左孩子，因此兄弟 `S` 在右侧；若 `X` 在右侧，所有方向镜像。

### 4.1 兄弟为红：先转换形状

兄弟 `S` 为红时，根据红节点孩子必黑可知它的孩子都是黑，同时父 `P` 必黑。把 `S` 染黑、`P` 染红，再对 `P` 左旋。旋转不会直接消除双黑，但会让 `X` 获得一个黑色兄弟，从而转入后面三类标准情况。

这一转换保持黑高：旋转前路径先经过黑色 `P`；旋转后先经过黑色 `S`，承担黑层的节点改变，但路径黑节点总数不变。

### 4.2 黑兄弟且两个侄子都黑：把差额向上交给父

兄弟 `S` 染红，相当于让兄弟一侧路径少一个黑节点，于是它与 `X` 所在的缺黑路径在 `P` 以下重新相等。原来的差额被推到 `P`：

- 若 `P` 原为红，把 `P` 染黑就补回一个黑节点，修复结束；
- 若 `P` 原为黑，把 `P` 视作新的双黑节点，继续向祖父处理；
- 若 `P` 是根，额外黑色直接移除，所有根到叶路径仍同时满足相同黑高。

### 4.3 近侄子红、远侄子黑：转换为远侄子红

对左侧双黑而言，兄弟的左孩子是近侄子，右孩子是远侄子。把近侄子染黑、兄弟染红，再对兄弟右旋。新兄弟会拥有红色远侄子，问题转为最终情况。这个步骤类似插入修复中的“折线先拉直”。

### 4.4 远侄子红：一次旋转完成修复

让兄弟继承父的原颜色，把父染黑、远侄子染黑，然后对父向双黑方向的反方向旋转；本例即左旋 `P`。旋转后两侧路径都获得正确黑层，`X` 的额外黑色可以移除，修复结束。

颜色安排的含义是：兄弟继承父色，维持局部子树与更高祖先之间的黑高；父和远侄子染黑，使旋转后经过新根左右两侧的路径都包含同样数量的黑节点。

### 4.5 两个数值推演

以下未提到的孩子都是黑色 `NIL`，并固定被删位置在父节点左侧：

- **差额上移到根**：初始根为 `10(B)`，左孩子为 `5(B)`，右孩子为 `15(B)`。删除黑叶子 5 后，替代位置 `X=NIL` 变成双黑；兄弟 15 为黑，两个侄子也都黑，所以把 15 染红，差额上移到父 10。10 已是根，额外黑色直接移除。最终是黑根 10 与红色右孩子 15，根到左右 `NIL` 的黑高再次相同。
- **远侄子红，一次结束**：在上一棵初始树中再让 15 有红色右孩子 `20(R)`，此时仍是合法红黑树。删除 5 后，兄弟 15 为黑、远侄子 20 为红，直接进入 4.4：15 继承父色（仍为黑），10 和 20 染黑，再对 10 左旋。旋转后局部根为 `15(B)`，左孩子为 `10(B)`，右孩子为 `20(B)`，双黑消除且每条路径黑高相同。

本文后三种语言代码聚焦插入、查找、遍历和不变量验证，没有实现删除。删除讲解用于建立完整模型；补代码时应为每种情况和镜像情况分别构造测试，并在每次删除后检查 BST 上下界、父指针、根颜色、红红冲突和黑高。

## 5. 复杂度、适用场景与实现边界

| 操作 | 时间复杂度 | 额外空间（示例实现） |
| --- | --- | --- |
| 查找 | `O(log n)` | `O(1)` |
| 插入 | `O(log n)` | `O(1)` |
| 删除 | `O(log n)` | `O(1)` |
| 中序遍历 | `O(n)` | 递归栈 `O(log n)` |
| 单次旋转/变色 | `O(1)` | `O(1)` |

红黑树适合动态有序集合与映射，既要范围查询、前驱后继，又有较频繁的插入删除。Java `TreeMap/TreeSet`、C++ `map/set` 的典型实现就是红黑树（标准通常只规定行为和复杂度，不强制具体结构）。

对比：AVL 更严格，查找路径通常略短；红黑树更新调整较温和，综合读写负载更常见。哈希表等值查询平均更快，但不维护顺序。数据库磁盘索引仍通常选择 B/B+ 树。

查找只沿一条高度为 `O(log n)` 的路径。插入的 BST 下降与向上变色最多各走一个树高，旋转总数至多两次；删除搜索和双黑上移也最多走一个树高，单轮只做常数工作。因此三种核心操作都具有最坏 `O(log n)`。

红黑树适合需要稳定有序语义、范围查询、前驱后继，并且插入删除较频繁的内存结构。若负载几乎全是查找，AVL 的更严格高度可能更有利；若只做等值查询，哈希表通常更简单；若只反复取极值，堆的常数和实现更直接。

实现时尤其要守住以下边界：

- 旋转不仅要改左右孩子，还要更新孩子父指针、原祖父连接和根指针。
- 共享 `NIL` 必须始终保持黑色，真实节点的空孩子必须统一指向它。
- 插入重复键的策略要明确；示例选择忽略重复键。
- 验证不能只检查中序有序和“没有红红”，还必须从每个节点比较左右黑高。
- 节点带 `size、sum` 等增强字段时，旋转后必须同步重算，否则树颜色正确但增强查询错误。
- 多线程环境下，一次旋转会修改多个连接，不能把单个指针写入误认为完整操作是原子的。

```cpp
enum Color { RED, BLACK };

struct Node {

    Node* parent;

class RedBlackTree {
private:
    Node* nil;
    Node* root;

    void rotateLeft(Node* x) {
        Node* y = x->right;
        x->right = y->left;
        if (y->left != nil) y->left->parent = x;
        y->parent = x->parent;
        if (x->parent == nil) root = y;
        else if (x == x->parent->left) x->parent->left = y;
        else x->parent->right = y;
        y->left = x;
        x->parent = y;

        Node* x = y->left;
        y->left = x->right;
        if (x->right != nil) x->right->parent = y;
        x->parent = y->parent;
        if (y->parent == nil) root = x;
        else if (y == y->parent->left) y->parent->left = x;
        else y->parent->right = x;
        x->right = y;
        y->parent = x;
    }

    void fixInsert(Node* z) {
        while (z->parent->color == RED) {
            if (z->parent == z->parent->parent->left) {
                Node* uncle = z->parent->parent->right;
                if (uncle->color == RED) {
                    z->parent->color = BLACK;
                    uncle->color = BLACK;
                    z->parent->parent->color = RED;
                    z = z->parent->parent;
                } else {
                    if (z == z->parent->right) {
                        z = z->parent;
                        rotateLeft(z);
                    }
                    z->parent->color = BLACK;
                    z->parent->parent->color = RED;
                    rotateRight(z->parent->parent);
                }
            } else {
                Node* uncle = z->parent->parent->left;
                if (uncle->color == RED) {
                    z->parent->color = BLACK;
                    uncle->color = BLACK;
                    z->parent->parent->color = RED;
                    z = z->parent->parent;
                } else {
                    if (z == z->parent->left) {
                        z = z->parent;
                        rotateRight(z);
                    }
                    z->parent->color = BLACK;
                    z->parent->parent->color = RED;
                    rotateLeft(z->parent->parent);
                }
            }
        }
        root->color = BLACK;
    }

    void inorder(Node* node, vector<string>& out) const {
        if (node == nil) return;
        inorder(node->left, out);
        out.push_back(to_string(node->key) + (node->color == RED ? "(R)" : "(B)"));
        inorder(node->right, out);
    }

    int validate(Node* node, Node* expectedParent,
                 long long low, long long high, bool& ok) const {
        if (node == nil) return 1;
        if (node->parent != expectedParent) ok = false;
        if (node->key <= low || node->key >= high) ok = false;
        if (node->color == RED && (node->left->color == RED || node->right->color == RED)) ok = false;
        int leftBlackHeight = validate(node->left, node, low, node->key, ok);
        int rightBlackHeight = validate(node->right, node, node->key, high, ok);
        if (leftBlackHeight != rightBlackHeight) ok = false;
        return leftBlackHeight + (node->color == BLACK ? 1 : 0);
    }

    void destroy(Node* node) {
        if (node == nil) return;
        destroy(node->left);
        destroy(node->right);
        delete node;
    }

public:
    RedBlackTree() {
        nil = new Node(0, BLACK);
        nil->left = nil->right = nil->parent = nil;
        root = nil;
    }

    ~RedBlackTree() {
        destroy(root);
        delete nil;
    }

    void insert(int key) {
        Node* parent = nil;
        Node* current = root;
        while (current != nil) {
            parent = current;
            if (key < current->key) current = current->left;
            else if (key > current->key) current = current->right;
            else return; // 忽略重复键

        Node* z = new Node(key, RED);
        z->left = z->right = nil;
        z->parent = parent;
        if (parent == nil) root = z;
        else if (key < parent->key) parent->left = z;
        else parent->right = z;
        fixInsert(z);
    }

        Node* current = root;
        while (current != nil) {
            if (key == current->key) return true;
            current = key < current->key ? current->left : current->right;
        }
        return false;
    }

    vector<string> inorder() const {
        vector<string> out;
        inorder(root, out);
        return out;
    }

    bool isValid() const {
        if (root == nil) return true;
        bool ok = root->color == BLACK && nil->color == BLACK;
        validate(root, nil, numeric_limits<long long>::min(),
                 numeric_limits<long long>::max(), ok);
        return ok;
    }
};
```

## 8. Go 完整示例：插入、查找、遍历与验证

```go
package main

import "fmt"

type Color bool

const (
	red   Color = false
	black Color = true
)

type Node struct {
	Key                 int
	Color               Color
	Left, Right, Parent *Node
}

type RedBlackTree struct {
	root     *Node
	sentinel *Node
}

func newRedBlackTree() *RedBlackTree {
	s := &Node{Color: black}
	s.Left, s.Right, s.Parent = s, s, s
	return &RedBlackTree{root: s, sentinel: s}
}

func (t *RedBlackTree) rotateLeft(x *Node) {
	y := x.Right
	x.Right = y.Left
	if y.Left != t.sentinel {
		y.Left.Parent = x
	}
	y.Parent = x.Parent
	if x.Parent == t.sentinel {
		t.root = y
	} else if x == x.Parent.Left {
		x.Parent.Left = y
	} else {
		x.Parent.Right = y
	}
	y.Left = x
	x.Parent = y
}

func (t *RedBlackTree) rotateRight(y *Node) {
	x := y.Left
	y.Left = x.Right
	if x.Right != t.sentinel {
		x.Right.Parent = y
	}
	x.Parent = y.Parent
	if y.Parent == t.sentinel {
		t.root = x
	} else if y == y.Parent.Left {
		y.Parent.Left = x
	} else {
		y.Parent.Right = x
	}
	x.Right = y
	y.Parent = x
}

func (t *RedBlackTree) fixInsert(z *Node) {
	for z.Parent.Color == red {
		if z.Parent == z.Parent.Parent.Left {
			uncle := z.Parent.Parent.Right
			if uncle.Color == red {
				z.Parent.Color = black
				uncle.Color = black
				z.Parent.Parent.Color = red
				z = z.Parent.Parent
			} else {
				if z == z.Parent.Right {
					z = z.Parent
					t.rotateLeft(z)
				}
				z.Parent.Color = black
				z.Parent.Parent.Color = red
				t.rotateRight(z.Parent.Parent)
			}
		} else {
			uncle := z.Parent.Parent.Left
			if uncle.Color == red {
				z.Parent.Color = black
				uncle.Color = black
				z.Parent.Parent.Color = red
				z = z.Parent.Parent
			} else {
				if z == z.Parent.Left {
					z = z.Parent
					t.rotateRight(z)
				}
				z.Parent.Color = black
				z.Parent.Parent.Color = red
				t.rotateLeft(z.Parent.Parent)
			}
		}
	}
	t.root.Color = black
}

func (t *RedBlackTree) insert(key int) {
	parent := t.sentinel
	current := t.root
	for current != t.sentinel {
		parent = current
		if key < current.Key {
			current = current.Left
		} else if key > current.Key {
			current = current.Right
		} else {
			return
		}
	}
	z := &Node{Key: key, Color: red, Left: t.sentinel, Right: t.sentinel, Parent: parent}
	if parent == t.sentinel {
		t.root = z
	} else if key < parent.Key {
		parent.Left = z
	} else {
		parent.Right = z
	}
	t.fixInsert(z)
}

func (t *RedBlackTree) contains(key int) bool {
	current := t.root
	for current != t.sentinel {
		if key == current.Key {
			return true
		}
		if key < current.Key {
			current = current.Left
		} else {
			current = current.Right
		}
	}
	return false
}

func (t *RedBlackTree) inorder(node *Node, out *[]string) {
	if node == t.sentinel {
		return
	}
	t.inorder(node.Left, out)
	label := "B"
	if node.Color == red {
		label = "R"
	}
	*out = append(*out, fmt.Sprintf("%d(%s)", node.Key, label))
	t.inorder(node.Right, out)
}

func (t *RedBlackTree) values() []string {
	out := []string{}
	t.inorder(t.root, &out)
	return out
}

func (t *RedBlackTree) validate(node, expectedParent *Node, low, high *int64) (int, bool) {
	if node == t.sentinel {
		return 1, true
	}
	if node.Parent != expectedParent {
		return 0, false
	}
	key := int64(node.Key)
	if (low != nil && key <= *low) || (high != nil && key >= *high) {
		return 0, false
	}
	if node.Color == red && (node.Left.Color == red || node.Right.Color == red) {
		return 0, false
	}
	leftBH, leftOK := t.validate(node.Left, node, low, &key)
	rightBH, rightOK := t.validate(node.Right, node, &key, high)
	if !leftOK || !rightOK || leftBH != rightBH {
		return 0, false
	}
	if node.Color == black {
		leftBH++
	}
	return leftBH, true
}

func (t *RedBlackTree) isValid() bool {
	if t.root == t.sentinel {
		return true
	}
	if t.root.Color != black || t.sentinel.Color != black {
		return false
	}
	_, ok := t.validate(t.root, t.sentinel, nil, nil)
	return ok
}

func main() {
	tree := newRedBlackTree()
	for _, key := range []int{41, 38, 31, 12, 19, 8} {
		tree.insert(key)
	}
	fmt.Println("inorder:", tree.values())
	fmt.Println("contains 19:", tree.contains(19))
	fmt.Println("contains 99:", tree.contains(99))
	fmt.Println("valid red-black tree:", tree.isValid())
}
```

## 10. 易错点与要点总结

- 红黑树首先必须满足 BST 不变量，颜色正确不能弥补键顺序错误。
- 哨兵 `NIL` 必须始终为黑色；代码中不要把普通空指针与哨兵混用。
- 旋转要同时更新孩子、父节点、祖父连接以及根指针，漏一条都会破坏结构。
- 插入节点应为红色；若直接插入黑色，会立刻改变一部分路径的黑高。
- 叔叔为红时主要做变色并继续向上；叔叔为黑时通过旋转加变色局部结束。
- 左右分支完全镜像，复制代码后最容易遗漏某个 `left/right` 反转。
- 根最后必须强制染黑。
- 验证不能只看“没有连续红节点”，还必须比较所有路径黑高，并检查根与孩子的父指针是否正确回指。
- 删除修复中的近侄子、远侄子是相对双黑节点定义的，画出方向再写镜像代码。
