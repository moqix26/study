## 5. 本轮重点修复

- `store:false` 的 Responses Tool 循环会请求 `reasoning.encrypted_content`，累计原始输入、全部 `response.output` 与 Tool output；
- 不再把 `store:false` 与不可用的 `previous_response_id` 续接方式混用；
- 非流式与 SSE 会处理 refusal、failed、incomplete、顶层 error 和缺失终态；
- 无效 Tool arguments 不再生成不可序列化的 API trace；
- 会话按 tenant + user + conversation 隔离，同一会话整轮串行；
- 只有模型成功或 SSE 收到 `response.completed` 后才提交本轮历史；
- 增加请求体、Header、读取时间和 conversation ID 边界；
- 演示接口拒绝客户端指定 `X-Tenant-ID`；
- pgvector `vector` HNSW 维度限制为 1～2000，并在连接数据库前校验。

相关 OpenAI 行为按官方资料核对：

- [Conversation state](https://developers.openai.com/api/docs/guides/conversation-state)
- [Reasoning models](https://developers.openai.com/api/docs/guides/reasoning)
- [Function calling](https://developers.openai.com/api/docs/guides/function-calling)
- [Streaming Responses](https://developers.openai.com/api/docs/guides/streaming-responses)

## 6. 未执行与真实限制

- 当前机器没有 Docker，未启动 PostgreSQL/pgvector 容器；adapter 通过单测、SQL 审查与维度校验，但数据库集成仍需在有 Docker 或 PostgreSQL 的环境复验；
- 没有使用真实 OpenAI Key，真实账号、模型权限、额度、延迟和供应商错误需要单独集成验收；
- `X-User-ID` 仍是可伪造的演示身份，不是生产鉴权；
- 内存会话和内存 RAG 没有总容量、TTL、持久化与分布式限流；
- pgvector 尚无 RLS、异步 ingest、outbox、删除任务和 ANN tenant recall 调优；
- 日志、指标、trace、敏感信息脱敏与数据保留仍是 V3 工作。

因此，可以把当前状态描述为“V1 离线基线已验证，V2 adapter 已实现并完成本地单测”，不能描述成“真实模型和数据库生产部署已完成”。
