## 1. 大厂面试到底考什么（先看懂游戏规则）

一场典型大厂技术面（45～60 分钟）的组成：

| 环节 | 占比 | 考什么 | 失败常见原因 |
|------|------|--------|-------------|
| 自我介绍 + 项目 | 25% | 你做过什么、技术深度、表达 | 项目讲不清、被追问就崩 |
| 八股连环问 | 25% | 基础扎实度 + 深度 | 只会背不会推导，追问两层就卡 |
| 场景设计题 | 25% | 系统设计能力、权衡意识 | 上来就钻细节、没全局、不考虑扩展 |
| 手撕代码 | 20% | 编码基本功、边界处理 | 语法错、边界漏、写不完 |
| 反问环节 | 5% | 思维成熟度 | 问不出好问题 |

**关键认知**：
- **不是"全答对"就过**，是"每个环节都不崩 + 有亮点"就过。
- **追问是探底**——面试官一直问直到你答不出为止，答不出正常，但要"倒在有深度的地方而不是浅处"。
- **表达 > 答案**——讲得有条理、有取舍，比背出标准答案更受青睐。

> 这一章就是逐环节给你武器。

## 2. 场景设计题：4 步框架 + 高频题示范

### 2.1 通用 4 步框架（背下来，每题都套）

**第 1 步：澄清需求（3～5 分钟，别跳！）**

很多人上来就画架构，大忌。先问清楚：
- **功能需求**：核心功能是什么？（限流系统：限谁、限什么维度、限到什么程度）
- **非功能需求**：QPS 量级？延迟要求？一致性要求（可不可偶尔漏限）？
- **规模约束**：用户量？数据量？读多写多？
- **边界**：要不要支持分布式？要不要动态调规则？

> 问需求体现"工程师素养"。面试官期待你先问再设计。

**第 2 步：总体架构（5～10 分钟）**

画核心组件 + 数据流。用 mermaid 或口述：
- 客户端 → 网关 → 业务服务 → 存储
- 标出每个组件用什么（Redis？MySQL？MQ？）
- 讲数据怎么流、控制怎么流

**第 3 步：核心模块深挖（15～20 分钟）**

挑 1～2 个最核心模块讲实现：
- 用什么数据结构/算法（限流用令牌桶/滑动窗口、短链用发号器/Base62）
- 关键流程的伪代码或文字步骤
- 为什么这么选（对比其它方案的取舍）

**第 4 步：权衡与扩展（5～10 分钟，加分项）**

主动讲：
- **瓶颈在哪**：CPU？IO？网络？存储？
- **怎么扩展**：水平扩、分片、读写分离、缓存
- **CAP 取舍**：选 CP 还是 AP，为什么
- **容灾**：单点挂了怎么办、数据丢了怎么办

> 这 4 步是骨架。下面 5 道高频题示范怎么套。

### 2.2 高频题示范 1：设计一个限流系统

**第 1 步澄清**：限 API 还是限用户？维度（用户/IP/接口）？量级（万 QPS）？单机还是分布式？要不要动态调阈值？——确认：分布式、按用户+接口维度、10 万 QPS、可动态调。

**第 2 步架构**：网关层（Nginx/Spring Cloud Gateway）做粗粒度限流 → 业务层用 Redis 做细粒度分布式限流 → 规则存配置中心可热更新。

**第 3 步核心**：算法选型——
- 计数器：简单但有临界突刺。
- 滑动窗口：平滑但内存高。
- **令牌桶**：允许突发、平滑，**最常用**。Redis + Lua 原子取令牌（`CL.THROTTLE` 或自写 Lua）。
- 漏桶：恒速输出，不适合允许突发的场景。

伪代码（令牌桶 Lua）：
```lua
-- key=限流key, capacity=桶容量, rate=每秒补充数, now=当前毫秒
local last = tonumber(redis.call('hget', KEYS[1], 'last')) or now
local tokens = tonumber(redis.call('hget', KEYS[1], 'tokens')) or capacity
local delta = math.max(0, now - last) / 1000 * rate
tokens = math.min(capacity, tokens + delta)
if tokens < 1 then return 0 end
tokens = tokens - 1
redis.call('hmset', KEYS[1], 'last', now, 'tokens', tokens)
redis.call('expire', KEYS[1], 60)
return 1
```

**第 4 步权衡**：
- 瓶颈：Redis 单点 → 用 Cluster 分片或本地+中心两层（本地令牌桶预分配 + 中心定期校准）。
- 精度 vs 性能：用 Lua 保证原子，但每次都打 Redis 慢 → 本地缓存桶 + 异步同步。
- 降级：Redis 挂了，降级到本地单机限流（牺牲全局精度保可用）。

### 2.3 高频题示范 2：设计短链服务

**澄清**：长链转短链、点击跳转、要统计点击量、量级 1 亿短链、QPS 万级写、10 万级读。

**架构**：发号服务（生成短码）→ 存储（短码→长链映射）→ 跳转服务（302 重定向）→ 统计（异步写 MQ 聚合）。

**核心**：
- **短码生成**：自增 ID + Base62 编码（6 位够 568 亿）。或哈希（冲突要处理）。自增更短可控，发号器用 Redis incr 或雪花算法。
- **存储**：MySQL 存映射（短码 PK）+ Redis 缓存热点短码。
- **跳转**：先查 Redis，miss 查 DB，302（不要 301，301 浏览器永久缓存无法统计/改向）。

**权衡**：
- 发号器单点 → 雪花算法分片或多个发号器各管一段 ID 区间。
- 缓存穿透（不存在的短链）→ 布隆过滤器 + 空值缓存。
- 统计写多 → Kafka 异步聚合批量写，不阻塞跳转。

### 2.4 高频题示范 3：设计秒杀系统

**澄清**：瞬时高并发抢购、防超卖、防刷、库存有限。

**架构**：前端（按钮防重复+静态化）→ 网关限流 → 秒杀服务 → Redis 预减库存 → MQ 异步下单 → DB 最终扣减。

**核心**：
- **防超卖**：Redis Lua 原子预减库存（库存≤0 拒绝）+ DB 用 `update stock=stock-1 where id=? and stock>0` 兜底。
- **异步下单**：预减成功发 MQ，消费者慢慢落库，前端直接返回"排队中"。
- **防刷**：限流（每用户每秒 1 次）+ 验证码 + 风控。

**权衡**：
- Redis 挂了 → 降级直接拒（保不超卖）而非放行。
- MQ 丢消息 → 本地消息表 + 重试，最终一致。
- 库存预热 + 内存标记售罄（售罄后直接拒绝不打 Redis）。

### 2.5 高频题示范 4：设计一个 RAG 服务（Agent 岗高频）

**澄清**：文档量、QPS、延迟要求、准确率要求、是否要引用、是否多租户。

**架构**：直接套 [23 章](23-Agent与Java端到端项目实战.md) §3.2 架构。

**核心**：
- 检索：混合检索（BM25+向量+RRF）+ rerank。
- 生成：prompt 模板 + 引用编号 + 无依据拒绝回答。
- 评估：RAGAS 闭环。
- 成本：模型路由 + prompt 压缩 + 缓存。

**权衡**：见 [23 章 §4 选型](23-Agent与Java端到端项目实战.md)。这道题你有现成项目，**直接讲你的 23 章项目就是满分答案**。

### 2.6 高频题示范 5：设计消息推送/IM 系统

**澄清**：一对一/群/在线离线？量级？消息要不要保序？已读回执？

**架构**：客户端长连接（WebSocket/SSE）→ 接入层（连接管理）→ 路由层（找用户在哪台接入机）→ 消息存储 → 推送。

**核心**：
- **连接管理**：单机连接数上限 → 水平扩接入层，用户→接入机映射存 Redis。
- **消息保序**：会话级单调递增 seq。
- **离线消息**：存 DB/Redis，上线后拉取。
- **群消息扩散**：写扩散（写每个成员收件箱，小群）vs 读扩散（写群时间线，大群），按群规模选。

**权衡**：写扩散对大群存储爆炸，读扩散对活跃用户拉取重——按群规模混合策略。

> 场景设计题**框架比答案重要**。练 5 道就能套所有题。

## 3. 高频手撕代码（练到肌肉记忆）

> 大厂手撕常考这 6 类。每题给正确实现 + 易错点 + 面试讲法。**练法**：不看答案默写 3 遍直到一次写对。

### 3.1 LRU 缓存（最高频）

```java
class LRUCache {
    private static class Node {
        int key, val; Node prev, next;
        Node(int k, int v) { key = k; val = v; }
    }
    private final int capacity;
    private final Map<Integer, Node> map = new HashMap<>();
    private final Node head = new Node(0,0), tail = new Node(0,0);

    public LRUCache(int capacity) {
        this.capacity = capacity;
        head.next = tail; tail.prev = head;
    }

    public int get(int key) {
        if (!map.containsKey(key)) return -1;
        Node node = map.get(key);
        moveToHead(node);
        return node.val;
    }

    public void put(int key, int value) {
        if (map.containsKey(key)) {
            Node node = map.get(key);
            node.val = value;
            moveToHead(node);
        } else {
            Node node = new Node(key, value);
            map.put(key, node);
            addToHead(node);
            if (map.size() > capacity) {
                Node tailNode = tail.prev;
                removeNode(tailNode);
                map.remove(tailNode.key);
            }
        }
    }

    private void addToHead(Node n) {
        n.next = head.next; n.prev = head;
        head.next.prev = n; head.next = n;
    }
    private void removeNode(Node n) {
        n.prev.next = n.next; n.next.prev = n.prev;
    }
    private void moveToHead(Node n) { removeNode(n); addToHead(n); }
}
```

**易错点**：双向链表指针操作错（忘改 prev）；忘了 remove 时要从 map 删 key。
**面试讲法**：HashMap + 双向链表，HashMap 提供 O(1) 查找，双向链表维护访问顺序，访问/写入移到表头，超容量删表尾。 LinkedHashMap 重写 removeEldestEntry 一行也行，但面试官想看手写。

### 3.2 DCL 单例（双重检查锁）

```java
class Singleton {
    private static volatile Singleton instance;  // volatile 关键！
    private Singleton() {}
    public static Singleton getInstance() {
        if (instance == null) {                   // 第一次检查，无锁快路径
            synchronized (Singleton.class) {
                if (instance == null) {           // 第二次检查，防并发重复创建
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

**易错点**：忘 `volatile`。
**为什么 volatile 必需**：`new Singleton()` 不是原子操作，分三步：分配内存→初始化→赋引用。JVM 可能重排成"分配→赋引用→初始化"，另一个线程在赋引用后、初始化前拿到非 null 但未初始化的对象，崩。volatile 禁止重排。**这是面试必问的追问，答不出 volatile 就漏了点**。呼应 [Java/03](../Java/03-Java并发编程与JVM.md) 锁升级/JMM。

### 3.3 生产者-消费者（BlockingQueue 版）

```java
class ProducerConsumer {
    private final BlockingQueue<Integer> queue = new LinkedBlockingQueue<>(10);

    public void producer() throws InterruptedException {
        int i = 0;
        while (true) queue.put(i++);        // 满了自动阻塞
    }
    public void consumer() throws InterruptedException {
        while (true) System.out.println(queue.take());  // 空了自动阻塞
    }
}
```

**面试讲法**：BlockingQueue 的 put/take 内部用 Condition（两个等待队列：notFull、notEmpty），满时生产者挂 notFull、空时消费者挂 notEmpty，自动唤醒。也可手写 synchronized + wait/notify，但 BlockingQueue 简洁且原子。**追问**：为什么用两个 Condition 而不是一个？——一个 Condition 会惊群，两个精确唤醒对应方。

### 3.4 令牌桶限流（单机版）

```java
class TokenBucket {
    private final long capacity;       // 桶容量
    private final double rate;         // 每秒补充令牌数
    private double tokens;             // 当前令牌
    private long lastRefillNanos;      // 上次补充时间

    public TokenBucket(long capacity, double rate) {
        this.capacity = capacity; this.rate = rate;
        this.tokens = capacity; this.lastRefillNanos = System.nanoTime();
    }

    public synchronized boolean tryAcquire() {
        long now = System.nanoTime();
        double add = (now - lastRefillNanos) / 1e9 * rate;
        tokens = Math.min(capacity, tokens + add);
        lastRefillNanos = now;
        if (tokens >= 1) { tokens -= 1; return true; }
        return false;
    }
}
```

**易错点**：忘按时间差补充令牌（应该是懒补充：取时算过了多久补多少，而不是定时加）。
**面试讲法**：懒补充令牌（取时按时间差补），允许突发到桶容量，平滑限流。分布式版换 Redis + Lua（见 §2.2）。

### 3.5 快速幂（处理溢出 + 二分）

```java
// 求 base^exp % mod，O(log n)
long quickPow(long base, long exp, long mod) {
    long result = 1; base %= mod;
    while (exp > 0) {
        if ((exp & 1) == 1) result = result * base % mod;  // 指数为奇数累乘
        base = base * base % mod;                          // 底数平方
        exp >>= 1;                                         // 指数减半
    }
    return result;
}
```

**易错点**：每步 % mod 防溢出（long 也会爆）；exp 用位运算判断奇偶更快。
**面试讲法**：把 O(n) 暴力乘降到 O(log n)，分治思想。处理大数取模防溢出是边界考点。

### 3.6 简化线程池（手写考察原理）

```java
class SimpleThreadPool {
    private final BlockingQueue<Runnable> queue;
    private final List<Worker> workers;
    SimpleThreadPool(int poolSize, int queueSize) {
        queue = new LinkedBlockingQueue<>(queueSize);
        workers = new ArrayList<>();
        for (int i = 0; i < poolSize; i++) {
            Worker w = new Worker(); workers.add(w); w.start();
        }
    }
    void execute(Runnable task) { queue.offer(task); }  // 简化：满时丢弃，生产用 put 阻塞或拒绝策略
    private class Worker extends Thread {
        public void run() {
            while (true) {
                try { queue.take().run(); }
                catch (InterruptedException e) { break; }
            }
        }
    }
}
```

**面试讲法**：BlockingQueue + 固定 Worker 线程死循环 take 任务执行。这其实就是 JDK ThreadPoolExecutor 的简化核心——真实版还多了 core/max 参数、拒绝策略、空闲超时回收。**追问**：和 JDK 线程池执行流程比差什么？→ 见 [Java/03 面试深挖](../Java/03-Java并发编程与JVM.md)：core→queue→max→拒绝，我这个简化版没区分 core/max。

> 手撕要练到：拿到题不用想语法，边讲边写。每个题的"易错点"和"讲法"是面试加分关键。

## 4. 八股总览速查（从全库抽，面试前 1 小时过）

> 把全库 200+ 篇里的高频考点聚成速查表。每条只列"核心结论"，深挖回去看对应章。**这是面试前最后一遍的过题单**。

### 4.1 Java 并发与 JVM（[Java/03](../Java/03-Java并发编程与JVM.md)）

- synchronized 锁升级：偏向→轻量→重量（JDK15 废弃偏向）
- AQS：state + CLH 队列 + 模板方法，ReentrantLock/Semaphore/CountDownLatch 底座
- 线程池执行顺序：core→queue→max→拒绝（不是 core→max→queue）
- CAS 的 ABA：AtomicStampedReference 加版本号解决
- ConcurrentHashMap：JDK7 分段锁 vs JDK8 桶级 CAS+synchronized+红黑树
- G1（Region+Garbage First）/ ZGC（染色指针+读屏障 <10ms）
- 双亲委派 + 打破：SPI 上下文类加载器 / Tomcat / OSGi
- volatile：可见性 + 禁重排（DCL 单例必需）
- happens-before：前操作对后操作可见的规则

### 4.2 MySQL（[Java/06](../Java/06-MySQL基础索引与事务.md)）

- MVCC：隐藏字段 + undo 版本链 + ReadView，RC 每次/RR 首次生成 ReadView
- 索引下推 ICP：5.6+，引擎层先用联合索引剩余列过滤再回表
- 间隙锁/临键锁：RR 专属，等值命中唯一索引退化记录锁、未命中退化间隙锁
- B+ 树：叶子链表、范围查询友好、3～4 层撑千万级
- 聚簇 vs 二级索引、覆盖索引、回表、最左前缀
- change buffer（仅非唯一索引）/ AHI / MRR
- explain 看 type/key/Extra（Using index/where/index condition）

### 4.3 Redis（[Java/07](../Java/07-Redis核心原理与缓存实战.md)）

- 5 种结构底层：SDS / listpack（7.0 替代 ziplist）/ quicklist / skiplist+dict / intset
- 单线程 + 6.0 多线程 IO（命令执行仍单线程）
- 过期删除（惰性+定期）+ 内存淘汰（8 种，近似 LRU/LFU）
- 哨兵（主观/客观下线+Raft）vs Cluster（16384 槽+CRC16+Gossip）
- RedLock 争议 + Redisson 看门狗（不传 leaseTime 启用，每 1/3 TTL 续期）
- AOF 重写 + 混合持久化（5.0 默认，RDB 段+AOF 段）
- 缓存三大问题：穿透（布隆）/ 击穿（互斥重建）/ 雪崩（TTL 随机+多级）
- 分布式锁：SETNX→Redisson→RedLock 逐步演进

### 4.4 Spring / 框架（[Java/04](../Java/04-SpringBoot核心开发.md)、[Java/05](../Java/05-MyBatis事务与接口工程化.md)）

- IoC/DI、AOP（动态代理 JDK vs CGLIB）
- Bean 生命周期、循环依赖（三级缓存）
- 事务传播：REQUIRED/REQUIRES_NEW/NESTED 区别
- @Transactional 失效：自调用、非 public、异常被吞

### 4.5 Linux（[Linux/06](../Linux/06-进程与服务管理.md)、[Linux/07](../Linux/07-网络命令与防火墙基础.md)）

- 进程状态 R/S/D/Z/T，D 状态 kill-9 无效
- 僵尸进程（父没 wait）/ 孤儿进程（init 收养无害）
- OOM Killer（oom_score，区别 JVM 堆 OOM）
- systemd unit 编写、daemon-reload、SuccessExitStatus=143
- TCP 状态、TIME_WAIT 堆积（长连接/tcp_tw_reuse）、CLOSE_WAIT 堆积（应用 bug）
- ss vs netstat、tcpdump 抓三次握手

### 4.6 AI/Agent（[AIAgent 01-22](00-学习路线图与说明.md)、[Java/03 深挖](../Java/03-Java并发编程与JVM.md)）

- RAG 全流程 + 混合检索 + rerank + RAGAS 评估
- Function Calling 原理 + 安全
- 多智能体模式（Router/Sequence/Parallel/Supervisor）
- LLM 原理：Transformer/Attention/Decoder-only、Pretrain→SFT→DPO/RLHF
- Prompt 进阶：few-shot/CoT/self-consistency、结构化输出
- 成本延迟优化：模型路由/prompt 缓存/KV Cache
- 微调：LoRA/QLoRA、SFT/DPO/GRPO
- MCP/A2A 协议、vLLM 本地推理、量化
- 可观测：Langfuse trace + RAGAS 评估闭环

## 5. 项目深挖应对模板（对接 23 章项目）

### 5.1 讲项目的 3 分钟结构

1. **背景（30 秒）**：什么场景、解决什么问题、为什么需要 AI。
2. **架构（60 秒）**：整体数据流 + 核心组件（用 23 章 §3.3 那段）。
3. **亮点 + 量化（60 秒）**：1～2 个技术亮点 + 量化结果（准确率 0.72→0.89、成本降 40%）。
4. **难点（30 秒）**：最难的是什么、怎么解决（23 章 §9.4 那条"准确率和成本平衡"）。

> 别流水账式列技术栈。架构 + 亮点 + 量化 + 难点，4 段式最有效。

### 5.2 通用追问预案模板

对项目里**每个**技术点，准备 3 张牌：

| 牌 | 内容 | 举例（PGVector） |
|----|------|-----------------|
| **为什么用它** | 选型理由 + 对比 | 百万级向量够用、复用 PG、关系向量同库同事务 |
| **什么情况换** | 边界条件 | 十亿级或纯向量高并发换 Milvus |
| **踩过什么坑** | 真实问题 | HNSW 索引参数 ef_search 调小召回率掉 |

> 23 章 §9 已经给 RAG/Agent/工程/项目深挖四类追问答案。面试前把项目每个技术点都填这 3 张牌。

### 5.3 被问"没做过"的怎么办

- 实事求是："这个我没在生产做过，但我的理解是……（讲思路）。"比硬编强。
- 转到相关："没做过 X，但我做过类似的 Y，原理是……"。
- 反问探边界："您指的是 A 场景还是 B 场景？"——有时面试官只是泛问，确认后好答。

> 编造是面试大忌。一旦被发现一处编的，整场可信度崩。

## 9. FAQ（8 问）

**Q1：没有大厂实习/项目，靠这个能进吗？**
A：能，但项目要扎实。23 章项目跑通 + 能扛追问 + 手撕过硬 + 八股扎实，校招/转行完全有机会。社招更看真实经历，这个项目是"加分项 + 证明能力"。

**Q2：八股背多少够？**
A：§4 速查表覆盖 80%。剩下的在面试中被问到能讲思路即可，不必全会。深度 > 广度：3 个讲透的 > 10 个讲不清的。

**Q3：场景设计题没做过类似系统怎么办？**
A：场景设计考的是"思路 + 框架"，不是"做过"。套 4 步框架，每步讲清取舍，没做过也能答好。平时多看 §2 的 5 道题理解套路。

**Q4：手撕写不完怎么办？**
A：先写主逻辑（别先纠结边界），跑通再补边界。面试官看的是"思路 + 主干"，写完 80% 比卡在边界上没写完强。边写边讲思路。

**Q5：被追问到答不出怎么办？**
A：别说"不知道"就停。说"这个我不确定，但我的推测是……因为……"——展示推理过程。倒在有深度的地方比倒在浅处好。

**Q6：Agent 岗和纯 Java 岗面试重点差什么？**
A：Agent 岗除 Java 八股外，必考 RAG/Function Calling/多智能体/LLM 原理/Prompt/成本/微调/MCP（23/24 章和 AIAgent 13-22）。项目轮重点考 Agent 架构和选型。

**Q7：面试官问"还有什么要问的"问什么？**
A：问团队技术栈/方向、业务挑战、新人成长机制、最近的技术决策。别问薪资福利（HR 轮再问）、别问"加班多吗"。

**Q8：这章和 23 章先学哪个？**
A：先做 23 章项目（有项目才能讲），再学这章（讲法/框架/手撕）。但手撕代码可以和做项目并行练。

## 10. 闭卷自测（10 题）

**概念题（6）**
1. 场景设计题的 4 步框架是什么？第 1 步为什么不能跳？
2. DCL 单例为什么必须 volatile？不 volatile 会出什么问题？
3. LRU 的数据结构选型？为什么双向链表不能换单链表？
4. 限流用令牌桶 vs 漏桶 vs 滑动窗口，各自适合什么场景？
5. 讲项目 3 分钟的 4 段式结构是什么？
6. TIME_WAIT 堆积和 CLOSE_WAIT 堆积分别说明什么问题？

**动手题（2）**
7. 默写 LRU 的 get/put 核心方法（不看答案）。
8. 默写令牌桶限流的 tryAcquire（懒补充版）。

**综合题（2）**
9. 面试官问"设计一个支持 10 万 QPS 的限流系统"，用 4 步框架口述你的回答。
10. 你的 23 章项目，准备"最难的地方是什么"的 1 分钟答案。

<details>
<summary>自测参考答案</summary>

1. 澄清需求→总体架构→核心模块深挖→权衡扩展。第 1 步不能跳是因为不澄清需求就设计会跑偏，且体现工程师素养。
2. new 对象非原子（分配→初始化→赋引用可重排成分配→赋引用→初始化），无 volatile 另一线程可能拿到未初始化对象。volatile 禁重排。
3. HashMap + 双向链表。双向是因为移动节点要 O(1) 拿到前驱改指针，单链表拿前驱要 O(n)。
4. 令牌桶允许突发（桶存令牌）；漏桶恒速输出不允许突发；滑动窗口平滑无临界突刺但内存高。多数场景用令牌桶。
5. 背景→架构→亮点+量化→难点。
6. TIME_WAIT 堆积=短连接多（协议正常产物，调参缓解）；CLOSE_WAIT 堆积=应用没 close 连接（必是 bug，改代码）。
7. 见 §3.1。
8. 见 §3.4。
9. 澄清：分布式、按用户+接口、10万QPS、可动态调。架构：网关粗限→Redis 细限→配置中心热更新。核心：令牌桶 + Redis Lua 原子取令牌 + 本地预分配减 Redis 压力。权衡：Redis Cluster 分片、降级本地单机限流、限流精度 vs 性能取舍。
10. 见 23 章 §9.4：准确率和成本平衡，混合检索+模型路由+RAGAS 评估闭环，0.72→0.89、成本降 40%。
