# 02 Go 模型调用与结构化输出
> 适用环境：Go 1.26，Responses API，net/http 与 Gin。
>
> 本章重点：让模型输出可解析的数据，但仍由 Go 服务完成最终校验与业务决策。
## 本章目标

学完本章，你应当能够：
1. 区分普通 JSON 提示、JSON mode 与 strict JSON Schema。
2. 为一个真实业务响应设计简洁、封闭、可验证的 Schema。
3. 使用 Responses API 的 text.format 请求结构化输出。
4. 将 output_text 严格解码为 Go 结构体。
5. 同时处理结构校验、语义校验、拒绝和输出不完整。
6. 用契约测试防止 Go 类型与 JSON Schema 静默漂移。
## 1. 为什么需要结构化输出

普通文本适合直接展示，但不适合稳定驱动程序。
例如，工单分类模型自由回答：
~~~text
这似乎是支付问题，优先级应该比较高，也许需要人工处理。
~~~
人能看懂，Go 程序却很难可靠提取：
- 分类到底是 payment 还是 other；
- 优先级是 high 还是 urgent；
- 是否需要人工；
- 模型是否遗漏字段；
- 模型是否附加了无法解析的解释。
我们希望得到：
~~~json
{
  "category": "payment",
  "priority": "high",
  "needs_human": true,
  "summary": "用户付款成功但订单仍显示未支付"
}
~~~
这仍然只是“模型判断”，但至少形成了可校验接口。
## 2. 三种常见做法的差异

### 2.1 只在提示词中要求 JSON
~~~text
请只返回 JSON，不要输出其他内容。
~~~
问题：
- 可能带 Markdown 代码围栏；
- 字段可能改名；
- 枚举可能写成近义词；
- 类型可能漂移；
- 某些情况下仍会输出自然语言。
它适合快速试验，不适合作为可靠协议。
### 2.2 JSON mode
部分模型或供应商提供“保证输出是合法 JSON”的模式。
它通常只保证：
- 输出能被 JSON 解析。
它通常不保证：
- 必须有哪些字段；
- 字段类型；
- 枚举范围；
- 是否允许额外字段；
- 嵌套对象的具体结构。
### 2.3 strict JSON Schema
Structured Outputs 使用 JSON Schema 描述期望结构，并通过 strict 模式约束输出。
它解决的是：
- 字段名称；
- 数据类型；
- 必填字段；
- 枚举值；
- 对象是否允许额外属性；
- 数组元素结构。
但它仍不解决：
- 事实真实性；
- 业务权限；
- 数据是否过期；
- 金额是否在业务范围内；
- 模型选择的分类是否符合你的内部政策。
一句话：
> Schema 负责结构，Go 业务代码负责语义和权限。
## 3. Responses API 中的结构化输出

请求核心形态：
~~~json
{
  "model": "YOUR_MODEL",
  "instructions": "根据用户描述分类工单。",
  "input": "我已经付款，但订单仍显示未支付。",
  "text": {
    "format": {
      "type": "json_schema",
      "name": "ticket_classification",
      "strict": true,
      "schema": {
        "type": "object",
        "properties": {
          "category": {
            "type": "string",
            "enum": ["payment", "delivery", "account", "other"]
          },
          "priority": {
            "type": "string",
            "enum": ["low", "medium", "high"]
          },
          "needs_human": {"type": "boolean"},
          "summary": {"type": "string"}
        },
        "required": [
          "category",
          "priority",
          "needs_human",
          "summary"
        ],
        "additionalProperties": false
      }
    }
  }
}
~~~
### 3.1 name 的意义
name 是结构化输出格式的稳定标识。
建议：
- 使用小写英文、数字和下划线；
- 表达业务含义；
- Schema 不兼容变更时升级名称，例如 ticket_classification_v2；
- 在日志中记录格式名和版本。
### 3.2 strict 的意义
strict 表示要求模型遵循所给 Schema。
常见注意点：
- 对象通常应设置 additionalProperties: false；
- 字段应明确列入 required；
- 可选概念可用包含 null 的联合类型表达；
- 不要假定所有 JSON Schema 关键字都被所有模型支持；
- Schema 太深、枚举太大时，应重新审视接口设计。
具体支持范围必须以当前官方文档和所选模型为准。
### 3.3 可选字段怎么设计
在 strict 结构中，与其直接省略字段，不如显式允许 null：
~~~json
{
  "type": "object",
  "properties": {
    "tracking_number": {
      "type": ["string", "null"]
    }
  },
  "required": ["tracking_number"],
  "additionalProperties": false
}
~~~
对应 Go：
~~~go
type DeliveryResult struct {
	TrackingNumber *string `json:"tracking_number"`
}
~~~
这样可以区分：
- 有字符串；
- 明确没有值；
- 输出结构违规。
## 4. Schema 设计原则

### 4.1 只返回程序真正需要的字段
坏设计：
~~~json
{
  "analysis": "一大段思维过程",
  "all_possible_categories": [],
  "confidence_explanation": "...",
  "internal_reasoning": "..."
}
~~~
更好的设计：
~~~json
{
  "category": "payment",
  "priority": "high",
  "needs_human": true,
  "summary": "..."
}
~~~
不要用结构化输出索取或暴露原始 Chain-of-Thought。
如果需要可解释性，返回简短、面向用户或审核员的 reasons：
~~~json
{
  "decision": "manual_review",
  "reasons": [
    "用户报告重复扣款",
    "当前信息不足以确认退款状态"
  ]
}
~~~
### 4.2 优先枚举，不让下游猜近义词
如果程序只接受三个优先级，就定义：
~~~json
{"type":"string","enum":["low","medium","high"]}
~~~
不要让模型自由生成“紧急”“很高”“P1”或“important”。
显示给用户时，由 Go 代码完成本地化映射。
### 4.3 金额与时间谨慎设计
金额不要让模型输出 float64 后直接记账。
可选设计：
~~~json
{
  "currency": "CNY",
  "amount_minor": 1990
}
~~~
但服务端仍要检查：
- currency 是否在允许列表；
- amount_minor 是否非负；
- 是否超过业务上限；
- 金额是否来自可信订单系统。
时间建议使用明确格式，并由 Go 二次解析：
~~~json
{"occurred_at":"2026-07-15T10:30:00+08:00"}
~~~
解析时使用 time.Parse(time.RFC3339, value)。
### 4.4 不要把超大动态列表塞进枚举
数万个商品、城市或客户 ID 不适合作为 Schema enum。
更合适的是：
- 让模型提取查询条件；
- 服务端查数据库；
- 再通过 Tool Calling 返回候选；
- 或让用户从受控列表确认。

## 6. 手把手实验：工单分类

### 6.1 定义业务类型
internal/llm/structured.go：
~~~go
package llm
type TicketClassification struct {
	Category   string `json:"category"`
	Priority   string `json:"priority"`
	NeedsHuman bool   `json:"needs_human"`
	Summary    string `json:"summary"`
}
func (v TicketClassification) Validate() error {
	switch v.Category {
	case "payment", "delivery", "account", "other":
	default:
		return fmt.Errorf("不支持的 category: %q", v.Category)
	}
	switch v.Priority {
	case "low", "medium", "high":
	default:
		return fmt.Errorf("不支持的 priority: %q", v.Priority)
	}
	if strings.TrimSpace(v.Summary) == "" {
		return errors.New("summary 不能为空")
	}
	if utf8.RuneCountInString(v.Summary) > 120 {
		return errors.New("summary 不能超过 120 个字符")
	}
	return nil
}
~~~
上面需要导入 errors、fmt、strings 和 unicode/utf8。
这里仍然做 Validate，原因是：
- 兼容供应商可能没有真正执行 strict；
- 代码和 Schema 可能发生漂移；
- 长度等业务规则不应只依赖模型；
- 防御式校验能给出更清晰的本地错误。

### 6.3 定义可测试客户端方法
internal/llm/structured.go：
~~~go
type responseEnvelope struct {
	ID     string       `json:"id"`
	Status string       `json:"status"`
	Output []outputItem `json:"output"`
}
type outputItem struct {
	Type    string        `json:"type"`
	Content []contentItem `json:"content"`
}
type contentItem struct {
	Type    string `json:"type"`
	Text    string `json:"text"`
	Refusal string `json:"refusal"`
}
func (c *Client) CreateStructured(
	ctx context.Context,
	instructions string,
	input string,
	formatName string,
	schema json.RawMessage,
	dst any,
) (string, error) {
	var schemaValue any
	if err := json.Unmarshal(schema, &schemaValue); err != nil {
		return "", fmt.Errorf("非法 schema: %w", err)
	}
	requestBody := map[string]any{
		"model":        c.model,
		"instructions": instructions,
		"input":        input,
		"text": map[string]any{
			"format": map[string]any{
				"type":   "json_schema",
				"name":   formatName,
				"strict": true,
				"schema": schemaValue,
			},
		},
	}
	raw, err := json.Marshal(requestBody)
	if err != nil {
		return "", fmt.Errorf("编码请求: %w", err)
	}
	req, err := http.NewRequestWithContext(
		ctx,
		http.MethodPost,
		c.baseURL+"/responses",
		bytes.NewReader(raw),
	)
	if err != nil {
		return "", fmt.Errorf("创建请求: %w", err)
	}
	req.Header.Set("Authorization", "Bearer "+c.apiKey)
	req.Header.Set("Content-Type", "application/json")
	resp, err := c.http.Do(req)
	if err != nil {
		return "", fmt.Errorf("调用模型: %w", err)
	}
	defer resp.Body.Close()
	respRaw, err := io.ReadAll(io.LimitReader(resp.Body, 4<<20))
	if err != nil {
		return "", fmt.Errorf("读取模型响应: %w", err)
	}
	if resp.StatusCode < 200 || resp.StatusCode >= 300 {
		return "", fmt.Errorf(
			"模型状态=%d request_id=%q body=%s",
			resp.StatusCode,
			resp.Header.Get("x-request-id"),
			truncate(string(respRaw), 500),
		)
	}
	var envelope responseEnvelope
	if err := json.Unmarshal(respRaw, &envelope); err != nil {
		return "", fmt.Errorf("解析响应对象: %w", err)
	}
	output, refusal, err := findStructuredText(envelope)
	if err != nil {
		return envelope.ID, err
	}
	if refusal != "" {
		return envelope.ID, fmt.Errorf("模型拒绝: %s", refusal)
	}
	decoder := json.NewDecoder(strings.NewReader(output))
	decoder.DisallowUnknownFields()
	if err := decoder.Decode(dst); err != nil {
		return envelope.ID, fmt.Errorf("解码结构化输出: %w", err)
	}
	if decoder.Decode(&struct{}{}) != io.EOF {
		return envelope.ID, errors.New("结构化输出包含尾随内容")
	}
	return envelope.ID, nil
}
~~~
辅助函数：
~~~go
func findStructuredText(resp responseEnvelope) (text string, refusal string, err error) {
	for _, item := range resp.Output {
		if item.Type != "message" {
			continue
		}
		for _, content := range item.Content {
			switch content.Type {
			case "output_text":
				if content.Text != "" {
					return content.Text, "", nil
				}
			case "refusal":
				return "", content.Refusal, nil
			}
		}
	}
	if resp.Status != "" && resp.Status != "completed" {
		return "", "", fmt.Errorf("响应未完成，status=%q", resp.Status)
	}
	return "", "", errors.New("没有结构化 output_text")
}
~~~
### 6.4 组合业务调用
~~~go
func (c *Client) ClassifyTicket(
	ctx context.Context,
	userText string,
) (TicketClassification, error) {
	userText = strings.TrimSpace(userText)
	if userText == "" {
		return TicketClassification{}, errors.New("工单内容不能为空")
	}
	if utf8.RuneCountInString(userText) > 2000 {
		return TicketClassification{}, errors.New("工单内容过长")
	}
	var result TicketClassification
	_, err := c.CreateStructured(
		ctx,
		"你负责工单初步分类。只依据用户输入，不编造订单事实。",
		userText,
		"ticket_classification_v1",
		TicketClassificationSchema,
		&result,
	)
	if err != nil {
		return TicketClassification{}, err
	}
	if err := result.Validate(); err != nil {
		return TicketClassification{}, fmt.Errorf("语义校验: %w", err)
	}
	return result, nil
}
~~~

## 7. 结构校验之后还要做什么

### 7.1 业务语义校验
即使 Schema 合法，也可能出现：
~~~json
{
  "category": "payment",
  "priority": "high",
  "needs_human": false,
  "summary": "用户要求立即退回一百万元"
}
~~~
程序仍要根据内部规则判断：
- 高风险支付问题是否强制人工；
- 文本是否包含敏感信息；
- 用户是否有权查看关联订单；
- 是否需要调用订单工具确认事实。
### 7.2 处理拒绝
模型可能因为安全策略拒绝处理输入。
服务端应：
1. 将 refusal 作为独立分支处理；
2. 对用户返回合适说明；
3. 不把 refusal 当成 JSON 继续解码；
4. 记录脱敏指标；
5. 不通过反复改写提示词绕过安全策略。
### 7.3 处理不完整输出
常见原因：
- 输出 token 上限；
- 内容过滤；
- 网络中断；
- 上游内部错误。
不要把半段 JSON 自动补括号后当成功。
正确策略：
- 检查响应状态和不完整原因；
- 对安全可重试的场景进行有界重试；
- 仍失败则返回稳定业务错误；
- 保留 trace_id 便于定位。
## 8. Schema 与 Go 类型如何防漂移

手写 Go 类型与 Schema 可能在字段、tag、枚举或类型上漂移。可以单向生成，也可以像本课程前期一样手写并用契约测试锁定；项目稳定后再评估生成工具。
至少准备：
~~~text
testdata/
├─ ticket_classification.valid.json
├─ ticket_classification.extra_field.json
├─ ticket_classification.bad_enum.json
└─ ticket_classification.missing_field.json
~~~
测试目标：valid 通过；extra_field 被 DisallowUnknownFields 拒绝；bad_enum 被 Validate 拒绝；missing_field 被额外检查发现。
注意：Go 的零值会让缺失字段不一定在 Unmarshal 阶段报错。
必须区分缺失与零值时，可使用指针字段、JSON Schema 校验器、自定义 UnmarshalJSON 或自定义值类型。
## 9. 供应商能力核验

切换 base URL 前，为结构化输出单独验证：
- 是否支持 Responses API；
- text.format 的字段层级是否相同；
- 是否支持 type: json_schema；
- strict 是否真的拒绝额外字段；
- 支持哪些 JSON Schema 关键字；
- refusal 如何表示；
- 输出不完整如何表示；
- 流式结构化输出有哪些事件；
- 模型是否明确支持 Structured Outputs。
如果供应商只支持 JSON mode：
1. 明确标记为降级；
2. 保留本地严格解码与语义校验；
3. 增加失败率监控；
4. 不向业务方宣传为 strict 等价能力。
## 10. 常见故障表

| 现象 | 常见原因 | 处理 |
|---|---|---|
| 400 invalid schema | Schema 关键字或层级不受支持 | 从最小 object 开始逐项增加 |
| strict 仍有额外字段 | 供应商未真正兼容 | 本地 DisallowUnknownFields 并做契约测试 |
| JSON 前有代码围栏 | 实际走了普通文本或 JSON mode | 检查 text.format 与模型能力 |
| Unmarshal 成功但字段全是零值 | JSON tag 不一致或字段缺失 | 检查 tag，并做 required 的本地验证 |
| 枚举值不合法 | Schema 漂移或兼容实现不严 | Validate 拒绝，不做模糊猜测 |
| 200 但没有 output_text | refusal、工具调用或输出不完整 | 遍历类型并分支处理 |
| 尾部还有解释文字 | 未启用 strict 或供应商降级 | 检测尾随内容 |
| 中文 summary 被字节截断 | 使用 len 按字节限制 | 用 utf8.RuneCountInString |
| Schema 很大且延迟高 | 接口承担过多职责 | 拆分任务，缩小字段和枚举 |
| 模型分类不符合政策 | 结构正确但语义错误 | 加业务规则、工具事实与人工复核 |
## 11. 练习

1. 为“文章审核建议”设计 strict Schema：action 为 allow、review、block；reasons 最多三条；contains_personal_data 为布尔值。
2. 为什么以下类型不足以发现 score 字段缺失？
~~~go
type Result struct {
	Score int `json:"score"`
}
~~~

## 12. 参考答案

1. 顶层 object；action 使用三值 enum，reasons 为 string array，contains_personal_data 为 boolean；全部 required 且 additionalProperties 为 false。数组上限还应在 Go 中复核。
2. 缺少 score 时 int 保留零值，无法区分缺失与明确返回 0；可用 *int、Schema 校验器或自定义解码。
3. 不能。未经标注集校准与持续评估，它只是模型生成的数值。
4. Go 使用 OrderID *string；Schema type 为 ["string","null"]，仍列入 required。
5. 返回带 debug 的 output_text；DisallowUnknownFields 应产生 unknown field 错误。
