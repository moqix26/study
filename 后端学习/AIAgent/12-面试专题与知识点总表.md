## 1. 面试表达的底线

每一句项目结论都应归入四类之一：

| 标签 | 含义 | 例子 |
|---|---|---|
| 已验证 | 有命令、测试或记录可复现 | `go test ./...` 在某 commit 通过 |
| 已实现待评估 | 功能存在，但质量/性能未形成结论 | 已接 pgvector，尚未建立标注集 |
| 设计方案 | 能解释接口和取舍，代码尚未完成 | 计划加入 OIDC 和资源级授权 |
| 未做 | 明确承认并说明会怎样验证 | 未做真实并发压测 |

最危险的表达：

- “准确率提升 30%”，但没有 baseline、样本和公式；
- “支持百万 QPS”，但没有硬件、负载和压测报告；
- “生产级”，但没有认证、授权、监控、发布和演练；
- “解决模型幻觉”，但只写了 prompt；
- “多 Agent”，但实际只有一个循环调用两个函数。

不会的内容可以说不会。

可信的回答是：“我没在项目中实现过，但我会先验证 A，再根据 B 选择 C。”

## 4. `agentgo` 三分钟项目讲解

### 4.1 问题

```text
很多 Agent 示例必须先配置云 Key，而且模型、工具、会话和 HTTP 粘在一起，
导致控制流难以离线测试，也难以替换 provider 或存储。
```

### 4.2 方案

```text
我在 Go 中按 llm、memory、tool、rag、agent、httpapi 分包，
默认使用 mock provider 和内存存储，因此无 Key 也能跑测试和服务。
Agent loop 有 context deadline 和最大步骤；工具调用由服务端 registry 校验执行。
HTTP 同时提供普通 JSON 和 SSE，steps 只返回已执行工具的摘要，不返回隐藏 Thought。
```

### 4.3 证据

只讲当前能展示的内容：

- `go test ./...`；
- `go test -race ./...`；
- 五组真实路由；
- `httptest` provider adapter；
- 一次取消、工具失败或 max steps 测试。

### 4.4 边界

```text
V1 的内存 RAG 和测试 embedder只证明接口与控制流，不能证明语义召回质量。
pgvector 是 V2 可选实现，需要真实 embedding、标注集和隔离测试。
真实认证、限流、发布和告警属于 V3，未完成前不称生产级。
```

### 4.5 追问：为什么不用现成 Agent 框架

可以回答：

```text
这个项目的学习目标是理解最小控制面，所以先用标准库和小接口实现有界 loop。
如果工作流复杂到需要持久状态图、人工审批和恢复，我会再评估框架；
但会保留 provider、tool policy 和业务类型边界，避免框架类型扩散到所有包。
```

不要攻击框架，也不要声称手写一定更优。

## 5. Go 语言基础高频题

### 5.1 slice

应该讲清：

- slice header 包含指针、长度、容量的概念；
- `append` 可能复用底层数组，也可能分配新数组；
- 子 slice 可能让大数组无法回收；
- 多个 slice 共享底层数组时修改可互相影响；
- 并发读写共享底层数组需要同步。

验证代码：

```go
func main() {
	a := make([]int, 2, 2)
	b := a[:1]
	b = append(b, 9)
	fmt.Println(a, b)
}
```

先预测，再运行；随后把容量改大观察差异。

### 5.2 map

- map 不是并发安全容器；
- 未初始化 map 可读但不可写；
- 迭代顺序不能依赖；
- 删除不存在 key 是允许的；
- 取值用 `v, ok` 区分零值与不存在。

不要只回答“底层是哈希表”，还要联系实际并发和 API 设计。

### 5.3 interface

- interface 由动态类型和值共同决定；
- typed nil 放进 interface 后，interface 可能不等于 nil；
- 小接口由使用方定义；
- 返回具体类型、接收接口常更易演进；
- 不为了 mock 把所有结构都抽象成接口。

typed nil 示例：

```go
func typedNilExample() {
	var p *MyError
	var err error = p
	fmt.Println(err == nil) // false
}
```

### 5.4 defer

- defer 参数在注册时求值；
- 多个 defer 后进先出；
- 适合 close/unlock/结束 span；
- 循环中 defer 可能延迟到函数结束，造成资源积累；
- named return 与 defer 修改返回值需要谨慎。

### 5.5 error

- 使用 `%w` 包装保留错误链；
- 使用 `errors.Is/As` 判断语义；
- 不靠字符串匹配业务错误；
- 只在有恢复策略时分类重试；
- HTTP 层映射错误，内部错误不直接泄露。

## 6. Go 并发高频题

### 6.1 Goroutine 与线程

回答重点：

- goroutine 由 Go runtime 调度；
- 初始栈和增长策略由实现管理，不背固定字节数；
- 多个 goroutine 映射到 OS thread；
- blocking syscall、网络轮询和调度由 runtime 协作处理；
- 轻量不等于可以无限创建。

### 6.2 WaitGroup

安全模式：

```go
func processAll(jobs []Job) {
	var wg sync.WaitGroup
	for _, job := range jobs {
		job := job
		wg.Add(1)
		go func() {
			defer wg.Done()
			process(job)
		}()
	}
	wg.Wait()
}
```

追问：为什么 `Add` 通常在启动 goroutine 前？

因为若在 goroutine 内 Add，主 goroutine 可能先执行 Wait 并错误返回，还可能触发不安全的 Add/Wait 交错。

### 6.3 Mutex 与 RWMutex

- Mutex 保护共享不变量，不只是某个字段；
- 锁范围要覆盖“检查再修改”的整体；
- 不在持锁期间做慢网络调用；
- RWMutex 只在真实读多写少且测得收益时采用；
- `go test -race` 检测 race，不证明不存在死锁或业务竞态。

### 6.4 Channel

- 无缓冲 channel 的发送/接收需要同步配对；
- 有缓冲 channel 在容量范围内解耦；
- 通常由发送方关闭；
- 向已关闭 channel 发送会 panic；
- 从已关闭 channel 接收先读完缓冲，随后得到零值和 `ok=false`；
- nil channel 的发送/接收永久阻塞，可用于动态关闭 select 分支。

### 6.5 select

```go
select {
case item := <-jobs:
	use(item)
case <-ctx.Done():
	return ctx.Err()
}
```

若多个 case 同时可用，选择不应被业务当作固定顺序。

`default` 会把阻塞等待变成轮询，可能造成 CPU 空转。

### 6.6 并发不是并行

并发描述多个任务的组织与交错；并行描述同一时刻在多个执行资源上运行。

面试时把它联系到：

- 多个 provider/tool 请求；
- 有界 worker pool；
- backpressure；
- context 取消；
- 不对写工具盲目并发。

## 7. Context

### 7.1 作用

`context.Context` 用于：

- cancel；
- deadline/timeout；
- 请求范围值。

不用于传递任意可选业务参数，也不应保存在长期结构体中。

### 7.2 传播

```text
HTTP request context
  → Agent loop
  → provider
  → tool
  → database
```

任何一层改用 `context.Background()` 都可能切断取消链。

### 7.3 timeout 分层

总请求 deadline 内可设置更短的下游 timeout，但子 deadline 不应超过父级。

需要区分：

- 客户端取消；
- 本地 deadline exceeded；
- provider 自己超时；
- 代理网关超时。

## 8. HTTP、JSON 与 SSE

### 8.1 Handler 需要处理

- method 与 path；
- body 大小；
- Content-Type；
- JSON unknown fields 策略；
- 必填与范围；
- request ID；
- context；
- 状态码和稳定错误格式；
- panic recovery 和安全日志。

### 8.2 SSE vs WebSocket

SSE 适合服务端到客户端的事件流，并复用 HTTP 语义。

WebSocket 适合真正双向、低开销持续消息。

聊天生成通常可以用 SSE，但如果需要高频双向实时协作再评估 WebSocket。

SSE 注意：

- 事件边界不是 TCP chunk；
- 代理缓冲；
- flush；
- 重连和重复副作用；
- 客户端取消；
- error/done 事件。

### 8.3 幂等

GET 应无副作用。

创建短链或写工具可能因网络超时出现“服务端已成功、客户端未知”。

可使用幂等键，把同一业务请求映射到同一结果。

## 9. 数据库与短链设计

### 9.1 短链基础流程

```text
POST /links
  → 校验长 URL
  → 生成或接受 short code
  → 唯一约束写库
  → 返回短链

GET /{code}
  → 查缓存/数据库
  → 检查禁用与过期
  → 记录异步统计
  → 3xx redirect
```

### 9.2 Code 生成取舍

| 方案 | 优点 | 风险/处理 |
|---|---|---|
| 自增 ID + Base62 | 无碰撞、短 | 可预测，需要 ID 策略 |
| 随机码 | 分布自然、难预测 | 碰撞重试，唯一约束兜底 |
| 内容哈希截断 | 同 URL 可稳定 | 碰撞、规范化和隐私泄漏 |
| 自定义 alias | 用户友好 | 抢占、敏感词、授权 |

无论哪种，数据库唯一约束才是最终并发兜底之一。

### 9.3 重定向状态码

不要背“短链一定用 301”。

- 永久重定向可能被客户端和中间层长期缓存；
- 临时重定向便于动态禁用、变更和统计；
- 302/303/307/308 对方法保留语义不同。

根据产品语义选择并测试客户端行为。

### 9.4 MySQL/PostgreSQL 索引

面试回答结构：

1. 查询是什么；
2. 过滤、排序、limit 是什么；
3. 候选索引列顺序；
4. 用执行计划验证；
5. 权衡写放大和空间；
6. 用真实数据分布测试。

不要说“加索引就快”。

### 9.5 事务

事务保证一组数据库操作的原子性/一致性边界，但不自动覆盖缓存、消息和外部 API。

跨系统一致性需要 outbox、幂等、补偿或业务状态机等设计。

### 9.6 Redis 缓存

短链读多时可缓存 code→URL。

必须回答：

- cache-aside 还是其他模式；
- TTL；
- 失效顺序；
- 热点 key；
- 穿透、击穿、雪崩；
- 禁用/删除后的陈旧窗口；
- Redis 故障时怎样回源和限流。

## 10. RAG 高频题

### 10.1 完整链路

```text
ingest → parse → chunk → embed → index
query  → embed → retrieve → optional rerank
       → build context → generate → verify citation
```

### 10.2 如何定位答非所问

先把问题拆成两层：

1. Retrieval：相关 chunk 是否进入 top-k？
2. Generation：进入后模型是否正确使用？

只看最终回答无法定位。

### 10.3 指标

- Recall@k：相关 chunk 是否被召回；
- MRR：首个相关 chunk 排名；
- groundedness：主张是否有证据；
- citation correctness：引用是否真的支持主张；
- latency/cost：质量之外的工程约束。

指标必须附数据集和标注规则。

### 10.4 pgvector

应该讲清：

- embedding 维度与列定义一致；
- distance/operator 与索引一致；
- exact search 与 ANN 的取舍；
- 索引参数依数据和查询评估；
- tenant filter；
- 模型更换与向量版本；
- `EXPLAIN` 验证查询计划。

## 11. Agent 与 Tool Calling 高频题

### 11.1 Agent 是什么

工程回答：

```text
Agent 是把模型决策、状态、工具和有界控制循环组合起来完成任务的应用系统。
```

不要只说“会自主思考的 AI”。

### 11.2 Tool Calling 流程

```text
应用给模型工具定义
  → 模型产生结构化调用意图
  → 应用做 schema/授权/确认
  → 执行工具
  → 用 call ID 回填结果
  → 模型继续或结束
```

模型提出参数，不代表参数可信。

### 11.3 怎样防无限循环

- max steps；
- 总 deadline；
- 相同 tool+args 检测；
- 工具失败明确标记；
- token/成本预算；
- 人工审批点；
- 达到上限返回可解释错误。

### 11.4 为什么不返回 Thought

- 应用不需要依赖隐藏推理文本；
- 它不稳定且可能含敏感信息；
- 可观测性应记录输入摘要、工具动作、结果摘要、状态和耗时；
- 用户需要的是结论、证据和已执行动作。

### 11.5 Prompt injection

prompt injection 无法只靠一句“忽略恶意指令”彻底解决。

防线包括：

- 数据与指令分层；
- 工具 allowlist；
- 服务端授权；
- URL/文件/SQL 限制；
- 输出校验；
- 人工确认；
- 日志与告警；
- 对抗测试。

## 12. LLM 原理高频题

### 12.1 Prefill vs Decode

- prefill：处理已有输入并建立 KV cache，影响 TTFT；
- decode：逐 token 生成，影响 TPOT 和输出总时长。

长 prompt 与长输出的性能影响不同。

### 12.2 KV cache

保存每层历史 token 的 K/V，避免 decode 时反复计算全部历史。

占用与层数、token、KV heads、head dimension、dtype 和并发序列近似相关。

不要把它说成业务会话数据库。

### 12.3 RAG vs Fine-tuning

- 最新/私有/可引用事实：优先 RAG/Tool；
- 稳定行为、风格、任务映射：微调候选；
- 可以组合；
- 两者都必须评估。

### 12.4 LoRA、DPO、RLVR

- LoRA：怎样高效更新参数；
- DPO：怎样从偏好对优化；
- RLVR：怎样用可程序验证奖励做强化学习。

它们不是互斥的同一层选项。

## 13. MCP 与 A2A 高频题

### 13.1 MCP

MCP 连接 AI Host 与工具/上下文 Server，使用初始化和 capability negotiation。

重点概念：

- Host / Client / Server；
- tools / resources / prompts；
- stdio / Streamable HTTP；
- 协议版本协商；
- 外部内容不可信；
- 工具发现不等于授权。

### 13.2 A2A

A2A 面向独立 Agent 服务之间发现能力、交换消息和跟踪任务。

重点概念：

- Agent Card；
- Message/Part；
- Task；
- Artifact；
- 流式或异步状态；
- 对端自报能力不等于信任。

### 13.3 区别

```text
MCP：应用怎样使用工具和上下文。
A2A：独立 Agent 怎样协作完成任务。
```

精确 API 和字段以面试当天官方规范为准，不背旧 SDK 构造函数。

## 14. 系统设计回答框架

面对“设计短链/Agent 服务”，按以下顺序：

1. 澄清功能与非功能需求；
2. 给出数量级假设，并明确是假设；
3. 定义 API 与数据模型；
4. 画主读写链路；
5. 找瓶颈和失败点；
6. 讨论一致性、安全和可观测性；
7. 给扩展顺序；
8. 说明怎样验证。

### 14.1 数量级估算

如果面试官没给数据，可以说：

```text
为了讨论容量，我先假设日创建量 X、重定向峰值 QPS Y、读写比 Z；
这些是设计输入，不是我的项目真实成绩。
```

然后展示计算，不把假设写进简历。

### 14.2 Agent 服务设计检查

- provider 抽象与 fallback；
- timeout、重试、幂等；
- conversation 和 tenant；
- Tool policy；
- RAG 隔离；
- SSE backpressure；
- token/成本预算；
- trace 与评估；
- prompt/schema 版本；
- 数据保留和删除。
