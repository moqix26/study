# 08｜评估、可观测、安全与成本：让 RAG 与 Agent 可以上线

> 贯穿工程：<code>Go工程/agentgo</code>
> 本章定位：V3 生产增强
> 核心结论：一次漂亮回答不是质量证明；必须分层评估、受控观测并设置工程护栏
## 1. 学习目标

学完本章，你应当能够：

- 分开衡量 retrieval、answer、tool 和 task；建立版本化的人工评估集；解释 LLM-as-a-Judge 的偏差与使用边界；设计不默认记录原文和 CoT 的 trace；
- 计算模型、embedding、reranker、工具与基础设施成本；分解端到端延迟；把限流与熔断放在正确层级；建立 RAG 与 Agent 的威胁模型；
- 为发布、回滚和事故响应定义可执行条件。
## 2. 为什么必须分层

用户只看到最终答案，但失败可能来自不同位置：

~~~text
资料没有写入
  -> 检索没召回
  -> 上下文装配丢证据
  -> 模型曲解证据
  -> 工具选择错误
  -> 工具参数错误
  -> 外部工具失败
  -> 任务未完成
~~~

只给最终答案打一个总分，无法知道应该修分块、索引、Prompt、工具 schema 还是状态机。 本章采用四层：

1. retrieval；
2. answer；
3. tool；
4. task。
每层都包含离线指标、线上信号和人工复核。
## 3. 评估数据模型

一条评估样本不只是 question 和 answer。

~~~go
package eval
type Case struct {
	ID                 string
	TenantFixture      string
	Question           string
	RelevantChunkIDs   []string
	RequiredClaims     []string
	ForbiddenClaims    []string
	ExpectedCitations  []string
	ExpectedTools      []ExpectedTool
	TaskOutcome        map[string]any
	Tags               []string
	DatasetVersion     string
}
type ExpectedTool struct {
	Name          string
	RequiredArgs  map[string]any
	ForbiddenArgs []string
	MayRepeat     bool
}
~~~

评估 fixture 使用虚构 tenant，不使用生产客户原文。 如果必须使用真实样本，应经过授权、脱敏和受控保留。
## 4. Retrieval 指标

Retrieval 评估只看“证据有没有被找到和排好”，暂时不看生成语言是否漂亮。
### 4.1 Recall@K

~~~text
Recall@K =
Top K 中相关 chunk 数
/
该问题全部标注相关 chunk 数
~~~

它回答：“需要的证据有多少进入前 K？” 如果一个问题有两个必要证据，只召回一个，Recall@K 不应被记为满分。
### 4.2 Precision@K

~~~text
Precision@K =
Top K 中相关 chunk 数
/
K
~~~

它回答：“送入上下文的内容有多少真正相关？” 高 recall 但低 precision 会浪费上下文，并增加模型被干扰的概率。
### 4.3 MRR

MRR 关注第一个相关结果的位置：

~~~text
RR = 1 / first_relevant_rank
MRR = 所有查询 RR 的平均值
~~~

它适合“一个关键答案块应尽量排在前面”的任务。 如果问题需要多个证据，仅用 MRR 不够。
### 4.4 nDCG

nDCG 支持多级相关性：

- 直接回答问题：高相关；提供背景：部分相关；只有关键词相似：低相关；无关：零。
它对排序质量比二元 precision 更细致。 但标注成本也更高。
### 4.5 Context Recall 与 Context Precision

当 chunk 标注不稳定时，可以按答案所需 claim 评估：

- context recall：所需 claim 有多少能在上下文找到支持；context precision：上下文中有多少内容真正用于支持答案。
这仍需要人工或可靠标注，不能只靠字符串包含。
### 4.6 Retrieval 切片

总体均值会掩盖问题，至少按以下标签切片：

- 中文、英文、混合语言；短问、长问；精确编号、语义问题；单文档、多文档；
- 新资料、旧资料；大 tenant、小 tenant；强 metadata 过滤；无答案问题；
- 同义词和错别字；Prompt Injection 文档。
不要只优化最容易的热门问法。
## 5. Answer 指标

### 5.1 Correctness

答案是否给出正确结论。 可以与 reference answer 或 required claims 比较。 reference answer 不是唯一合法措辞。 评估应关注事实和约束，而不是字面完全一致。
### 5.2 Faithfulness

答案中的每个事实 claim 是否受本轮证据支持。 流程：

1. 将答案拆成 atomic claims；
2. 为每个 claim 找支持证据；
3. 标记 supported、contradicted 或 not enough information；
4. 聚合成忠实度。
有 citation ID 不代表 faithful。 第 05 章已经解释，合法引用也可能被曲解。
### 5.3 Completeness

答案是否覆盖所有 required claims。 例如用户问退款条件和期限，只回答期限，即使内容正确也不完整。
### 5.4 Citation 指标

可以分别衡量：

- citation validity：引用 ID 是否来自本次证据；citation precision：被引用证据是否支持对应 claim；citation recall：需要来源的 claim 是否都有引用；citation location：引用是否紧跟对应主张；
- source integrity：URL、页码是否由服务端元数据映射。
不能把 citation validity 当成 citation precision。
### 5.5 Refusal 与无答案

评估两个方向：

- 应拒绝时是否拒绝；有足够证据时是否错误拒绝。
只追求“少幻觉”可能让系统对所有问题都回答不知道。 因此同时看 unsafe answer rate 与 false refusal rate。
### 5.6 格式与业务规则

结构化输出还要检查：

- JSON 是否通过 schema；枚举值是否合法；金额和日期格式；不应出现的敏感字段；
- 长度限制；输出语言；链接是否来自允许来源。
这些确定性规则优先由代码验证，不必交给 LLM judge。
## 6. Tool 指标

Agent 调工具需要独立评估。
### 6.1 选择质量

- tool selection accuracy：是否选择正确工具；unnecessary tool rate：本可直接回答却调用工具；missing tool rate：需要工具却未调用；forbidden tool rate：选择了无权限或不允许工具。
### 6.2 参数质量

- schema valid rate；required argument accuracy；identity override attempt；sensitive argument leakage；
- repair turns：参数错误后用了几轮修复。
ValidatedArgs 必须是服务端校验并注入身份后的参数。 不要用模型原始 JSON 作为最终审计事实。
### 6.3 执行质量

- tool success rate；timeout rate；transient retry rate；ambiguous outcome rate；
- duplicate side-effect rate；compensation rate；P50/P95/P99 duration。
重复副作用即使比例很低，也可能是严重事故，不能只看平均成功率。
### 6.4 轨迹合规

Step 只允许：

- tool 名称；脱敏后的 ValidatedArgs；ResultSummary；Status；
- Duration。
评估应扫描：

- 是否出现原始 CoT；是否保存完整系统 Prompt；是否出现 token、密钥、Cookie；是否保存超大原始工具结果；
- 是否越过租户边界。
## 7. Task 指标

Task 层关心用户目标是否完成。 常见指标：

- task success rate；partial completion rate；human takeover rate；approval rejection rate；
- cancellation rate；budget exhausted rate；steps per successful task；cost per successful task；
- end-to-end latency；user correction rate；repeated request rate。
成功定义必须可操作。 “模型说完成了”不是任务完成。 例如创建工单的成功条件应是：

~~~text
外部工单系统返回稳定 ticket_id
且 ticket 状态可查询
且本地 execution 已落库
~~~

## 8. 建立人工评估集

### 8.1 来源

- 产品需求与核心用例；历史故障；用户明确纠正过的问题；支持团队高频问题；
- 安全团队攻击样本；边界条件和无答案问题；合成样本，但需人工审核。
### 8.2 标注

每条至少有：

- 问题；适用 tenant fixture；relevant chunk 或证据；required claims；
- forbidden claims；是否应该拒绝；预期工具和关键参数；最终任务成功条件；
- 标签与难度。
### 8.3 版本化

评估集像代码一样版本化：

~~~text
testdata/eval/
├─ dataset-v1.jsonl
├─ dataset-v2.jsonl
├─ retrieval-golden.jsonl
├─ safety-cases.jsonl
└─ README.md
~~~

每次报告必须记录：

- dataset version；应用 commit；Prompt version；embedding version；
- generation model；reranker version；tool schema version；评估器版本。
否则两个分数不可复现。
### 8.4 划分

- development set：日常调试；validation set：选择方案；holdout set：发布前确认；adversarial set：安全与极端输入；
- production shadow set：脱敏后的真实分布样本。
不要反复针对 holdout 调 Prompt，否则它不再是 holdout。
## 9. LLM-as-a-Judge 的局限

Judge 适合辅助评估开放文本，但它不是事实裁判。 常见偏差：

- position bias：偏好先出现的候选；verbosity bias：偏好更长、更像解释的答案；self-preference：偏好同家族模型风格；prompt sensitivity：评分随措辞变化；
- model drift：供应商升级后标准变化；reference anchoring：过度贴近参考措辞；injection：待评文本诱导 judge 改规则；domain weakness：专业领域判断错误。
### 9.1 正确使用方式

1. 先定义 rubric；
2. 给出明确等级和反例；
3. 随机交换候选顺序；
4. 对关键样本使用多次或多 judge；
5. 与人工标注计算一致性；
6. 定期重新校准；
7. 保存 judge 模型与 Prompt 版本；
8. 确定性规则仍由代码执行；
9. 高风险结论由人工负责。
Judge 分数适合趋势和筛查。 它不适合单独批准付款、医疗、法律或权限决策。
### 9.2 防止被评文本注入

Judge Prompt 要把 candidate answer 和 context 放在明确数据边界中。 即便如此，仍可能受注入影响。 可增加：

- 结构化字段；转义与长度限制；多 judge 对照；对可疑指令的规则检测；
- 人工复核。
## 10. 离线、影子与线上评估

### 10.1 离线

优点：

- 可复现；不影响用户；易比较多个版本。
缺点：

- 分布可能过时；难覆盖真实交互；人工标注有限。
### 10.2 Shadow

复制脱敏请求给候选版本，但不把候选答案返回用户。 比较：

- 检索差异；工具决策；延迟；token；
- 成本；安全规则触发。
Shadow 仍会产生供应商费用和数据传输，必须受隐私政策约束。
### 10.3 Canary 与 A/B

Canary 用小流量验证稳定性和回滚。 A/B 用于比较用户结果，但要注意：

- 用户分桶稳定；tenant 不跨实验泄漏；指标窗口足够；模型成本差异；
- 高风险功能不盲目实验；明确停止条件。
## 11. agentgo 的评估 Harness

~~~text
internal/eval/
├─ dataset.go
├─ retrieval.go
├─ answer.go
├─ tool.go
├─ task.go
├─ judge.go
└─ report.go
~~~

建议接口：

~~~go
package eval
import "context"
type System interface {
	RunCase(ctx context.Context, c Case) (Observed, error)
}
type Metric interface {
	Name() string
	Score(expected Case, observed Observed) Result
}
type Report struct {
	DatasetVersion string
	SystemVersion  string
	Results        []Result
	Slices         map[string]Summary
}
~~~

评估运行默认使用 mock 和 fixture 数据。 在线供应商评估通过显式环境变量开启，并设置费用上限。
## 12. Trace：记录控制流，不复制所有内容

### 12.1 推荐 Span

~~~text
http POST /api/rag/ask
├─ auth.resolve_identity
├─ rag.embed_query
├─ rag.vector_search
├─ rag.pack_context
└─ llm.generate
http POST /api/agent/run
├─ agent.model_turn
├─ tool.validate
├─ tool.authorize
├─ tool.execute
└─ agent.checkpoint
~~~

每个 span 记录开始、结束、状态和受控属性。
### 12.2 属性白名单

可记录：

- request_id、trace_id、run_id；哈希或受控 tenant 标识；provider、model、operation；Prompt template version；
- embedding version；token 数；candidate 数和 chunk ID；tool name；
- step ordinal；duration；retry count；error category；
- estimated cost。
默认不记录：

- 完整 Prompt；完整用户问题；完整文档 chunk；完整工具结果；
- API Key、Cookie、Authorization；原始 CoT 或隐藏 reasoning。
### 12.3 日志、指标与 Trace

- 日志：离散事件与错误上下文；指标：聚合趋势与告警；Trace：一次请求跨组件的时间线。
三者通过 request_id、trace_id、run_id 关联。 不要把所有数据都塞进日志模拟 tracing。
## 13. 隐私与数据保留

先做数据分类：

| 类别 | 示例 | 默认策略 |
|---|---|---|
| public | 公开文档 ID | 可记录 |
| internal | 内部模型版本 | 控制访问 |
| confidential | 客户问题、合同片段 | 默认不记录原文 |
| restricted | 密钥、身份证、支付信息 | 禁止进入日志 |
工程措施：

- 采集前最小化；字段白名单；服务端脱敏；tenant 级访问控制；
- 静态和传输加密；短保留期；删除流程；审计访问；
- 非生产环境使用合成数据。
哈希不是万能匿名化。 低熵邮箱、手机号可能被字典反推，应使用带密钥的 HMAC 或受控映射。
## 14. 成本公式

### 14.1 生成模型

~~~text
generation_cost =
  input_tokens / 1,000,000 * input_price_per_million
  +
  output_tokens / 1,000,000 * output_price_per_million
~~~

若缓存 token、推理 token或批处理有不同价格，应分别计算。 价格表必须带生效日期和供应商版本。
### 14.2 embedding

~~~text
ingest_embedding_cost =
  document_embedding_tokens / 1,000,000
  * embedding_price_per_million
query_embedding_cost =
  query_embedding_tokens / 1,000,000
  * embedding_price_per_million
~~~

写入成本和查询成本分开。 文档更新、重嵌入和迁移也属于写入成本。
### 14.3 Agent 总成本

~~~text
run_cost =
  sum(model_turn_cost)
  + sum(embedding_cost)
  + sum(reranker_cost)
  + sum(paid_tool_cost)
  + allocated_infrastructure_cost
~~~

每成功任务成本：

~~~text
cost_per_success =
  all_run_cost
  /
  successful_task_count
~~~

只看单请求平均成本会掩盖失败重试和超长尾任务。
### 14.4 估算与账单

Usage 响应可用于实时预算，但最终账单可能有舍入、缓存和价格变更。 因此区分：

- estimated cost：请求时估算；billed cost：供应商账单；allocated cost：基础设施分摊。
定期对账，监控估算偏差。
## 15. 延迟分解

~~~text
total_latency =
  auth
  + queue_wait
  + query_embedding
  + retrieval
  + reranking
  + context_pack
  + model_first_token
  + model_generation
  + tool_calls
  + persistence
~~~

流式响应要分别看：

- time to first byte；time to first meaningful token；total completion time。
优化前先知道时间花在哪里。 不能只看 HTTP 总耗时猜测数据库慢。
## 16. 限流放在哪里

只在公网入口限请求数不够。 不同资源需要不同限流器。
### 16.1 边缘与身份层

- 每 IP 的粗粒度防滥用；每 UserID 与 TenantID 的配额；未认证路由更严格；请求体大小限制。
演示版 X-User-ID 不能视为生产认证。
### 16.2 路由层

- <code>/api/chat</code>：请求数、并发和 token 估算；<code>/api/rag/ingest</code>：文档大小、任务数与每日 token；<code>/api/rag/ask</code>：查询频率和 top_k 上限；<code>/api/agent/run</code>：并发 run、max steps 和费用预算。
### 16.3 依赖层

模型 gateway：

- provider/model 维度的并发；request 与 token bucket；429 自适应退避。
工具层：

- 每个 tool 独立并发；写操作更严格；tenant 配额；外部 API 配额。
Worker：

- 队列长度；消费并发；embedding 批次；数据库连接池。
限流拒绝应返回可识别错误和 Retry-After，而不是伪装 500。
## 17. 熔断放在哪里

熔断针对远程依赖的持续失败：

- 某个模型 provider + operation；embedding endpoint；reranker；每个外部工具；
- 必要时数据库只读降级路径。
不要用一个全局熔断器把所有供应商和所有操作绑在一起。 聊天模型失败不一定意味着 embedding 也失败。
### 17.1 状态

~~~text
closed
  -> 失败率或慢调用达到条件
open
  -> 经过冷却时间
half-open
  -> 少量探测成功
closed
~~~

只统计可归因于依赖的失败。 用户参数错误、权限拒绝和本地 schema 错误不应触发供应商熔断。
### 17.2 降级

可选降级：

- 切备用模型；RAG 只返回检索片段；Agent 暂停有副作用步骤；使用缓存的公开结果；
- 返回可重试状态。
降级不能绕过安全。 备用模型也必须通过相同数据与工具权限控制。
## 18. 安全威胁模型

### 18.1 身份与租户

- TenantID 只来自已验证身份；repository 强制 tenant 谓词；工具身份由服务端注入；缓存键包含 tenant；
- trace 查询也做 tenant 与角色授权；跨租户测试是发布门槛。
### 18.2 Direct Prompt Injection

用户直接要求忽略规则、泄露提示或越权调用工具。 处理依赖：

- 系统边界；工具授权；参数 schema；最小权限；
- 输出过滤；审计与限流。
仅增加一句“不要被注入”不够。
### 18.3 Indirect Prompt Injection

RAG 文档、网页、邮件和工具结果都可能包含恶意指令。 它们是不可信数据，不能获得系统指令地位。 措施：

- 明确数据边界；抓取源 allowlist；内容扫描与风险标签；工具调用独立授权；
- 高风险动作审批；限制工具结果进入上下文的内容；对外部 URL 做 SSRF 防护。
### 18.4 SSRF

读取 URL 工具必须：

- 只允许 http/https；解析 DNS 后阻止内网、回环、链路本地地址；重定向后重新校验；限制端口；
- 限制响应大小与时间；禁止访问云元数据地址；记录最终目标的受控摘要。
字符串前缀检查不足以防 DNS rebinding 和重定向。
### 18.5 工具最小权限

- 读写工具分开；凭据按工具和环境隔离；默认拒绝未知工具；写操作使用幂等；
- 高风险动作审批；限制批量大小；设置 tenant 范围；不把通用 shell、任意 SQL 暴露给模型。
### 18.6 数据投毒

攻击者可能上传错误资料，让其在检索中排名靠前。 需要：

- 上传者权限；来源与版本；审核状态；文档签名或可信来源；
- 异常重复检测；删除和回滚；评估集中的投毒样本。
### 18.7 输出安全

模型输出仍是不可信文本。 前端展示时：

- HTML 转义；Markdown sanitizer；链接协议 allowlist；禁止脚本和危险属性；
- 下载内容设置正确 Content-Type；结构化字段再次校验。
## 19. Secrets 与供应链

Secrets：

- 不写入仓库；不进入 Prompt；不进入 trace；用环境或 secret manager 注入；
- 最小权限；定期轮换；泄露后可吊销。
依赖与模型供应链：

- 固定模块版本；漏洞扫描；记录模型与 Prompt 版本；审查 MCP 或工具服务来源；
- 对工具 schema 变更做兼容测试；新模型先离线和 canary；保留回滚路径。
## 20. SLI、SLO 与告警

建议 SLI：

- 可用请求比例；task success；retrieval recall 的离线趋势；unsafe answer rate；
- duplicate side-effect count；P95/P99 latency；429 与熔断比例；cost per success；
- queue age；pending/failed job 数；跨租户测试结果。
SLO 数值必须根据产品风险和历史基线制定。 不要从其他公司复制百分比。 告警要指向行动：

- 发生了什么；影响哪些 tenant、路由或 provider；从何时开始；当前降级是否生效；
- runbook 在哪里；谁负责处理。
## 21. 发布门槛

一次变更可能涉及 Prompt、模型、分块、索引或工具。 发布报告至少包含：

1. 变更项和版本；
2. 固定评估集结果；
3. 各关键 slice；
4. 安全 adversarial 结果；
5. 延迟和成本；
6. 失败案例；
7. canary 计划；
8. 回滚条件；
9. 数据迁移状态；
10. 负责人。
如果总体分数提高但跨租户安全测试失败，不能发布。 安全门槛不是可以被平均质量抵消的指标。
## 22. 事故响应

发生异常回答或错误工具调用时：

1. 根据 request_id、trace_id、run_id 定位；
2. 冻结相关版本或工具；
3. 必要时打开熔断和降级；
4. 保存最小必要证据；
5. 判断是否涉及数据泄漏或外部副作用；
6. 撤销凭据或执行补偿；
7. 通知相关负责人；
8. 建立可复现评估样本；
9. 修复并离线验证；
10. canary 恢复；
11. 更新 runbook 和评估集。
事故复盘关注系统为何允许错误扩大，而不只归因于“模型不稳定”。
## 23. 常见故障表

| 现象 | 常见原因 | 诊断 | 修复 |
|---|---|---|---|
| 总分高但用户仍不满 | 评估集分布失真 | 看 slice 与反馈 | 补真实难例 |
| Recall 高但答案差 | 上下文噪声或生成不忠实 | answer 指标 | 调装配和 Prompt |
| 引用都合法仍答错 | citation validity 被当忠实度 | claim 证据对照 | faithfulness |
| Judge 分数漂移 | 模型或 Prompt 变更 | 版本与人工集 | 重新校准 |
| Trace 泄露客户原文 | 默认采全 payload | 字段审计 | 白名单与脱敏 |
| 成本突然上升 | 重试、上下文膨胀、模型切换 | cost breakdown | 预算和版本告警 |
| 只在入口限流仍 429 | 内部并发扇出 | provider/tool 指标 | 分层限流 |
| 熔断器经常误开 | 把 4xx 算依赖故障 | 错误分类 | 只统计可恢复依赖错 |
| 备用模型越权 | 降级绕过工具策略 | 对比执行路径 | 共用授权层 |
| 工具成功率高但任务失败 | 选错工具或结果未用于目标 | tool 与 task 对照 | 修策略和成功定义 |
| 平均延迟正常但投诉多 | 长尾被均值掩盖 | P95/P99 | 查慢 span |
| 安全测试被平均分掩盖 | 门槛设计错误 | 发布报告 | 安全硬门槛 |
## 24. 练习

### 练习 1

一个问题需要两个证据，Top 5 找到其中一个，另外四个无关。 计算 Recall@5 与 Precision@5。
### 练习 2

答案引用 S1，S1 确实存在，但不支持对应 claim。 分别说明 citation validity 与 citation precision。
### 练习 3

为什么不能仅用同一个生成模型给自己的答案打分并作为发布依据？
### 练习 4

Agent 有三轮模型调用和两个付费工具。 写出总成本表达式，并指出为什么还要看 cost per success。
### 练习 5

模型供应商持续 503，但用户参数错误也很多。 熔断器应统计哪些错误？
### 练习 6

设计一条跨租户安全发布门槛。
## 25. 参考答案

### 答案 1

总相关证据为 2，召回 1：

~~~text
Recall@5 = 1 / 2 = 0.5
Precision@5 = 1 / 5 = 0.2
~~~

### 答案 2

S1 属于本轮可用来源，所以 citation validity 通过。 S1 不支持该 claim，所以 citation precision 失败，答案也可能不忠实。
### 答案 3

存在 self-preference、verbosity、position、Prompt sensitivity 与模型漂移。 Judge 只能作为辅助趋势，应与人工集、确定性规则和独立评估结合。
### 答案 4

~~~text
run_cost =
  model_turn_1
  + model_turn_2
  + model_turn_3
  + paid_tool_1
  + paid_tool_2
  + allocated_infrastructure
~~~

如果大量 run 花钱后失败，平均单次成本可能不高，但每成功任务成本会很差。
### 答案 5

统计可归因于该供应商的 503、连接失败、超时和慢调用。 用户参数 4xx、权限拒绝和本地 schema 错误不应触发供应商熔断。
### 答案 6

准备两个 tenant 的相似资料。 所有 RAG、缓存、Agent 工具和 trace 查询均以 A 身份尝试读取 B。 任何一次返回 B 数据都阻止发布，不能用总体平均分抵消。

## 27. 与后续项目衔接

到这里，agentgo 已形成三层递进：

~~~text
V1
  mock + memory
  离线掌握接口和控制流
V2
  pgvector + outbox
  可靠知识库写入和检索
V3
  durable agent + eval + tracing + safety
  可恢复、可评估、可治理
~~~

后续项目不应先追求更多框架和更多 Agent。 优先把一条真实业务链做深：

1. 建人工评估集；
2. 跑通 RAG 与引用；
3. 接一个只读工具；
4. 接一个有幂等保护的写工具；
5. 加状态机与恢复；
6. 加分层指标和安全门槛；
7. 用数据决定下一项优化。
当你能解释每个失败发生在哪一层，并用测试证明修复没有越过租户和预算边界，才算真正具备可上线的 Agent 工程能力。
