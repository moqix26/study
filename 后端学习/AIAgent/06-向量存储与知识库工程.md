# 06｜向量存储与知识库工程：PostgreSQL + pgvector

> 贯穿工程：<code>Go工程/agentgo</code>
> 本章阶段：V2，通过 <code>RAG_STORE=pgvector</code> 切换存储
> 第一实现：PostgreSQL + pgvector，而不是先堆叠独立向量数据库
>
> **当前实现边界**：`agentgo` 目前只有同步 `internal/rag/pgvector_store.go`、一张 `rag_chunks` 表和 128 维 Hash Embedding；默认单测不启动数据库，也没有真实 Embedding、documents/outbox/worker。下面更完整的 Schema、迁移与异步任务是演进设计，不是已完成功能。
## 1. 学习目标

完成本章后，你应当能够：

- 解释为什么 agentgo 首先选择 PostgreSQL + pgvector；设计 documents、chunks、embedding jobs 与 outbox 表；正确使用向量距离操作符；理解 HNSW 的图结构、构建成本与查询参数；
- 分析 tenant 过滤为何可能影响近似检索结果；用查询计划和评估数据选择索引，而不是使用固定规模阈值；安全迁移 embedding 模型和向量维度；用 outbox、幂等键、租约和重试完成可靠入库；
- 处理更新、删除、备份、恢复与数据保留。
## 2. 从 V1 到 V2

第 05 章的内存 Store 保留不动。当前基线通过同一 `rag.Store` 切换：

~~~text
internal/rag/
├─ types.go
├─ memory_store.go
├─ pgvector_store.go
├─ chunker.go
└─ embedder.go
~~~

`documents`、outbox、租约 worker 和 embedding worker 是本章要求你设计的后续扩展，不在当前文件树中。

启动时依据配置选择：

~~~text
RAG_STORE=memory
RAG_STORE=pgvector
DATABASE_URL=postgres://...
~~~

业务层仍依赖 Repository 接口。 HTTP handler、检索服务和答案服务不应知道底层是 slice 还是 PostgreSQL。
## 3. 为什么优先 PostgreSQL + pgvector

对许多后端项目，知识库还需要：

- 用户和租户关系；文档状态；唯一约束；事务；
- 审计字段；metadata 过滤；删除与保留策略；备份和恢复。
PostgreSQL 已经擅长这些关系数据能力。 pgvector 在同一事务系统中增加向量类型和距离索引，能减少早期系统数量与同步边界。 这不表示 pgvector 永远优于独立向量库。 当需求出现以下特征时，应基于压测与运维约束重新评估：

- 多地域和特殊复制要求；复杂向量原生能力；极高并发或特殊延迟目标；托管服务能力差异；
- 团队已有成熟平台。
不要用“数据超过 N 条必须换库”作结论。 数据维度、过滤选择性、硬件、召回目标、写入速度与查询并发共同决定结果。
## 4. 版本与迁移原则

数据库能力会随 PostgreSQL 与 pgvector 版本变化。 工程中应记录：

- PostgreSQL 主版本；pgvector 扩展版本；数据库驱动版本；migration 版本；
- embedding 模型、版本和维度；距离度量；索引参数。
SQL 示例必须先在项目固定版本上验证。 如果使用 iterative scan 等较新能力，应检查已安装扩展是否支持，而不是只看网络文章。
## 5. 基础 Schema

以下 SQL 假设一个示例 embedding 维度为 1536。它不是当前工程实际值：当前 `DefaultDimension=128`。1536 不是推荐值，也不是通用标准；它必须与实际模型输出一致。

当前 Compose 固定 pgvector 0.8.1，基线会创建 `vector` 的 HNSW 索引，因此 adapter 只应接受该索引支持的维度范围（`vector` HNSW 上限为 2000）。若真实 embedding 超出范围，必须重新设计列类型/索引或使用精确检索，不能只改维度常量。
### 5.1 扩展

~~~sql
CREATE EXTENSION IF NOT EXISTS vector;
~~~

生产迁移账号是否有创建扩展权限，应在部署前确认。 不要让普通应用账号拥有不必要的扩展管理权限。
### 5.2 文档表

~~~sql
CREATE TABLE rag_documents (
    tenant_id        text        NOT NULL,
    id               text        NOT NULL,
    external_key     text        NOT NULL,
    title            text        NOT NULL,
    source_uri       text,
    content_hash     text        NOT NULL,
    version          bigint      NOT NULL,
    status           text        NOT NULL,
    created_at       timestamptz NOT NULL DEFAULT now(),
    updated_at       timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY (tenant_id, id),
    UNIQUE (tenant_id, external_key, version),
    CHECK (status IN ('pending', 'ready', 'failed', 'deleting'))
);
~~~

联合主键使 tenant 成为文档身份的一部分。 外部系统的 document ID 不一定全局唯一，不应暗自假设它唯一。
### 5.3 chunk 表

~~~sql
CREATE TABLE rag_chunks_v1 (
    tenant_id          text         NOT NULL,
    id                 text         NOT NULL,
    document_id        text         NOT NULL,
    ordinal            integer      NOT NULL,
    content            text         NOT NULL,
    content_hash       text         NOT NULL,
    token_count        integer      NOT NULL,
    heading_path       text[],
    page_number        integer,
    metadata           jsonb        NOT NULL DEFAULT '{}'::jsonb,
    embedding_model    text         NOT NULL,
    embedding_version  text         NOT NULL,
    embedding          vector(1536) NOT NULL,
    created_at         timestamptz  NOT NULL DEFAULT now(),
    PRIMARY KEY (tenant_id, id),
    UNIQUE (tenant_id, document_id, ordinal),
    FOREIGN KEY (tenant_id, document_id)
        REFERENCES rag_documents (tenant_id, id)
        ON DELETE CASCADE
);
~~~

外键同时包含 tenant_id，避免把 A 租户 chunk 关联到 B 租户文档。 这是一项数据库层不变量，不应只依赖应用自觉。
### 5.4 辅助索引

~~~sql
CREATE INDEX rag_chunks_document_idx
    ON rag_chunks_v1 (tenant_id, document_id, ordinal);
CREATE INDEX rag_chunks_metadata_gin_idx
    ON rag_chunks_v1 USING gin (metadata);
~~~

是否保留 metadata GIN 索引，要根据真实谓词和写入成本决定。 索引不是免费午餐：它增加磁盘、写放大和 vacuum 工作。
## 6. 距离操作符

pgvector 常见操作符包括：

| 操作符 | 含义 | 排序方向 |
|---|---|---|
| <code>&lt;-&gt;</code> | L2 距离 | 越小越近 |
| <code>&lt;#&gt;</code> | 负内积 | 越小排序越接近 |
| <code>&lt;=&gt;</code> | cosine distance | 越小越近 |
如果使用 cosine distance，可把展示相似度写为：

~~~text
similarity = 1 - cosine_distance
~~~

但该值不能直接当成概率或跨模型置信度。
### 6.1 精确查询

~~~sql
SELECT
    id,
    document_id,
    content,
    1 - (embedding <=> $1::vector) AS similarity
FROM rag_chunks_v1
WHERE tenant_id = $2
ORDER BY embedding <=> $1::vector
LIMIT $3;
~~~

对索引能否使用而言，ORDER BY 表达式形态很重要。 不要先在应用层拉出整个租户数据再算距离。
### 6.2 Go 参数

使用 pgx adapter 时，应由库提供的 vector codec 或明确转换层编码向量。 不要通过字符串拼接生成 SQL 向量，也不要拼接 tenant。

~~~go
func (r *PostgresRepository) Search(
	ctx context.Context,
	q rag.SearchQuery,
) ([]rag.SearchHit, error) {
	if q.TenantID == "" {
		return nil, errors.New("tenant is required")
	}
	if len(q.Vector) != r.dimension {
		return nil, errors.New("vector dimension mismatch")
	}
	limit := clamp(q.CandidateLimit, 1, r.maxCandidates)
	// 使用参数化查询执行 SQL，并扫描 chunk 与 distance。
	return r.searchCosine(ctx, q.TenantID, q.Vector, limit)
}
~~~

业务入口必须限制 CandidateLimit。 否则客户端可以用超大 top_k 制造昂贵查询。
## 7. HNSW 是什么

HNSW 是 Hierarchical Navigable Small World。 可以把它理解为多层近邻图：

- 上层节点少，用于快速跳到大致区域；下层节点多，用于更细致地寻找近邻；查询沿图边前进，不需要扫描全部向量；返回的是近似近邻，因此速度与召回存在权衡。
它不是“打开就一定更快”。 小表、强过滤、缓存状态、并发与磁盘条件都会改变结果。
### 7.1 创建索引

~~~sql
CREATE INDEX rag_chunks_embedding_hnsw_idx
ON rag_chunks_v1
USING hnsw (embedding vector_cosine_ops);
~~~

操作符类必须与查询的距离度量一致。 如果索引用 cosine，查询却按 L2 排序，不能期待该索引正确服务该排序。
### 7.2 关键参数

HNSW 常见参数：

- <code>m</code>：每个节点连接规模的控制参数；<code>ef_construction</code>：构建时搜索宽度；<code>ef_search</code>：查询时搜索宽度。
通常：

- 更大的构建参数可能提高图质量；更大的查询搜索宽度可能提高召回；代价是构建时间、内存、查询 CPU 或延迟增加。
不要仅凭“推荐参数表”上线。 应在自己的语料、过滤条件和并发模型下比较。
### 7.3 何时建 HNSW

不使用固定行数阈值。 决策流程应是：

1. 明确 P95/P99 延迟、吞吐与 recall 目标；
2. 建立代表性查询集和过滤分布；
3. 测精确扫描基线；
4. 创建 HNSW 并测构建、写入和查询；
5. 比较索引大小、缓存命中和 vacuum 影响；
6. 检查查询计划；
7. 在可接受召回下降内选择参数。
只有数据量，没有查询分布，无法做可靠选择。
## 8. 过滤与近似检索

### 8.1 tenant 过滤不是可选项

查询必须是：

~~~sql
WHERE tenant_id = $2
ORDER BY embedding <=> $1::vector
~~~

TenantID 来自认证后的服务端身份。 不能由 <code>POST /api/rag/ask</code> 请求体指定。
### 8.2 为什么过滤可能少返回

近似索引先探索一部分候选，普通过滤条件可能在之后淘汰候选。 当 tenant 或 metadata 过滤很强时，最终可能拿不到期望数量。 处理方向包括：

- 在支持的 pgvector 版本中调整查询搜索宽度；使用受支持的 iterative scan 能力；对低基数且稳定的条件建立 partial index；根据租户分布考虑分区；
- 先筛选候选子集后做精确排序；调整召回候选数并通过评估验证。
不同办法有不同写入与运维成本。 不能只把 ef_search 无限调大。
### 8.3 partial index

如果只有少数固定类别，可能使用：

~~~sql
CREATE INDEX rag_chunks_product_hnsw_idx
ON rag_chunks_v1
USING hnsw (embedding vector_cosine_ops)
WHERE metadata @> '{"category":"product"}';
~~~

但若 category 值频繁增加，为每个值建索引会失控。 索引策略必须根据基数、变化率与查询占比设计。
### 8.4 分区

按 tenant 分区可能改善部分场景，也可能制造大量分区和管理负担。 不应为每个小租户无条件创建独立分区。 评估时关注：

- 租户数量和增长；大小租户倾斜；单租户与跨租户查询；分区裁剪；
- migration 和备份复杂度；连接与 catalog 压力。
### 8.5 查看查询计划

~~~sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT id, document_id
FROM rag_chunks_v1
WHERE tenant_id = 'tenant-demo'
ORDER BY embedding <=> '[0.1,0.2,0.3]'::vector
LIMIT 10;
~~~

示例向量维度需要与真实列一致，实际执行时应换成完整测试向量。 重点观察：

- 是否走预期索引；实际行数与估算行数；过滤掉多少候选；buffer 命中；
- 排序和扫描耗时；冷缓存与热缓存差异。
## 9. metadata 设计

不是所有字段都应塞进 JSONB。 适合普通列：

- tenant_id；document_id；status；embedding_version；
- 高频等值过滤字段；有外键或强约束的字段。
适合 JSONB：

- 来源特有、稀疏且变化快的属性；低频展示元数据；暂未稳定的解析结果。
同一个字段若成为高频查询谓词，应考虑提升为普通列。 Schema 演进比长期承受不可控查询更健康。
## 10. 混合检索

向量检索擅长语义相似，但对精确编号、错误码和专有名词可能不稳定。 知识库可组合：

- PostgreSQL full-text search；trigram 或精确字段匹配；向量召回；reranker；
- Reciprocal Rank Fusion 等融合方式。
不要把不同分数直接相加，除非已做归一化和评估。 融合参数要在固定查询集上验证。 本章仍以向量 Repository 为主，混合检索可作为 V3 增强。
## 11. embedding 模型迁移

### 11.1 不能原地混用

旧模型和新模型即使维度相同，也未必处于同一向量空间。 不能因为列能写入就混合检索。
### 11.2 推荐版本表

一种做法是每个 embedding 主版本使用独立 chunk 表或独立 embedding 表：

~~~text
rag_chunks_v1 -> model-a, dimension 1536
rag_chunks_v2 -> model-b, dimension 1024
~~~

另一种做法是公共 chunk 内容表加版本化 embedding 子表。 选择取决于更新频率、存储重复与查询复杂度。
### 11.3 在线迁移步骤

1. 创建新表和新索引；
2. 新写入开始双写任务，而非盲目双写网络调用；
3. 后台扫描缺失的新版本 embedding；
4. 使用幂等键避免重复；
5. 对比覆盖率、召回指标、延迟和成本；
6. 小比例查询切到新版本；
7. 扩大灰度并保留旧路径；
8. 稳定后停止旧版本写入；
9. 过保留期再删除旧表或索引。
回滚条件必须在切换前定义。 不要等线上变差后才讨论什么算失败。
## 12. 为什么需要 outbox

一次入库同时涉及：

- PostgreSQL 事务；外部 embedding API；可能的对象存储；后台任务。
无法用普通数据库事务原子包住外部 HTTP 调用。 若先提交文档再调用模型，进程可能在中间崩溃。 若先调用模型再提交数据库，也可能模型收费成功但数据库失败。 Outbox 解决的是“业务状态与待处理事件同事务落库”。
### 12.1 outbox 表

~~~sql
CREATE TABLE outbox_events (
    id             uuid        PRIMARY KEY,
    aggregate_type text        NOT NULL,
    aggregate_id   text        NOT NULL,
    tenant_id      text        NOT NULL,
    event_type     text        NOT NULL,
    payload        jsonb       NOT NULL,
    status         text        NOT NULL DEFAULT 'pending',
    attempts       integer     NOT NULL DEFAULT 0,
    available_at   timestamptz NOT NULL DEFAULT now(),
    lease_owner    text,
    lease_until    timestamptz,
    last_error     text,
    created_at     timestamptz NOT NULL DEFAULT now(),
    completed_at   timestamptz,
    UNIQUE (tenant_id, event_type, aggregate_id)
);
~~~

唯一键是示意。 如果同一 aggregate 允许多个版本事件，幂等键必须包含版本。
### 12.2 同事务写入

~~~text
BEGIN
  upsert rag_documents，状态设为 pending
  insert outbox_events，事件为 document.embedding.requested
COMMIT
~~~

提交成功意味着“文档记录和待处理事件同时存在”。 提交失败则二者都不存在。
## 13. Worker、租约与幂等

### 13.1 抢占任务

Worker 可使用短事务和 <code>FOR UPDATE SKIP LOCKED</code> 抢任务。 抢到后写 lease_owner、lease_until 与 attempts，然后提交事务。 不要在持有行锁的事务中等待远程 embedding API。 这会造成长事务、锁竞争和连接占用。
### 13.2 处理步骤

1. 短事务抢占 pending 或租约过期事件；
2. 提交租约；
3. 加载文档并分块；
4. 调 embedding API；
5. 校验向量数量、模型和维度；
6. 新事务幂等 upsert chunk；
7. 将文档标为 ready；
8. 将 outbox 标为 completed；
9. 失败时记录分类并设置下一次 available_at。
### 13.3 租约恢复

Worker 可能在调用模型后、提交结果前崩溃。 租约过期后其他 Worker 会再次处理。 因此所有最终写入都必须幂等。 至少需要：

- 稳定 chunk ID；文档版本；embedding 版本；唯一约束；
- compare-and-set 状态更新；最大尝试次数；dead-letter 或人工处理状态。
### 13.4 重试

适合重试：

- 429；临时 5xx；网络超时；数据库连接瞬断。
不适合自动重试：

- 文档格式永久非法；API Key 无权限；模型不存在；向量维度错误；
- 租户或文档状态冲突。
指数退避应加 jitter，避免大量 Worker 同时醒来。 重试仍受总尝试次数、截止时间和费用预算约束。
## 14. 更新和删除

### 14.1 文档更新

推荐版本化替换：

1. 新版本状态 pending；
2. 新 chunk 构建完成；
3. 原子切换 active version；
4. 查询只读取 active ready 版本；
5. 旧版本进入保留或删除流程。
如果先删除旧版本再生成新版本，失败期间知识库会出现空窗。
### 14.2 删除

删除请求要明确：

- 是逻辑删除还是物理删除；chunk 是否级联；outbox 待处理事件如何取消；缓存何时失效；
- 对象存储原文件是否删除；备份中的保留期限；审计记录保留什么。
对于隐私删除，不能只把 status 改成 deleted 就宣称完成。
## 15. 一致性状态

对外暴露文档状态：

| 状态 | 含义 | 是否参与检索 |
|---|---|---|
| pending | 已接收，尚未完成 embedding | 否 |
| ready | 当前版本可检索 | 是 |
| failed | 处理失败，等待重试或人工修复 | 否 |
| deleting | 正在清理 | 否 |
查询 SQL 必须只选择 ready 且 active 的版本。 否则用户可能看到一半新、一半旧的 chunk。
## 16. 连接池与超时

数据库连接池不是越大越好。 应结合：

- 数据库 max_connections；API 实例数；Worker 实例数；单请求并发 SQL；
- 查询耗时；事务长度。
每次查询使用 context deadline。 区分 acquire timeout、statement timeout 和整个请求 timeout。 超时后应取消数据库工作，而不是只让 HTTP handler 返回。
## 17. 备份与恢复

向量可重新生成，不代表知识库无需备份。 至少保护：

- 原始文档或可重新获取的来源；文档版本与元数据；chunk 规则版本；embedding 模型版本；
- active version 指针；outbox 与任务状态；migration 历史。
恢复演练要验证：

1. Schema 和扩展可创建；
2. 数据可恢复；
3. HNSW 索引可重建；
4. 查询只读取 ready 版本；
5. 未完成 outbox 可继续；
6. 租户隔离测试仍通过。
备份存在不等于恢复可用。
## 18. 测试与基准

### 18.1 Repository 契约测试

内存 adapter 和 pgvector adapter 应通过同一组测试：

- upsert 幂等；tenant 隔离；文档过滤；维度错误；
- 稳定排序；删除级联；ready 状态过滤。
### 18.2 迁移测试

- 空数据库能升级到最新版本；有旧数据时迁移成功；重复执行按工具约定安全；回滚或前滚策略明确；
- 扩展缺失时错误清楚；新旧模型不混查。
### 18.3 性能测试

固定记录：

- 数据集和维度；查询集与 tenant 分布；冷、热缓存；并发；
- recall@k；P50/P95/P99；CPU、内存、I/O；索引大小；
- 写入吞吐和索引构建时间。
没有 recall 的“快”可能只是漏掉了正确结果。 没有过滤分布的测试也无法代表多租户线上流量。
## 19. 常见故障表

| 现象 | 常见原因 | 诊断 | 修复 |
|---|---|---|---|
| HNSW 没被使用 | 操作符类不匹配、表太小、计划估算 | EXPLAIN ANALYZE | 修查询或统计信息 |
| 强过滤后不足 K 条 | 近似候选被过滤 | 查看 filtered rows | 调搜索、分区或 partial index |
| 查询突然变慢 | 冷缓存、膨胀、计划变化 | BUFFERS、监控 | 分析 vacuum 与计划 |
| 向量写入报错 | 维度不一致 | 模型响应和列定义 | 拒绝并重建版本 |
| A 租户看到 B 数据 | 漏 tenant 谓词 | 审计 SQL 与测试 | 强制仓储参数、RLS |
| 文档长期 pending | outbox 未消费或租约卡死 | 事件状态和 lease | 恢复过期租约 |
| 重试产生重复 chunk | ID 不稳定或无唯一约束 | 查重复键 | 稳定 ID + upsert |
| Worker 锁住连接 | 远程调用放在事务中 | 长事务监控 | 短事务租约 |
| 更新期间答到半成品 | 查询未限制 active ready | SQL 谓词 | 版本化切换 |
| 换模型后质量骤降 | 新旧空间混用或未评估 | 模型版本分布 | 隔离表并灰度 |
| 索引占用过大 | 参数或冗余索引 | 索引统计 | 用基准删减 |
| 删除后仍可检索 | 缓存、旧版本、异步任务 | 全链路状态 | tombstone 与清理 |

### 练习 2

### 练习 3

### 练习 4

### 练习 5

## 21. 参考答案

### 答案 1

行数不能代表维度、硬件、过滤选择性、查询并发、缓存和召回要求。 应使用代表性数据与查询集，对精确扫描和 HNSW 测 recall、延迟、资源与写入成本，再做决定。
### 答案 2

用两个 tenant 写入语义相同但内容不同的 chunk。 以 customer-a 的可信 TenantID 查询，断言结果只含 A。 再尝试从请求体伪造 B，断言 handler 不采用该值。 最后直接调用 repository，空 TenantID 必须报错。
### 答案 3

通常不能。 维度相同只表示数组长度一致，不表示坐标空间一致。 查询向量与旧文档向量的距离没有可解释性，应按 embedding 版本隔离并重建。
### 答案 4

outbox 事件仍处于租约中。 租约过期后另一 Worker 重新领取，使用相同文档版本和稳定 chunk ID 再次调用。 最终 upsert 与唯一约束避免重复，成功后原子更新 ready 和 completed。
### 答案 5

固定语料、查询集、过滤分布、维度、机器、数据库版本和并发模型。 记录 recall@k、各分位延迟、吞吐、CPU、内存、I/O、索引大小、构建时间与写入性能。
