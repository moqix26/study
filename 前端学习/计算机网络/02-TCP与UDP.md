# TCP 与 UDP

> 本章目标：理解端口和连接，知道 TCP 为什么可靠，并能排查 Go 服务的端口问题。
>
> 前置：[01 · 网络分层与通信基础](./01-网络分层与通信基础.md)　下一章：[03 · IP 地址与 DNS](./03-IP地址与DNS解析.md)

---

## 1. 先记住三个结论

1. **TCP 和 UDP 都工作在传输层，用端口把数据交给正确的进程。**
2. **TCP 是可靠、有序的字节流；UDP 是尽力而为的一份份数据报。**
3. **HTTP 404 是应用层问题，Connection refused 才是典型的 TCP 端口问题。**

---

## 2. 为什么有 IP 还需要端口

IP 能找到一台电脑，但电脑上可能同时运行：

```text
Go API      → 8080
MySQL       → 3306
Redis       → 6379
SSH         → 22
HTTPS       → 443
```

操作系统收到数据后，需要根据**目标端口**把它交给正确进程。

把它理解为：

```text
IP 地址 = 一栋楼的地址
端口     = 楼里的房间号
进程     = 房间里正在工作的人
```

“端口属于谁”不是固定的。端口只是 `0～65535` 的数字，当前成功监听它的进程才拥有它。

### 监听端口是什么意思

Go 代码：

```go
http.ListenAndServe(":8080", nil)
```

大致表示：

1. 向操作系统申请一个 TCP socket；
2. 绑定本地 8080 端口；
3. 进入监听状态；
4. 接受客户端连接；
5. 在连接上读取 HTTP 请求。

如果另一个进程已经占用 8080，常见错误是：

```text
bind: Only one usage of each socket address is normally permitted
```

或者：

```text
address already in use
```

## 3. 什么是 socket 和四元组

### socket 不等于端口

Socket 是操作系统提供的网络通信接口。端口是 socket 地址的一部分。

一条 TCP 连接通常由四元组唯一识别：

```text
源 IP、源端口、目标 IP、目标端口
```

例如：

```text
127.0.0.1:53142 → 127.0.0.1:8080
```

因此，一个服务端 8080 端口可以同时服务很多客户端：

```text
127.0.0.1:53142 → 127.0.0.1:8080
127.0.0.1:53143 → 127.0.0.1:8080
127.0.0.1:53144 → 127.0.0.1:8080
```

虽然目标端口相同，但源端口不同，所以是三条不同连接。

## 4. TCP 到底提供了什么

TCP 可以先理解成：

> 在两个进程之间建立一条可靠、有序、双向的字节流通道。

### 4.1 面向连接

发送业务数据前，双方先建立连接。连接不是一根真实的专用网线，而是双方操作系统保存的一组通信状态，例如序号、确认号、窗口和超时计时器。

### 4.2 可靠

底层 IP 可能丢包、重复或乱序。TCP 通过多种机制尽量向应用提供可靠数据流：

- 序号：标记字节在数据流中的位置；
- 确认：接收方告诉发送方已经收到哪些数据；
- 重传：超时或判断丢包后重新发送；
- 校验和：发现传输中的错误；
- 去重与排序：重复数据丢弃，乱序数据重新排列。

“可靠”不表示永远成功。如果网络长期中断，TCP 最终仍会超时并向应用报告错误。

### 4.3 有序字节流

TCP 交给应用的是连续字节流，不保留应用写入时的消息边界。

发送方可能调用两次：

```text
write("hello")
write("world")
```

接收方不保证正好读两次，也可能读成：

```text
"helloworld"
```

或先读 `"hel"`，再读剩余内容。因此 HTTP 必须用请求行、Header、空行和 Content-Length 等规则描述消息边界。

### 4.4 全双工

连接双方都可以发送和接收数据。客户端能发请求，服务端能回响应；长连接中双方都维护各自的发送方向。

## 5. 三次握手在确认什么

建立 TCP 连接时，简化过程是：

```text
客户端                         服务端
   | ---- SYN, seq=x --------> |
   | <--- SYN+ACK, seq=y,      |
   |      ack=x+1 -------------|
   | ---- ACK, ack=y+1 ------> |
   |       连接建立             |
```

### 第一次：客户端发送 SYN

客户端表示：

> 我想建立连接，我的初始序号是 x。

### 第二次：服务端发送 SYN + ACK

服务端同时做两件事：

- ACK：我收到了你的请求和序号；
- SYN：我也告诉你我的初始序号 y。

### 第三次：客户端发送 ACK

客户端表示：

> 我也收到了你的序号，双方收发能力已确认。

### 为什么不是两次

两次只能让客户端确认服务端收到了自己的 SYN，但服务端还不能确定客户端是否收到了服务端的响应。

第三次让服务端确认：

- 客户端能收到服务端的数据；
- 客户端知道本次连接使用的服务端初始序号；
- 这不是一个已经失效的旧连接请求被网络延迟后突然送到。

面试回答不必堆术语，核心是：**双方需要确认彼此的发送、接收能力和初始序号。**

---

## 6. TCP 怎样发现丢包

### 6.1 超时重传

发送方发出数据后启动计时。如果长时间没有收到确认，就重传。

超时时间不能固定得太小，否则正常网络波动也会造成大量无谓重传；TCP 会根据往返时间动态估计。

### 6.2 快速重传

如果接收方连续确认同一个位置，发送方可能推断中间某段丢失，不必等超时就重传。

### 6.3 累计确认

确认号通常表示：

> 这个序号之前的字节我都已经连续收到，下一次希望收到这里。

这减少了确认信息数量。

第一次学习不需要记算法参数，只需知道：**序号 + 确认 + 计时器 + 重传**共同构成可靠性基础。

---

## 7. 流量控制和拥塞控制不是一回事

### 流量控制：别把接收方撑爆

接收方缓冲区有限。它会通过接收窗口告诉发送方自己还能接收多少数据。

```text
问题对象：接收方处理不过来
目标：匹配发送速度与接收能力
```

### 拥塞控制：别把网络堵死

即使接收方很快，中间网络也可能承载不了过多流量。TCP 会根据丢包、延迟等信号调整发送速率。

```text
问题对象：整个网络路径过载
目标：减少网络拥塞并逐步探测可用带宽
```

面试常问区别：

- 流量控制看接收端；
- 拥塞控制看网络；
- 实际可发送量同时受两者限制。

## 8. 四次挥手为什么通常是四次

TCP 是全双工的，两个方向要分别关闭。

```text
主动关闭方                     被动关闭方
   | -------- FIN -----------> |
   | <------- ACK ------------ |
   |                           | 仍可发送剩余数据
   | <------- FIN ------------ |
   | -------- ACK -----------> |
```

第一次 FIN 表示：

> 我这边没有更多数据要发了。

它不等于对方也立即发完。对方先 ACK，处理完剩余数据后再发送自己的 FIN，所以通常拆成四步。

### 为什么有时抓包像三次

如果被动关闭方没有剩余数据，它可能把 ACK 和自己的 FIN 合并发送。概念上两个方向仍分别关闭。

---

## 9. TIME_WAIT 是干什么的

主动关闭连接的一方发送最后一个 ACK 后，通常不会立刻删除连接状态，而会进入 TIME_WAIT。

主要原因：

1. 如果最后一个 ACK 丢了，对方会重发 FIN；主动方还能够再次回复 ACK。
2. 让网络中的旧重复报文在新连接复用相同四元组前消失，避免干扰新连接。

TIME_WAIT 不是“系统坏了”。短连接过多时它可能很多，此时应该分析：

- 是否没有复用 HTTP 连接；
- 客户端是否频繁创建新连接；
- 超时和连接池是否合理；
- 谁在主动关闭连接。

不要一看到 TIME_WAIT 就粗暴修改系统参数。

---

## 10. TCP 连接常见状态

| 状态 | 含义 |
|---|---|
| LISTEN | 服务端正在等待连接 |
| SYN_SENT | 客户端已发送连接请求 |
| SYN_RECEIVED | 服务端已收到 SYN 并回复 |
| ESTABLISHED | 连接已建立，可传数据 |
| FIN_WAIT | 主动关闭方等待对方确认或关闭 |
| CLOSE_WAIT | 收到对方 FIN，本地应用还没关闭 |
| TIME_WAIT | 主动关闭方等待旧报文消失 |

如果服务器有大量长期 `CLOSE_WAIT`，常见原因是应用收到对方关闭后，没有及时关闭自己的 socket 或响应体。

## 11. UDP 与 TCP 有什么不同

UDP 提供面向数据报的传输：发送一份，接收方按一份读取，保留消息边界。

UDP 本身不提供：

- 建立连接的握手；
- 必达保证；
- 自动重传；
- 有序保证；
- TCP 那样的流量和拥塞控制。

这不表示 UDP “低级”或“一定更快”。它只是把更多控制权交给应用。

| 对比 | TCP | UDP |
|---|---|---|
| 是否建立连接 | 是 | 否 |
| 数据形式 | 字节流 | 数据报 |
| 可靠与有序 | 协议提供 | 应用需要时自行实现 |
| 头部与状态 | 相对更多 | 相对更少 |
| 常见场景 | HTTP/1.1、HTTP/2、数据库连接 | DNS、实时音视频、QUIC 的底层承载 |

### DNS 为什么常用 UDP

传统 DNS 查询通常一问一答、数据较小，用 UDP 可以减少建连成本。响应过大、需要可靠传输或区域传送时也可能使用 TCP。

### HTTP/3 为什么能使用 UDP

HTTP/3 基于 QUIC，QUIC 在 UDP 之上自行实现可靠传输、加密和拥塞控制。它不是“直接用裸 UDP 随便丢包”。

## 12. HTTP 和 TCP 的关系

以 HTTP/1.1 为例：

```text
先建立 TCP 连接
  ↓
在 TCP 字节流中发送 HTTP 请求文本
  ↓
服务端返回 HTTP 响应
  ↓
连接关闭，或保持以便复用
```

TCP 不理解 `GET`、`404` 或 JSON。它只负责搬运字节。

HTTP 不负责解决 IP 路由、丢包重传。它依赖下层提供传输能力。

HTTP/3 基于 QUIC/UDP，但“应用层规则”和“传输层能力分工”的思想仍然成立。

## 13. Go 服务 8080 连不上怎样排查

### 第一步：确认程序是否启动成功

看终端是否打印：

```text
listening on :8080
```

并确认没有 `bind`、panic 或配置错误。

### 第二步：确认端口是否监听

```powershell
Get-NetTCPConnection -LocalPort 8080 -State Listen -ErrorAction SilentlyContinue
```

或：

```powershell
netstat -ano | Select-String ":8080"
```

没有结果，说明服务没有成功监听 8080，先查 Go 程序。

### 第三步：测试 TCP

```powershell
Test-NetConnection localhost -Port 8080
```

如果 `TcpTestSucceeded` 为 `False`，还没到 HTTP 层。

### 第四步：用 curl 看应用层

```powershell
curl.exe -v http://localhost:8080/health
```

常见结果：

| 结果 | 含义 | 下一步 |
|---|---|---|
| Connection refused | 目标可达，但端口没有接受连接 | 查监听、进程、端口 |
| Connection timed out | 路径上没有及时响应 | 查 IP、防火墙、容器映射 |
| 404 | TCP 和 HTTP 已通，路径不存在 | 查路由注册 |
| 405 | 路径存在，方法不允许 | 查 GET/POST |
| 500 | 服务端执行失败 | 查日志和依赖 |

### 第五步：检查监听地址

如果本机能访问，局域网其他设备不能访问，检查：

- 服务是否只监听 `127.0.0.1`；
- 是否监听 `0.0.0.0:8080`；
- Windows 防火墙是否允许；
- 对方使用的是否是你的局域网 IP；
- 两台设备是否在可互通网络中。

### 第六步：容器场景检查端口映射

容器内服务监听 8080，不代表宿主机自动能访问。需要类似：

```text
宿主机 8080 → 容器 8080
```

Docker Compose 常见配置：

```yaml
ports:
  - "8080:8080"
```

并确保容器内应用监听 `0.0.0.0:8080`，而不是只监听容器自己的 `127.0.0.1`。

## 14. 超时为什么非常重要

网络操作可能一直等。生产服务必须设置合理超时。

常见超时包括：

- 建立连接超时；
- TLS 握手超时；
- 等待响应头超时；
- 读取整个响应超时；
- 服务端读请求和写响应超时；
- 空闲连接超时。

Go 客户端最简单的总超时示例：

```go
client := &http.Client{
	Timeout: 5 * time.Second,
}
```

真实项目还会细分 Transport 超时。关键观念是：**超时是系统边界，不是出错后再补的装饰。**

## 15. 常见误解

### “TCP 有连接，所以不会断”

不对。断网、进程退出、防火墙、超时都能让连接失败。连接只是双方维护状态，不是物理专线。

### “三次握手后 HTTP 一定成功”

不对。握手只证明 TCP 已建立，HTTP 路径、鉴权和业务仍可能返回 404、401、500。

### “UDP 完全不可靠，所以不能做可靠协议”

UDP 自身不保证可靠，但应用可以在 UDP 上实现可靠传输，QUIC 就是典型例子。

### “一个端口只能有一条连接”

不对。一个监听端口可以接受大量连接，每条连接由四元组区分。

### “TIME_WAIT 都应该清掉”

不对。它有协议上的必要作用，应先查连接复用和关闭方式。

## 16. 本章自测

1. 为什么服务端 8080 能同时连接很多客户端？
2. TCP 的可靠性由哪些机制构成？
3. 三次握手第三次 ACK 在确认什么？
4. 流量控制和拥塞控制分别保护谁？
5. 四次挥手为什么不能简单理解为双方同时挂断？
6. `Connection refused` 和 HTTP 404 有什么本质区别？
7. UDP 没有握手，为什么不能直接说它在所有场景都更快？

参考要点：

1. 每条连接四元组不同。
2. 序号、确认、重传、校验、排序去重等。
3. 客户端已收到服务端 SYN 和初始序号，双方收发能力得到确认。
4. 流量控制保护接收端，拥塞控制保护网络路径。
5. TCP 全双工，两个发送方向分别关闭。
6. 前者发生在 TCP 建连，后者说明 HTTP 服务已经返回响应。
7. 应用需求、丢包恢复、拥塞、加密等成本仍然存在。
