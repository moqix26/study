## 1. 最终目标与明确不做的事

### 1.1 最终目标

在 2026 年冬季或更晚开始认真投递前，形成以下能力：

- 能用 Go、Gin、MySQL、Redis 开发并排查完整后端服务。
- 有一个真正做深的短链项目，达到 V2 简历级门槛。
- 能解释 Go 并发、HTTP、数据库事务与索引、缓存一致性、鉴权、限流和异步统计。
- 能运行测试、阅读日志和指标、部署项目，并基于真实压测做一次性能优化。
- 能在面试中独立讲清核心链路，并现场完成一个小改动。

### 1.2 当前只做一个项目

主项目固定为：**短链服务 ShortURL**。

当前不做：

- 第二个商城、秒杀、IM 或 Todo 项目；
- 为了简历关键词强拆微服务；
- 没有真实需求就加 Go-Zero、gRPC、Kubernetes、Service Mesh、分库分表；
- 同时跟多套 Go 或后端课程；
- 用教程、他人项目或示例中的 QPS 冒充自己的测试结果。

项目保持“小而完整”：

- 一个模块化单体 API；
- 一个可选的统计 worker；
- MySQL 是权威数据源；
- Redis 用于缓存、限流和后续统计流，并允许受控降级。

### 1.3 什么叫“学会”

每个关键模块至少满足四件事：

1. **能运行**：自己能启动、调用并看到正确结果。
2. **能解释**：能画出请求经过哪些层，数据写到哪里。
3. **能排错**：能根据错误、日志或测试定位一个常见问题。
4. **能修改**：不靠 AI 直接代劳，独立完成一个小需求或错误分支。

## 3. 执行方式：阶段 + Sprint，不再按逐日表硬排

### 3.1 Sprint 定义

一个 Sprint 通常持续 **3～7 个有效学习日**，但不按日历强制结束。

每个 Sprint 只有：

- 一个主目标；
- 一个可运行产出；
- 一组验收问题；
- 一个复盘结果。

没有达到验收门槛就继续当前 Sprint；提前完成就直接进入下一关。

### 3.2 每日时间按状态伸缩

默认按 **4 小时有效学习**设计，但允许 2～7 小时波动。

| 当天状态 | 主线安排 |
|----------|----------|
| 约 2h | 主任务 90min；排错/复盘 15～30min；算法可做 0 题 |
| 约 4h（默认） | 动手 2h；文档/视频 1h；算法 0～45min；复盘 15～30min |
| 约 7h | 主线编码 3h；新知识 1.5h；测试/文档 1.5h；算法 0～45min；复盘 15min |

原则：

- 项目和动手优先于“看了多少页”。
- 当天主线卡住时，算法可以不做。
- 状态好时延长主线，不临时开新课程或新项目。
- 学习时长只用于观察节奏，不用于制造负罪感。

### 3.3 复盘不强制新建大量笔记

实时状态只维护在 [learn.md](learn.md)：

- 当前学什么；
- 已完成什么；
- 卡在哪里；
- 下一步验收是什么。

需要长期保留的内容才进入项目：

- README；
- OpenAPI；
- migration；
- ADR 技术决策；
- 测试和压测报告；
- 故障演练记录。

不要求每天新建 notes、日报或背诵卡片。Codex 与 Cursor 的对话接力由 [.memory](.memory/readme.md) 保存。

## 4. 与 Codex / Cursor 的协作规则

### 4.1 学习练习阶段：Go 06～09

- AI 可以在聊天中给出完整可运行代码和解释。
- 你负责把关键练习代码敲进自己的练习目录并运行。
- AI 默认不直接修改练习代码，除非你明确要求。
- 函数名和库 API 不要求死记，允许查官方文档和现有章节。

### 4.2 正式短链项目阶段：Go 10 起

为了把项目做得足够优秀，Codex 或 Cursor **可以直接创建、修改和测试正式项目代码**。

但每个模块完成后，你必须通过“所有权验收”：

1. 画出或口述核心调用链；
2. 说明 MySQL、Redis、内存队列分别保存什么；
3. 说明一个依赖故障时接口如何返回；
4. 自己运行测试并读懂至少一个失败；
5. 独立完成一个小改动，例如新增字段、调整 TTL、补错误分支。

掌握标准：

- 创建、跳转、鉴权、缓存、限流、统计等核心链路接近 100% 理解；
- 配置、脚手架、重复 CRUD 等样板代码理解作用和边界即可；
- 不宣称“全部从零手写”，但必须能证明架构、验收、调试和性能实验确实由自己掌握。

## 6. 阶段 0：并发收尾

### Sprint 0：赵珊珊课程重点收尾

认真掌握：

- goroutine 的启动与生命周期；
- WaitGroup 的 Add、Done、Wait；
- Mutex 与 RWMutex 的适用场景；
- 无缓冲和有缓冲 channel 的阻塞行为；
- channel 的关闭方、range 与读取关闭通道；
- select、default 和超时控制；
- race condition 的基本含义。

可以快速扫过：

- 已经熟悉的基础语法；
- 网络编程细节；
- 反射底层。

验收：

- [ ] 能解释为什么 WaitGroup 的 Add 通常要放在启动 goroutine 之前；
- [ ] 能说出 map 为什么需要锁，以及读多写少时 RWMutex 的意义；
- [ ] 能解释无缓冲 channel 为什么会同步发送方和接收方；
- [ ] 知道通常由发送方关闭 channel，接收方不随意关闭；
- [ ] 能用 select 写超时或多路接收；
- [ ] 一个小并发程序执行 go test -race 或 go run -race 没有竞态。

不要求现在背 GMP、GC 或 sync.Pool；这些在项目和 Go 16 再深入。

## 10. 阶段 4：短链 V2——简历级

主教材：

- [Go 11 短链项目下](后端学习/Go/11-短链服务项目实战下.md)；
- [Go 12 工程化](后端学习/Go/12-单元测试日志与配置工程化.md)；
- [Go 13 部署](后端学习/Go/13-Docker与Linux部署Go服务.md)。

### 10.1 先完成 Go 08 Redis

主教材：[Go 08](后端学习/Go/08-Redis与go-redis缓存实战.md)。

在 V1 已经能只靠 MySQL 正确运行后，再学习和接入：

- Redis 与 go-redis 连接、短超时和连接池；
- key 命名、TTL 与 Cache Aside；
- 正常 miss 与 Redis error 的区别；
- Redis 故障时受控回源 MySQL；
- 负缓存、TTL 抖动和 singleflight；
- 写 MySQL 后的缓存失效边界。

最低验收：第一次查 MySQL、第二次命中 Redis；更新后不会长期读旧值；停止 Redis 后真实链接仍可回源，依赖故障不会伪装成 404。

### 10.2 V2 必做能力

跳转与缓存：

- 302 跳转及 active、disabled、expired、deleted 状态语义；
- 正缓存、负缓存、TTL 抖动和过期裁剪；
- singleflight 合并单实例热点回源；
- Redis 短超时，故障时回源 MySQL；
- version 化缓存快照 + 可重试 outbox，避免旧值长期回填。

保护与统计：

- Redis Lua 原子限流；
- Redis 故障时按接口采用 fail-open、fail-closed 或本地后备；
- 点击统计不为每个请求创建 goroutine；
- 有界队列、固定 worker、批量落库和优雅排空；
- 明确近似统计可能少记的边界。

工程化：

- 稳定错误契约和结构化日志；
- request ID；
- Prometheus RED、缓存、限流和统计指标；
- 单元、集成、E2E、race 和故障测试；
- CI 质量门禁；
- 多阶段 Dockerfile、Compose、非 root 运行；
- /livez、/readyz、/metrics；
- 独立 migration、配置和 secret；
- 一次 Redis 故障演练和一次 worker 停止恢复演练。

性能证据：

- 固定 commit、机器、Go/MySQL/Redis 版本和数据集；
- 至少覆盖 DB-only、热缓存、混合流量和 Redis 故障；
- 保存脚本和原始输出；
- 报告 QPS、P50/P95/P99、错误率、CPU、内存、缓存命中和 DB QPS；
- 至少重复三次稳定场景，使用中位数或说明波动。

### 10.3 推荐 Sprint

#### Sprint 8：Go 08 + 跳转热路径与 Redis 降级

先完成 Go 08 的 Cache Aside 基础，再落地 302、负缓存、TTL 抖动、singleflight、version/outbox 和故障测试。

#### Sprint 9：限流与有界点击统计

完成 Lua 限流、可信代理、固定 worker、有界队列、批量写入、关闭排空和指标。

#### Sprint 10：工程化、部署与证据

完成测试分层、CI、Prometheus、Docker Compose、健康探针、HTTPS/演示环境、故障演练和基础压测报告。

### 10.4 V2 简历门槛

- [ ] V1 全部验收仍通过；
- [ ] Redis 故障下已有短链可受控回源，错误不会伪装成不存在；

- [ ] CI、Docker、探针、指标和优雅关闭已落地；
- [ ] 至少两项故障演练有记录；
- [ ] 有一份可复现的真实压测报告；

## 11. 阶段 5：短链 V3——强亮点，按证据选做

主教材：

- [Go 14 高并发与可靠消息](后端学习/Go/14-分布式入门与高并发场景.md)；
- [Go 16 运行时与性能](后端学习/Go/16-Go运行时内存GC与性能分析.md)。

V3 优先只做两类真正能拉开差距的内容。

### 11.1 Redis Streams 可靠统计

- 全局 event_id；
- XADD、消费组、XREADGROUP；
- MySQL 事务内 event_id 唯一约束 + 点击聚合；
- DB commit 后再 XACK；
- XAUTOCLAIM 接管 pending；
- 有限重试、DLQ、修复与重放；
- consumer lag 和告警；
- 明确“入流失败仍可能少记”的边界。

### 11.2 一次 pprof 性能闭环

按固定流程执行：

1. 建立可复现 baseline；
2. 采集 CPU、heap 或 goroutine profile；
3. 找到一个真实热点；
4. 一次只改一个主要变量；
5. 运行单测和 race；
6. 在相同环境复测；
7. 保存优化前后数据和副作用。

V3 选做：

- Grafana 看板和有效告警；
- 多实例边界验证；
- 布隆过滤器；
- 本地二级缓存。

只有监控或压测证明存在问题时才加入。V3 仍不要求微服务、Kafka、Kubernetes 或分库分表。

## 13. 算法安排

你已有竞赛基础，算法目标是适应 Go 面试手撕，而不是重新冲 ACM。

规则：

- 每天 **0～2 题**，没有强制连续天数；
- 主线项目卡住或学习时间少时可以做 0 题；
- 优先用 Go 写常见题型：哈希、链表、树、二分、栈队列、堆、简单 DP、BFS/DFS；
- 重点练 Go 标准库、边界处理和口述复杂度；
- Hot 100 可作为题库，不设“暑假必须 80+”的虚高指标；
- 面试前再从已做题中抽 20～30 道高频题复习。

验收比题量重要：

- 25～35 分钟内完成常见中等题；
- 能口述思路和复杂度；
- 能写出可运行、边界正确的 Go 代码。

## 14. 项目证据包

项目最终至少保留这些可核验材料：

| 证据 | 最低内容 |
|------|----------|
| README | 功能、架构、版本范围、启动、测试、部署、限制 |
| OpenAPI | 公开 API、请求响应和稳定错误码 |
| migration | 空库安装、升级路径、关键索引说明 |
| ADR | 短码、幂等/去重、缓存一致性、统计取舍 |
| 自动测试 | 单元、集成、E2E、race |
| CI | 格式、vet、测试、迁移、镜像构建 |
| 可观测性 | 日志字段、RED、缓存、限流、统计指标 |
| 故障演练 | 假设、步骤、实际结果、恢复时间和改进 |
| 压测报告 | 环境、commit、脚本、数据集、P95/P99、错误率和资源 |
| 性能优化 | profile、假设、改动、回归、前后对比 |

真实性规则：

- 没有实测就写“待实测”，不填漂亮数字；
- 只压一个热短码不能证明整体容量；
- 关闭日志、限流或统计时必须在报告中注明；
- QPS 必须和延迟、错误率、资源、数据量一起解释；
- 真实的较小数字比无法复现的“10 万 QPS”更有价值。

## 16. 每个 Sprint 的收尾模板

每个 Sprint 结束时，只回答下面五项，并写入 [learn.md](learn.md)：

1. **产出**：这次能运行或展示什么？
2. **理解**：能否脱离代码讲清核心链路？
3. **证据**：哪些测试、curl、日志或指标证明它正确？
4. **卡点**：还有哪个概念或错误没有解决？
5. **下一关**：下一个 Sprint 的唯一验收是什么？

如果“代码能跑但讲不清”，先补讲解和小改动；如果“理论懂但没运行”，先回到代码。
